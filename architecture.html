微服务 & DDD 核心实战

聚焦微服务拆分方法论、服务通信与治理、容错防雪崩、可观测性、灰度发布，以及 DDD 战略/战术设计、聚合根建模与四层代码组织，是大厂架构面试的必考方向。

● 微服务架构

微服务架构

微服务是把单体按业务域拆成可独立部署、独立伸缩、故障隔离的服务集合。本质是「用分布式复杂度，换取业务与组织的可扩展性」。

① 微服务 vs 单体

核心差异对比
单体架构
· 单一进程，统一部署
· 共享数据库，代码耦合高
· 技术栈统一，开发简单
· 大团队协作冲突多
· 局部变更需全量重部署
→
微服务架构
· 多进程，独立部署/伸缩
· 每服务自有数据库，松耦合
· 技术异构，按服务选型
· 小团队独立交付
· 局部变更独立发布
微服务的代价：分布式事务、网络不可靠（超时/重试/雪崩）、运维爆炸（每个服务需独立监控/灰度/排障）、跨服务调试难。
结论：团队 < 50 人、业务边界不清晰时优先单体；业务稳定、团队规模大时再考虑拆分。

② 服务拆分方法论

按业务域拆（正确）
拆为「订单 / 库存 / 用户」服务，边界来自 DDD 限界上下文，上下文内概念一致。
按技术层拆（反模式）
controller/service/dao 各一个服务 = 分布式单体：强耦合、一挂全挂、调用链爆炸。
粒度原则
能独立开发部署 + 故障隔离 + 小团队拥有为宜。过粗退化单体，过细带来分布式 overhead 与协作爆炸。
演进式策略
先粗后细，随业务与团队规模逐步拆，避免一开始过度设计（大多数系统死于过早的微服务化）。
面试要点：微服务拆分最常见反模式是按技术层拆。正确方式是按业务域（DDD 限界上下文）拆。「一个上下文 = 一个服务候选」是判断粒度的黄金法则。其他反模式：共享数据库（绕过服务直读别人库）、上帝服务（啥都管）。

③ 服务通信：同步 vs 异步

通信方式选型
维度同步（REST / gRPC）异步（MQ）
适用场景实时响应、强一致、简单调用链削峰、解耦、最终一致、事件驱动
典型技术OpenFeign（HTTP）、gRPC（Protobuf）Kafka、RocketMQ
耦合度较高（调用方等待响应）低（生产者不关心消费者）
故障传播下游慢会阻塞上游，需熔断保护天然隔离，下游故障不影响上游
口诀：核心链路用同步（事务保证 + 实时反馈）；旁路通知 / 审计 / 日志 / 数据同步用异步（MQ 解耦）。

④ 服务治理三件套

注册与发现
Nacos / Eureka / Consul。服务启动向注册中心注册，心跳续约健康检查，调用方拉取可用实例做负载均衡（轮询/加权/一致性哈希）。
配置中心
Nacos / Apollo。配置外置、环境隔离，支持热更新（长轮询 / 推送，应用无需重启）。防止「配置漂移」，所有环境配置统一管理。
API 网关
Spring Cloud Gateway / Nginx + Lua。职责：路由、鉴权、限流、灰度、协议转换、请求聚合。区别于 Nginx：Nginx 偏 L4/L7 流量转发，网关偏业务治理（可写 Filter 逻辑）。

⑤ 微服务下的分布式事务

首选：避免分布式事务
合并到同一服务同一库；或接受最终一致。能不用强一致就不用——分布式事务带来巨大性能代价与实现复杂度。
Seata AT（推荐）
无业务侵入，自动生成 undo log 补偿。一阶段直接提交本地事务（高性能），二阶段根据全局结果执行补偿。适合大多数场景。
TCC
Try/Confirm/Cancel 三阶段，强一致但业务侵入性强，需手写补偿逻辑并处理空回滚/悬挂/幂等。适合金融强一致场景。
Saga / 本地消息表
长流程跨多服务的最终一致，靠「消息 + 重试 + 补偿」。本地消息表：业务操作 + 写消息表原子提交，异步投递 MQ，消费方幂等处理。
实战口诀：下单 = 创建订单 + 扣库存 + 扣积分 → 用事务消息 / Saga 编排，失败则补偿回滚。优先考虑业务合并或最终一致，避免强一致，必须强一致时再考虑 Seata AT / TCC。

⑥ 服务容错——防雪崩五层防线

防雪崩防线（由外到内，层层兜底）
超时
任何远程调用必须设超时，避免线程 / 连接被无限占用拖垮整体。（connect timeout + read timeout 都要配）
重试
仅对幂等接口重试，配合指数退避 + 随机抖动（防重试风暴），设置最大重试次数。
熔断
依赖异常率 / 慢调用率超阈值 → OPEN（快速失败）→ HALF-OPEN（探测恢复）→ CLOSED。防级联雪崩。Sentinel / Resilience4j
降级
熔断打开时返回兜底数据（空列表、默认值、缓存旧值），关闭非核心功能，保主链路可用。
隔离
线程池隔离（各依赖用独立线程池，慢依赖不耗尽公共线程）或信号量隔离（轻量，不创建新线程）。
面试口诀：超时兜底、重试谨慎（仅幂等）、熔断隔故障、降级保核心、隔离限爆炸——五者协同，缺一不可。

⑦ 可观测性三支柱

📋 Logging（日志）
集中日志：ELK（Elasticsearch + Logstash + Kibana）
关键：结构化日志（JSON）+ 全链路 TraceID 透传，方便跨服务排查
📊 Metrics（指标）
Prometheus + Grafana
四大黄金指标：QPS（流量）、延迟（Latency）、错误率（Errors）、饱和度（Saturation）
🔗 Tracing（链路追踪）
SkyWalking / Jaeger / Zipkin
TraceID 跨服务透传（HTTP Header / MQ Header）
无侵入 agent 接入，可视化调用链

⑧ 灰度与流量治理

蓝绿发布
同时维护蓝（旧）绿（新）两套环境，整体切流、回滚秒级。代价：资源成本翻倍。
金丝雀发布
先引入 1%~5% 流量到新版本验证，再逐步放量到 100%。风险小，是生产环境推荐策略。
滚动发布
逐批替换实例，始终保有在线实例。K8s Deployment 默认策略。无需额外资源，但回滚慢。
标签路由
实例在注册中心打 version/env 标签，网关按 Header / 用户群体把流量路由到指定版本，实现按比例 / 按人群灰度。
● DDD 领域建模

DDD 与领域建模

领域驱动设计（DDD）是驯服复杂业务的建模方法论，核心是统一语言与领域建模。微服务拆分的边界，往往就来自 DDD 的限界上下文。

战略设计（Strategic Design）

统一语言（Ubiquitous Language）
业务与技术共用一套词汇，并直接写进代码（类名、方法名就是业务术语）。消除歧义，对齐认知。
限界上下文（Bounded Context）
明确划定的语义边界，上下文内概念一致；上下文之间用防腐层（ACL）隔离，避免概念互相污染。一个限界上下文 ≈ 一个微服务候选。
上下文映射（Context Map）
描述上下文间集成关系：防腐层 ACL（翻译外部模型）、开放主机服务 OHS（标准 API）、共享内核、客户-供应商、大泥球。

战术设计（Tactical Design）— 核心构件

DDD 核心构件速查
实体（Entity）
有唯一标识（ID）、有生命周期，标识不变而属性可变。例：Order（orderId 唯一）
值对象（Value Object）
无标识、靠属性值相等（如 Money、Address），不可变，可安全共享。用值替换而非修改。
聚合根（Aggregate Root）⭐
一组必须保持一致性边界的对象集合，聚合根是唯一对外入口，外部只能经根修改内部，保证不变量（Invariant）。例：Order 是聚合根，OrderItem 是聚合内实体。
领域服务（Domain Service）
不属于任何实体的跨实体行为（如「转账」涉及两个账户）。无状态，只有行为。
仓储（Repository）
领域层定义接口（面向聚合根），基础设施层实现（MyBatis/JPA）。领域层不依赖 ORM 框架。
面试要点：DDD 落地最大难点是聚合粒度：过粗导致锁竞争，过细无法保证不变量。原则：把必须一致的对象放同一聚合，能跨聚合引用 ID 的就引用 ID 而非对象。聚合应尽量小。

贫血模型 vs 充血模型

❌ 贫血模型（反模式）
对象只是数据容器（getter/setter），所有业务逻辑散落在 Service 层。校验规则重复、不变量无法强制、领域知识流失在代码中。
✅ 充血模型（推荐）
行为内聚在领域对象内，聚合根强制不变量（如「已关闭订单不可修改」「单笔最多 20 项」），Service 只做编排，不含业务规则。
// ❌ 贫血模型：规则散落在 Service
void addItem(Order o, Product p, int n) {
    if (o.items.size() >= 20) throw ...;  // 不变量在 Service 层，到处重复
    if ("CLOSED".equals(o.status)) throw ...;
    o.items.add(new OrderItem(p, n));
}

// ✅ 充血模型：行为封装在聚合根内
class Order {
    void addItem(Product p, int n) {  // 自己知道规则，外部无法绕过
        if (status != OrderStatus.OPEN) throw new IllegalStateException("已关闭");
        if (items.size() >= 20) throw new BizException("最多 20 项");
        items.add(new OrderItem(p, n));
    }
}

领域事件 & CQRS

领域事件（Domain Event）
聚合内发生的业务事实（如 OrderPaid），用于解耦上下文间的状态传播。
可靠投递：事件落本地事件表 → Canal binlog / 定时任务投递 MQ → 消费方幂等处理（去重表 / 唯一索引）。
CQRS（命令/查询职责分离）
写模型（领域聚合，强一致）与读模型（专为查询优化的 DTO / 宽表 / Elasticsearch）分离，各自独立优化。适合读写负载差异大的场景（如订单查询 vs 下单）。
Event Sourcing（了解）
不存当前状态，存「事件流」，当前状态由事件回放得到。天然审计、可追溯，代价是 schema 演进复杂与重放性能。适合审计要求严格的金融场景。

DDD 四层架构落地 

order-service/
├── interface/        # 用户接口层：Controller、DTO、入站适配（防腐层入口）
│   └── OrderController.java
├── application/      # 应用层：用例编排、事务边界、事件订阅（不含业务规则）
│   ├── OrderAppService.java
│   └── listener/OrderPaidEventListener.java
├── domain/           # 领域层：核心业务逻辑（不依赖任何框架/DB）
│   ├── model/Order.java (聚合根) · OrderItem.java (实体) · Money.java (值对象)
│   ├── service/OrderDomainService.java
│   └── repository/OrderRepository.java (接口，由 infra 层实现)
└── infrastructure/   # 基础设施层：DB、MQ、第三方 API（实现 domain 接口）
    ├── OrderRepositoryImpl.java
    └── mq/OrderEventPublisher.java
依赖方向：interface → application → domain ← infrastructure
核心原则：领域层不依赖任何外部技术（Spring、MyBatis 等），可独立单元测试。基础设施层通过依赖注入实现领域层定义的接口（依赖倒置）。
面试要点：防腐层（ACL）是上下文集成的关键——别把外部模型（如第三方 API 返回的 DTO）直接用进核心域，应在 infrastructure 层翻译为领域对象。这样外部变化不影响核心业务逻辑。
