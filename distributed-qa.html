分布式高频面试题

Q1 CAP 理论是什么？分布式系统为什么不能同时满足三者？ +
  • C（一致性）：所有节点同一时刻读到相同数据；A（可用性）：每个请求都能收到响应；P（分区容忍性）：网络分区时系统仍可运行。
  • P 在分布式系统中不可放弃——网络故障是客观存在的，真正的选择只在 C 和 A 之间。
  • 网络分区时：选 C（CP）则拒绝写请求直到节点同步，牺牲可用性；选 A（AP）则立即返回可能过期的数据，牺牲一致性。
  • 典型 CP：ZooKeeper、etcd（选主期间不可用）；典型 AP：Eureka（自我保护，不剔除实例）、Cassandra。
Q2 BASE 理论是什么？和 ACID 有什么区别？ +
  • BA（基本可用）：系统故障时允许降级，核心功能仍可用（如双十一限流降级、展示兜底数据）。
  • S（软状态）：允许中间状态存在，节点间数据同步有延迟，不要求实时一致。
  • E（最终一致性）：无新更新时，经过一段时间所有副本最终达到一致。
  • 与 ACID 对比：ACID 强调强一致性（事务内立即一致），适合金融/订单；BASE 强调高可用优先，适合社交/推荐等高并发场景。
  • BASE 不是"随便不一致"，需通过对账、重试补偿、版本号等机制保证最终收敛到一致。
Q3 分布式事务有哪些解决方案？各自适用什么场景？ +
  • 2PC：强一致，性能差（同步阻塞），业务侵入低，适合 DB 层 XA 协议；缺点是单点故障和数据不一致风险。
  • TCC（Try-Confirm-Cancel）：最终一致，性能好，业务侵入性强（需实现 3 个接口）；适合资金、库存等核心业务；需处理空回滚/悬挂/幂等三大异常。
  • Saga：将长事务拆成本地事务序列，每步立即提交，失败时逆向补偿；适合订单、物流等长流程业务。
  • 本地消息表：业务操作与消息写入同一本地事务，定时任务扫描发送 MQ；对业务侵入最小，适合跨服务异步通知。
  • 事务消息（RocketMQ）：生产者发送半消息，本地事务执行后提交/回滚，Broker 定期回查；适合生产者与消费者解耦场景。
Q4 2PC 的原理是什么？有哪些缺陷？ +
  • 同步阻塞：Prepare 阶段参与者持有锁等待协调者指令，高并发下锁竞争严重，吞吐量极低。
  • 单点故障：协调者宕机后，参与者将永久阻塞并持锁无法释放，系统僵死。
  • 数据不一致：Phase 2 协调者发出部分 Commit 后宕机，已提交和未提交的参与者数据出现分歧。
  • 过于保守：任一参与者返回 No，整个事务回滚；网络抖动可能导致误判，降低成功率。
  • 3PC 通过增加 CanCommit 阶段和超时机制部分缓解单点问题，但实现复杂，生产中极少使用。
Q5 TCC 的原理是什么？和 2PC 相比有什么优缺点？ +
  • 无全局锁：TCC 的 Try 阶段只预留资源不加全局 DB 锁，Confirm/Cancel 完成实际提交，并发性能远优于 2PC。
  • 业务侵入性：TCC 需要为每个服务实现 Try/Confirm/Cancel 三个接口，开发成本高；2PC 由 DB 层（XA 协议）自动处理，对业务透明。
  • 三大异常必须处理：
    • 空回滚：Try 未收到（网络超时），Cancel 先到达，需判断 Try 是否执行过再决定是否回滚
    • 悬挂：Cancel 先执行，Try 才到达，需拒绝后来的 Try（事务已结束）
    • 幂等：Confirm/Cancel 网络重试，需保证重复执行结果相同
  • 实践中推荐用 Seata TCC 模式，框架层处理幂等和空回滚，减少手动实现负担。
Q6 如何用 Redis 实现分布式锁？有哪些常见问题？ +
  • 加锁：SET lock_key ${uuid} NX PX 30000——NX 保证只有一个客户端能设置成功，PX 设置过期时间防止死锁，value 存 UUID 标识持锁方。
  • 释放锁：必须用 Lua 脚本原子执行"判断 value 相等 + DEL"，不能直接 DEL——否则可能误删其他线程的锁（A 锁过期，B 获锁，A 此时 DEL 会删 B 的锁）。
  • Redisson：封装了 SET NX + Lua 删除 + 看门狗续期，支持可重入锁、公平锁、读写锁，生产推荐使用。
  • 常见问题：锁过期业务未完成（看门狗解决）、主从切换锁丢失（RedLock 解决）、无法重入（Redisson 用 hash 结构记录重入次数）。
Q7 Redisson 的看门狗机制是什么？如何防止锁提前过期？ +
  • 看门狗（WatchDog）：调用 lock() 时如果不指定 leaseTime，Redisson 默认 leaseTime = 30s，同时启动后台 WatchDog 线程。
  • 续期机制：WatchDog 每隔 leaseTime / 3（默认 10s） 检查锁是否仍被持有，若是则重置过期时间为 30s，防止业务未完成时锁自动过期。
  • 停止续期：调用 unlock() 后，Redisson 删除锁并取消 WatchDog 定时任务，不再续期。
  • 注意：若指定了 leaseTime（如 lock(10, TimeUnit.SECONDS)），看门狗不会启动，锁在指定时间后自动过期，需自行估算业务执行时间。
  • JVM 进程崩溃时 WatchDog 线程也消失，锁会在 30s 后自动过期，不会死锁。
Q8 ZooKeeper 如何实现分布式锁？和 Redis 分布式锁有什么区别？ +
  • 实现步骤：① 在 /locks/ 下创建临时顺序节点；② 获取所有子节点并排序；③ 若自己是序号最小的节点则获锁；④ 否则 Watch 前一个节点（非 Watch 所有节点）。
  • 自动释放：持锁方宕机后会话断开，ZK 自动删除临时节点，触发后继节点的 Watch，不会死锁。
  • 羊群效应优化：Watch 前驱节点而非所有节点，避免节点释放时唤醒所有等待者（N-1 次无效唤醒）。
  • Redis vs ZK 对比：Redis 性能高（AP），ZK 可靠性强（CP）；Redis 主从切换可能锁丢失，ZK 临时节点无此问题；ZK 吞吐低，不适合高并发锁竞争。
Q9 Nacos、Eureka、ZooKeeper 作为注册中心有什么区别？如何选型？ +
  • Eureka（AP）：Spring Cloud 早期标配，自我保护机制（15min 内心跳丢失 >15% 不剔除）保障可用性，但已停止维护，新项目不推荐。
  • Nacos（AP/CP 可切换）：阿里开源，服务注册默认 AP（高可用），配置中心用 CP（强一致）；支持推送 + 健康检查，生态完善，国内首选。
  • ZooKeeper（CP）：Leader 选举期间服务不可用，服务发现场景下可用性差，适合做分布式锁/配置协调而非注册中心。
  • Consul（CP）：支持多数据中心，内置 DNS 服务发现，适合云原生和 K8s 场景。
  • 选型建议：国内 Spring Cloud 项目首选 Nacos；K8s 场景可用 Consul 或直接使用 etcd；强一致配置管理用 etcd/ZK。
Q10 Snowflake 算法的原理是什么？有哪些缺陷？ +
  • 结构（64 bit）：1位符号位（固定0）+ 41位时间戳（ms，可用约69年）+ 10位机器ID（5位datacenter + 5位workerId，最多1024台）+ 12位序列号（同毫秒内最多4096个ID）。
  • 特点：本地生成无网络依赖，每毫秒可生成 4096 × 1024 ≈ 420万个 ID；趋势递增（时间戳在高位），对 B+树友好。
  • 时钟回拨问题：NTP 校正导致系统时间倒退，若直接使用当前时间戳会生成重复 ID。
  • 解决方案：① 回拨量小（<5ms）时 spin 等待追上；② 回拨量大时抛异常由上层重试；③ 美团 Leaf 借用扩展位标记回拨，继续生成不重复 ID。
  • 实践中需保证机器ID不重复，可从 ZK/DB 动态分配；K8s 环境中 Pod 重启可能导致机器ID复用，需额外处理。
Q11 如何实现接口限流？令牌桶和漏桶算法有什么区别？ +
  • 令牌桶：以固定速率向桶中放令牌（容量=burst），请求消费令牌；桶满时令牌积累，可应对突发流量（允许瞬间通过 burst 个请求）。适合：API 限流、允许短暂突发的场景（Guava RateLimiter）。
  • 漏桶：请求进入漏桶排队，桶以固定速率漏出（处理）；桶满则丢弃新请求，强制平滑输出，无论输入多大，输出速率恒定。适合：保护下游固定处理速率的服务。
  • 核心区别：令牌桶允许突发（积累令牌），漏桶绝对平滑（固定输出）；接口保护通常用令牌桶（兼顾弹性和保护），消息队列消费速率控制用漏桶。
  • 固定窗口问题：窗口边界处可能放过双倍流量（前一窗口末尾 + 后一窗口开头）；滑动窗口通过细分子窗口解决此问题，Sentinel 默认使用。
Q12 什么是服务熔断？熔断器的三种状态是什么？ +
  • CLOSED（关闭）：正常工作状态，请求正常通过，同时统计最近时间窗口内的成功/失败率；当失败率超过阈值（如50%）或慢调用比例过高时，切换到 OPEN。
  • OPEN（打开）：熔断打开，所有请求直接快速失败（执行 fallback 逻辑），不再调用下游，防止级联雪崩；持续一段时间（如5秒）后自动进入 HALF-OPEN。
  • HALF-OPEN（半开）：探测恢复阶段，放行少量请求（如10%）到下游；若成功率达标则恢复 CLOSED，若仍有大量失败则回到 OPEN。
  • 熔断的意义：下游服务故障时快速失败，避免请求堆积耗尽线程池资源，防止级联雪崩（一个服务故障拖垮整个调用链）。
  • Sentinel vs Resilience4j：Sentinel 适合国内微服务，规则配置丰富；Resilience4j 轻量，适合 Spring Boot 项目，两者均支持三状态熔断器。
Q13 一致性哈希算法是什么？如何解决数据倾斜问题？ +
  • 哈希环：将 [0, 2³²-1] 的哈希空间首尾相连构成环，节点和 key 都映射到环上；key 沿顺时针方向找到第一个节点即为目标节点。
  • 优势：新增/删除节点只影响该节点到前驱节点之间的 key（约 1/N），其余 key 映射不变，极大减少缓存失效范围（传统取模则约 (N-1)/N 失效）。
  • 数据倾斜问题：节点少时分布不均，某节点可能承载大量数据；解决方案：虚拟节点——每个物理节点映射 150+ 虚拟节点均匀散布在环上，负载均衡。
  • 应用场景：Redis Cluster（16384槽的变种实现）、Memcached ketama 算法、Nginx 一致性哈希负载均衡、Dubbo 集群路由。
Q14 分布式系统中如何保证接口幂等性？ +
  • 唯一请求 ID：客户端生成全局唯一 requestId，服务端查 Redis/DB 判断是否已处理；已处理则直接返回上次结果。适合：支付、下单。
  • Token 机制：请求前先获取一次性 Token（存 Redis），提交时携带 Token，服务端原子判断+删除 Token（Lua）；Token 不存在则拒绝，防止表单重复提交。
  • 数据库唯一索引：在业务唯一字段（如 order_no）上建唯一索引，重复插入抛 DuplicateKeyException 并捕获返回成功。适合写入类幂等。
  • 状态机：通过状态流转（如 PENDING→PAID）保证幂等，已流转则不再重复处理；UPDATE ... WHERE status='PENDING' 乐观锁控制。
  • 乐观锁版本号：更新时携带 version，UPDATE ... WHERE id=? AND version=?，并发更新时只有一个成功。
Q15 什么是分布式链路追踪？TraceId 如何实现全链路透传？ +
  • 核心概念：TraceId 标识整条请求链路（全局唯一），SpanId 标识链路中的单个操作（父子关系组成调用树），一个 Trace 包含多个 Span。
  • MDC 透传：利用 MDC.put("traceId", traceId) 将 TraceId 存入线程本地变量，日志框架自动打印；跨线程需手动传递（线程池场景用 TransmittableThreadLocal）。
  • 跨服务透传：HTTP 调用时将 TraceId 放入请求头（如 X-Trace-Id），下游服务通过拦截器/Filter 提取并放入 MDC；MQ 场景放入消息属性。
  • Skywalking：字节码增强（Java Agent），无侵入，自动采集 TraceId/SpanId 并上报；Zipkin 需要业务代码集成（Spring Cloud Sleuth）。
  • 链路追踪解决分布式系统下日志分散的问题，结合 ELK/Jaeger 可以按 TraceId 聚合完整调用链，快速定位性能瓶颈和错误根因。
Q16 Raft 算法的 Leader 选举和日志复制流程是什么？和 Paxos 有什么区别？ +

Leader 选举（三步）：

  1. Follower 超过 Election Timeout（随机 150-300ms）未收到心跳，转为 Candidate，term++，先投自己一票
  2. 并发广播 RequestVote RPC，携带自身最后日志的 (index, term) 用于比较日志新旧
  3. 获多数票（>N/2）成为 Leader，立即广播心跳；否则随机等待后重试

投票安全规则：只投给日志"至少一样新"的 Candidate——先比 term，再比 index。确保当选 Leader 一定包含所有已提交日志。

日志复制（四步）：

  1. 写请求到达 Leader，追加到本地日志（未提交）
  2. 并发发 AppendEntries RPC 给所有 Follower，携带日志条目和前驱日志的 (index, term) 做一致性检查
  3. 多数 Follower 写入成功后，Leader 提交（apply 到状态机），回复客户端
  4. 下次心跳通知 Follower 更新 commitIndex，Follower 提交

Raft vs Paxos：

  • Paxos：解决单值共识，理论严谨，但 Multi-Paxos 没有规范工程实现，各家（Chubby、Spanner）做了大量私有扩展，实现复杂、难以理解和验证正确性
  • Raft：直接面向日志复制，Leader 唯一使写路径清晰；工程实现有参考规范（etcd、TiKV、Nacos CP、CockroachDB、Consul 均采用）；代价是 Leader 单点成为写瓶颈，可通过 Multi-Raft（多个 Raft 组分片）缓解
  • 面试答法：Raft 本质是 Multi-Paxos 的工程化简化版——限制了 Leader 的唯一性和日志只能从 Leader 流向 Follower，牺牲了一点灵活性，换取可理解性和实现确定性
⚡ 实战 Redisson 分布式锁续期失败导致业务异常，怎么排查？ +

一、Redisson Watchdog 机制回顾

Redisson 的 RLock 默认使用 watchdog 自动续期机制：加锁时如果未指定 leaseTime，锁超时时间为 30 秒（由 lockWatchdogTimeout 配置），后台 watchdog 线程每隔 lockWatchdogTimeout / 3 = 10 秒 执行一次续期操作（通过 Lua 脚本将 Redis 中锁的 TTL 重置为 30s）。只要持有锁的线程还活着，锁就不会过期释放。

关键源码路径：RedissonLock#scheduleRenewal() → 注册 Timeout 任务 → renewExpiration() 执行 Lua 续期脚本。如果客户端与 Redis 之间连接断开，watchdog 无法续期，锁将在 30s 后自动过期，其他客户端可以获取锁——这是导致业务异常的核心场景。

二、续期失败常见原因

  • Redis 节点宕机 / 网络分区：Redis 主节点不可达，watchdog 的续期请求无法送达。主从切换期间（Sentinel 故障转移通常 5-15s），锁的 key 可能尚未同步到从节点，新主节点上锁不存在，续期直接失败。
  • JVM GC 停顿（STW）：Full GC 或长时间 Young GC（如大堆内存 + 大量老年代对象）导致应用线程全部暂停，watchdog 续期任务无法按时执行。可通过 -Xlog:gc*:file=gc.log（JDK11+）或 -XX:+PrintGCDetails（JDK8）查看 GC 停顿时长。如果 STW 超过 10s，续期窗口被错过。
  • 应用重启 / OOM Kill：进程异常退出后 watchdog 线程随之终止，锁在 30s 后自动过期。重启后新实例无法感知旧锁的存在，可能导致并发问题。
  • Redis 慢查询阻塞：如果 Redis 单线程被慢命令（如 KEYS *、大 key 操作）阻塞，续期的 PEXPIRE 命令排队等待，可能超时。

三、排查步骤与命令

  • 检查 Redis 连接状态： redis-cli info clients 查看 connected_clients 是否骤降； redis-cli slowlog get 20 查看近期慢查询，确认是否有阻塞续期的慢命令； redis-cli info replication 确认主从同步状态和 offset 差距。
  • 检查 GC 日志：分析 GC 日志中是否有超过数秒的 STW 停顿。使用 jstat -gcutil <pid> 1000 实时监控，或用 GCEasy（gceasy.io）上传日志可视化分析。重点关注 Full GC 频率和耗时。
  • 确认锁参数配置：检查代码中 tryLock(waitTime, leaseTime, unit) 的设置。如果设置了 leaseTime（如 10s），则 watchdog 不会启动，锁到期自动释放——这是最常见的配置错误。正确做法：不传 leaseTime 或传入 -1 让 watchdog 生效。
  • 查看 Redisson 日志：Redisson 使用 org.redisson 包下的 logger，开启 DEBUG 级别可看到续期任务的执行情况： logback.xml: <logger name="org.redisson" level="DEBUG"/>

四、解决方案

  • tryLock 超时兜底：始终使用 tryLock(waitTime, TimeUnit) 而非 lock()，设置合理的等待时间避免无限阻塞。捕获返回 false 的情况走降级逻辑。
  • RedLock 多节点部署：Redisson 提供 RedissonRedLock，在 N 个独立 Redis 节点上加锁，超过 N/2+1 个节点加锁成功才算成功。但 Martin Kleppmann 与 Antirez 对此有过经典争论——RedLock 在时钟跳跃场景下仍不安全，不适合强一致性场景。
  • 关键业务降级方案：对资金类、库存扣减等核心操作，使用 数据库乐观锁（UPDATE stock SET count=count-1 WHERE id=? AND count>0）或 ZooKeeper 临时有序节点锁（Curator InterProcessMutex）作为降级，ZK 基于会话心跳检测，不存在续期问题。
  • 监控告警：对 Redisson 锁获取耗时（通过 AOP 切面统计）和锁等待队列长度设置 Prometheus 指标，锁获取超过阈值时告警。

一、AT 模式回滚原理

Seata AT（Automatic Transaction）模式的核心思路是无侵入自动补偿。事务提交阶段（Phase 1），各分支 RM（Resource Manager）在本地事务提交前，Seata 拦截 SQL 并自动记录 undo_log（包含 beforeImage 和 afterImage），与业务数据在同一个本地事务中提交。全局事务需要回滚时（Phase 2），TC（Transaction Coordinator）通知各 RM 根据 undo_log 生成反向补偿 SQL：INSERT 变 DELETE，UPDATE 用 beforeImage 还原，DELETE 重新 INSERT。

关键表：undo_log（每个业务库都需要建）记录回滚日志；global_table、branch_table、lock_table 在 Seata Server 端记录全局事务和分支状态。

二、回滚失败常见原因

  • undo_log 表被清理：运维误删或定时清理任务（如保留 7 天的 cron job）在事务尚未完成时删除了对应的 undo_log 记录，导致 RM 无法构造回滚 SQL。报错：io.seata.rm.datasource.undo.AbstractUndoExecutor: undo_log not found。
  • 脏写（Dirty Write）：AT 模式下全局锁（global_lock）保护被修改的行，但如果其他非 Seata 管理的事务（如直接操作数据库的脚本、未接入 Seata 的旧服务）修改了同一行数据，回滚时 afterImage 与当前数据不匹配，校验失败报错：DirtyDataException: data has been modified by other transaction。
  • 数据库连接断开：分支 RM 所在服务宕机或数据库连接池耗尽，Phase 2 回滚请求无法送达或执行超时。Seata Server 默认重试 5 次，每次间隔递增。
  • DDL 变更：在分布式事务期间对表执行了 ALTER TABLE（加字段、改类型），导致 beforeImage 的列结构与当前表结构不匹配，undo SQL 执行报错。

三、排查步骤

  • 查 Seata Server 日志：在 seata-server/logs/seata-server.log 中搜索 XID（全局事务 ID），定位哪个分支回滚失败。关注 BranchRollbackResponse 的 resultCode 和异常堆栈。也可以通过 Seata 1.5+ 的 Web Console（默认端口 7091）查看全局事务状态和各分支状态。
  • 检查 undo_log 表：SELECT * FROM undo_log WHERE xid = 'your-xid' ORDER BY log_created，确认回滚日志是否存在且完整。如果记录已被删除，需要手动构造补偿 SQL。
  • 检查 global_lock 表：SELECT * FROM lock_table WHERE xid = 'your-xid'，确认全局锁是否有残留（事务超时后锁未释放）。残留锁会阻塞其他事务获取同一行的全局锁，导致新事务超时。
  • 检查分支数据库：对比 undo_log 中的 afterImage 与当前数据库中的实际数据，确认是否被脏写：SELECT * FROM your_table WHERE id = ? ，与 afterImage 逐字段比对。

四、处理方式

  • 手动补偿 SQL：如果 undo_log 还在但自动回滚失败，可以手动取出 beforeImage 数据执行还原：UPDATE your_table SET col1=before_val1, col2=before_val2 WHERE id=?，然后手动删除 undo_log 记录并通过 Seata API 标记分支为已完成。
  • 调整重试与超时配置：在 application.yml 中设置 Seata 重试策略：service.enable-degrade=true 开启降级；client.rm.async-commit-buffer-limit 调整异步提交缓冲区；transport.thread-factory.boss-thread-size 增加通信线程。全局事务超时时间通过 @GlobalTransactional(timeoutMills=60000) 设置。
  • 监控告警：集成 Prometheus，监控 seata_transaction 指标（status=rollback 的数量），设置全局事务超时未完成时告警。Grafana 面板展示全局事务成功率、平均耗时、回滚率。

五、架构建议

  • 核心支付链路用 TCC 替代 AT：AT 模式依赖数据库层面的 undo_log 和全局锁，对脏写敏感且不适合跨异构数据源。TCC（Try-Confirm-Cancel）在业务层面实现三阶段操作——Try 冻结资源（如冻结金额到中间账户）、Confirm 确认扣减、Cancel 解冻——控制粒度更细，不依赖数据库机制，更适合金融场景。Seata 同时支持 TCC 模式，通过 @TwoPhaseBusinessAction 注解实现。
  • 防御性设计：接入 Seata 的库严格禁止非 Seata 管理的事务修改受保护的数据行；undo_log 清理任务必须检查事务状态（只清理 COMMITTED/ROLLBACKED 的记录）；生产环境对 DDL 变更加审批流程，禁止在活跃事务期间执行。
