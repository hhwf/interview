分布式系统核心知识

分布式系统将单体应用拆分为多个独立服务，带来水平扩展能力的同时，也引入了网络分区、数据一致性、服务协调等新挑战。 掌握 CAP/BASE 理论、分布式事务、分布式锁等核心原理，是设计高可用系统和通过大厂面试的必备基础。

CAP & BASE 理论

CAP 定理指出分布式系统不能同时满足一致性（C）、可用性（A）、分区容忍性（P），只能三选二。BASE 是对 ACID 的弱化，通过牺牲强一致性换取高可用与高性能。

CAP 三要素

C — 一致性 Consistency：所有节点在同一时刻看到相同的数据。写操作完成后，所有后续读都能读到最新值（强一致性）。
A — 可用性 Availability：每个请求都能收到响应（不一定是最新数据），系统不会拒绝服务。
P — 分区容忍性 Partition tolerance：网络分区（节点间通信中断）发生时，系统仍能继续运行。分布式系统中 P 几乎不可放弃。

为什么只能三选二？

  网络分区发生时（P 必须保留），节点 A 和节点 B 无法通信：

  若要保证 C（一致性）：写 A 后必须同步到 B 才能返回成功
    → B 不可达时必须拒绝写请求或阻塞等待 → 牺牲 A（可用性）

  若要保证 A（可用性）：写 A 直接返回成功，不等待 B 同步
    → B 的数据暂时落后 → 牺牲 C（一致性）

  结论：分区发生时，C 和 A 只能选一个。

CP vs AP — 典型系统选型

类型代表系统分区时行为适用场景
CPZooKeeper、etcd、HBase、Consul拒绝服务，保证数据一致分布式锁、选主、配置中心（数据不能错）
APEureka、Cassandra、DynamoDB、CouchDB继续服务，可能返回旧数据服务注册、购物车、DNS（可用性优先）
CA单机关系型数据库（MySQL 单节点）不存在网络分区问题单体应用（无法水平扩展）

BASE 理论

BASE 是 AP 系统的设计指导原则，牺牲强一致性换取高可用：

BA — 基本可用 Basically Available：系统出现故障时，允许损失部分可用性（如延迟增加、降级返回默认值），但核心功能仍可用。
S — 软状态 Soft state：允许系统中存在中间状态，数据在节点间同步存在延迟，不要求节点时刻一致。
E — 最终一致性 Eventually consistent：系统保证在没有新的更新操作后，经过一段时间，所有副本最终达到一致状态。
维度ACID（传统数据库）BASE（分布式系统）
一致性强一致性（事务内立即一致）最终一致性（允许短暂不一致）
可用性牺牲可用性保证一致性优先可用性
适用金融、订单（数据强一致要求）社交、推荐、购物车（高并发优先）
面试要点：实际系统中 P 不可放弃，真正的选择是 CP 还是 AP。Nacos 默认 AP 模式但可切换 CP；ZooKeeper 是 CP；Eureka 是 AP。"最终一致性"不等于"随便不一致"，需要通过补偿机制（重试、对账）保障。

分布式事务

跨服务 / 跨库操作无法依赖本地事务，需要 2PC、TCC、Saga、本地消息表等方案在一致性与可用性之间取得平衡。

主流方案对比

方案一致性性能业务侵入适用场景
2PC强一致低（阻塞）低数据库层（XA 协议）
TCC最终一致高高（需实现 3 接口）资金、库存等核心业务
Saga最终一致高中（需编写补偿）长流程业务（订单、物流）
本地消息表最终一致高低跨服务异步通知
事务消息（MQ）最终一致高低生产者与消费者解耦

2PC（两阶段提交）

  协调者（Coordinator）          参与者（Participant A / B）

  ── Phase 1: Prepare ──────────────────────────────────
  发送 Prepare 请求  ──────────►  执行本地事务，写 undo/redo log
                                  返回 Yes / No
  ── Phase 2: Commit / Rollback ────────────────────────
  全部 Yes → 发送 Commit ──────►  提交本地事务，释放锁
  任一 No  → 发送 Rollback ────►  回滚本地事务，释放锁

2PC 缺陷

同步阻塞Prepare 阶段参与者持有锁等待协调者指令，高并发下锁竞争严重
单点故障协调者宕机，参与者将永久阻塞（持锁无法释放）
数据不一致Phase 2 中协调者发送 Commit 后宕机，部分参与者提交、部分未提交

TCC（Try-Confirm-Cancel）

  Try      ── 预留资源（冻结库存 / 冻结余额），检查并预占，不提交
  Confirm  ── 正式提交，将预留资源转为实际扣减
  Cancel   ── 释放预留资源，回滚 Try 阶段的操作

  示例：转账 A→B 100元
    Try:     A 冻结 100，B 预增 100（两者均可回滚）
    Confirm: A 扣减 100，B 实际到账 100
    Cancel:  A 解冻 100，B 取消预增
优点无全局锁，性能好；业务逻辑在应用层，不依赖数据库 XA
缺点业务侵入性强（必须实现 3 个接口）；需处理空回滚、悬挂、幂等三种异常

Saga（长事务补偿）

  将长事务拆分为一系列本地事务，每个本地事务对应一个补偿事务：

  T1 → T2 → T3 → T4   （正向流程，每步立即提交）
       ↓ 失败
  T3 → C2 → C1         （反向补偿，从失败点向前回滚）

  两种编排方式：
  ① 编排（Choreography）：事件驱动，服务间通过 MQ 触发，去中心化
  ② 协调（Orchestration）：Saga 协调器统一调度（Seata Saga 模式）

本地消息表（最终一致性经典方案）

  ┌──────────────────────────────────────────────────────┐
  │  本地事务（原子）                                       │
  │  1. 执行业务操作（如扣减库存）                           │
  │  2. 写消息到本地消息表（status=PENDING）                 │
  └──────────────────────────┬───────────────────────────┘
                             │ 事务提交成功
                             ▼
  定时任务扫描 PENDING 消息 → 发送到 MQ → 消费者处理
                             │ 消费成功
                             ▼
                    更新消息状态 = DONE
  消费失败 → MQ 重试 → 消费者保证幂等
面试要点：TCC 的三种异常必须处理——空回滚（Try 未执行就收到 Cancel）、悬挂（Cancel 先于 Try 执行）、幂等（Confirm/Cancel 被重复调用）。生产中推荐使用 Seata 框架（支持 AT/TCC/Saga/XA 四种模式），AT 模式对业务无侵入，底层自动生成 undo log。

分布式锁

多节点并发写同一资源时需要分布式锁协调，主流方案包括基于 Redis（Redisson 看门狗）和 ZooKeeper（临时顺序节点）两种路线。

为什么需要分布式锁？

单机环境下 synchronized / ReentrantLock 只在同一 JVM 进程内生效，多实例部署后进程间无法共享内存，需要借助第三方存储（Redis、ZK）实现跨进程互斥。

Redis 方案

  // 1. 加锁（原子操作）
  SET lock_key ${uuid} NX PX 30000
      └── NX: 不存在时才设置   PX: 30000ms 自动过期

  // 2. 释放锁（Lua 脚本保证原子性）
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])   -- 只删自己的锁
  else
      return 0
  end
  -- value 存 uuid，防止误删其他线程的锁
SET NX PX 单条命令原子加锁，NX 防重入，PX 设置过期时间防死锁
Lua 脚本删锁 get + del 非原子，必须用 Lua 脚本保证"判断+删除"原子性，避免误删他人的锁
Redisson 看门狗 默认 leaseTime=30s，后台 WatchDog 每 10s（leaseTime/3）自动续期；调用 unlock() 后停止续期并删除锁
RedLock 向 N 个独立 Redis 实例加锁，超过半数（N/2+1）成功即视为获锁成功，提升可靠性（了解即可）

ZooKeeper 方案

  1. 所有竞争者在 /locks/ 下创建临时顺序节点
     /locks/lock-0000000001  (客户端A)
     /locks/lock-0000000002  (客户端B)
     /locks/lock-0000000003  (客户端C)

  2. 每个客户端判断自己是否序号最小
     → 是：获得锁，执行业务
     → 否：Watch 前一个节点（非最小）

  3. 前驱节点删除（持锁方释放 or 会话断开自动删临时节点）
     → ZK 通知后继节点 → 后继节点检查再获锁

  优化：Watch 前驱而非全量节点，避免"羊群效应"

Redis vs ZooKeeper 分布式锁对比

维度RedisZooKeeper
CAP 取向AP（主从可能不一致）CP（强一致，过半选主）
性能极高（内存操作，单机万级 TPS）较低（ZAB 协议同步，千级 TPS）
可靠性主从切换期间可能锁丢失高，临时节点会话断开自动释放
锁释放依赖过期时间 + 主动删除会话断开自动删临时节点，更安全
适用场景高并发、对一致性要求不是极高的场景强一致性场景（选主、协调）
注意：Redis 主从切换期间，主节点尚未同步到从节点的锁数据可能丢失——新主节点上该锁不存在，另一个客户端可能同时获锁，导致锁失效。高一致性场景应使用 RedLock 或改用 ZooKeeper。

注册中心与服务发现

服务注册中心负责维护服务实例列表，支持健康检查和动态上下线。Nacos（AP/CP 可切换）、Eureka（AP）、ZooKeeper（CP）各有取舍。

维度EurekaNacosZooKeeperConsul

分布式 ID

分布式环境下数据库自增 ID 无法保证全局唯一，需要 Snowflake、号段模式、Redis incr 等方案生成全局唯一、趋势递增的 ID。

分布式 ID 的核心要求

全局唯一 同一业务场景下 ID 不重复，是基本要求
趋势递增 MySQL B+ Tree 的叶节点顺序写入时性能最优，随机 ID（UUID）导致页分裂，写性能下降
高性能 每秒生成百万级 ID，不能成为系统瓶颈
高可用 ID 生成服务不能单点，需容灾和降级方案

主流方案对比

方案唯一性趋势递增性能依赖缺点
UUID全局唯一否（随机）极高（本地）无128bit 太长，不适合主键
DB 自增库内唯一是低（DB 瓶颈）数据库单点，分库后 ID 重复
Snowflake全局唯一是（毫秒级）极高（本地）时钟时钟回拨问题
号段模式全局唯一是高（批量取号）数据库每次需访问 DB
Redis INCR全局唯一是高Redis持久化重启后可能重复

Snowflake 算法详解

  64 bit long 整数结构：

  ┌─┬─────────────────────────────────────────┬──────────────┬────────────┐
  │0│          41 位时间戳（ms）                │  10位机器ID  │ 12位序列号 │
  └─┴─────────────────────────────────────────┴──────────────┴────────────┘
  符号位                                          ↑               ↑
                                           5位datacenter    序列号 0~4095
                                           + 5位workerId

  时间戳基准：2010-11-04 01:42:54 UTC（Twitter 纪元）
  41位时间戳：可用约 69 年（2^41 ms ≈ 69年）
  10位机器ID：最多 1024 台机器
  12位序列号：同一毫秒内最多生成 4096 个 ID
  → 理论 QPS：1024 × 4096 × 1000 ≈ 42 亿/秒

时钟回拨问题与解决方案

问题 NTP 时间校正导致系统时钟倒退，若用当前时间戳生成 ID 会出现重复
方案1：抛异常 检测到时钟回拨直接抛异常，由上层业务重试（简单直接，适合回拨量小的场景）
方案2：等待追上 回拨量小（<5ms）时 spin 等待时钟追上，回拨量大时抛异常
方案3：借用序列号 美团 Leaf 做法：回拨时用扩展位标记，借用未来时间的序列号空间生成 ID

号段模式（Leaf-segment）

  DB 表：id_alloc(biz_tag, max_id, step)

  1. 应用启动时从 DB 取一个号段：[max_id, max_id + step)
     UPDATE id_alloc SET max_id = max_id + step WHERE biz_tag = ?
     step = 1000（每次批量取 1000 个）

  2. 应用在内存中自增分配，用完再取下一号段
     → DB 压力：QPS 降低 1000 倍

  3. 双 Buffer 优化：当前号段剩余 20% 时，异步预加载下一号段
     → 消除取号段时的延迟毛刺

限流与熔断

限流（令牌桶 / 漏桶 / 滑动窗口）保护服务不被流量压垮，熔断（CLOSED→OPEN→HALF-OPEN 状态机）在下游异常时快速失败，防止级联雪崩。

四种限流算法对比

算法原理突发流量适用场景
固定窗口统计固定时间窗口内的请求数，超过阈值拒绝窗口边界处可放双倍流量简单限流，精度要求不高
滑动窗口将窗口细分为多个子窗口，滚动统计，避免边界问题处理较好，无边界突刺接口级限流（Sentinel 使用）
漏桶请求入桶，桶以固定速率出水；桶满则丢弃不允许，强制平滑输出平滑流量，保护下游固定速率处理
令牌桶按固定速率向桶中放令牌；请求需取到令牌才能通过允许（桶内令牌积累可应对突发）允许突发但有上限（RateLimiter）

令牌桶算法流程

  ┌──────────────────────────────────────────────────────┐
  │  令牌生成器：每隔 1/rate 秒放入 1 个令牌               │
  │  桶容量：burst（最多积累 burst 个令牌）                 │
  └─────────────────────┬────────────────────────────────┘
                        │ 按速率持续放令牌
                        ▼
                 ┌────────────┐
                 │   令牌桶   │  容量 = burst
                 │  ●●●●●○○○ │
                 └─────┬──────┘
                       │
          请求到来 ─────►  令牌充足？
                             ┌──┴──┐
                           YES    NO
                            │      │
                        获取令牌  丢弃 / 排队等待
                        通过请求

熔断器状态机

  ┌───────────┐  错误率超阈值   ┌──────────┐
  │  CLOSED   │ ─────────────► │  OPEN    │
  │（正常计数）│                │（快速失败）│
  └───────────┘                └────┬─────┘
        ▲                           │ 超过等待时间（如 5s）
        │                           ▼
        │               ┌──────────────────┐
        │ 探测成功        │   HALF-OPEN      │
        └──────────────── │ （放行少量请求探测）│
          / 失败→OPEN      └──────────────────┘
CLOSED 正常状态，统计最近时间窗口内的成功/失败率；失败率超过阈值（如 50%）则切换到 OPEN
OPEN 熔断打开，所有请求直接快速失败（fallback），不访问下游；持续一段时间后进入 HALF-OPEN
HALF-OPEN 半开探测，放行少量请求（如 10%）到下游；若成功率达标则恢复 CLOSED，否则回到 OPEN
主流框架：Sentinel 使用滑动窗口统计 + 令牌桶限流，支持 QPS、并发线程数、热点参数等多维度；Resilience4j 支持基于计数和时间窗口的熔断器，轻量无侵入；Hystrix（已停维）曾是 Netflix 标准方案，线程隔离 + 信号量隔离。

一致性哈希

一致性哈希将节点和数据都映射到同一哈希环，节点变动只影响相邻数据，结合虚拟节点解决数据倾斜，广泛用于分布式缓存和负载均衡。

传统取模哈希的缺陷

  hash(key) % N   →   N 变化时，几乎所有 key 的映射都改变！

  3 个节点：key → hash % 3 = {0, 1, 2}
  扩容到 4 个节点：key → hash % 4 = {0, 1, 2, 3}

  假设 key 的 hash 值为 7：
    3 节点：7 % 3 = 1  (Node1)
    4 节点：7 % 4 = 3  (Node3)  ← 映射完全改变

  结论：N 节点扩/缩时，约 (N-1)/N 的缓存失效，引发缓存雪崩

一致性哈希环原理

  哈希空间：[0, 2³²-1] 首尾相连形成环

                    0
          ┌─────────●──────────┐
          │  NodeA(hash=10) ●  │
          │                    │
  2³²-1 ──●                    ●── NodeB(hash=80)
          │                    │
          └─────────●──────────┘
               NodeC(hash=200)

  Key 定位规则：hash(key) 在环上顺时针查找，遇到第一个 Node 即为目标节点

  示例：hash(keyX)=50 → 顺时针找到 NodeB(80) → 存储在 NodeB
        hash(keyY)=5  → 顺时针找到 NodeA(10) → 存储在 NodeA

  新增/删除节点只影响该节点到前驱节点之间的数据（约 1/N）

虚拟节点解决数据倾斜

  问题：节点少时分布不均，一个节点承载大量数据（数据倾斜）

  解决：每个物理节点映射多个虚拟节点分散在环上

  NodeA → NodeA#1, NodeA#2, NodeA#3 ... NodeA#150
  NodeB → NodeB#1, NodeB#2, NodeB#3 ... NodeB#150
  NodeC → NodeC#1, NodeC#2, NodeC#3 ... NodeC#150

  总共 450 个虚拟节点均匀分布在环上
  → 每个物理节点平均承担约 1/3 的数据，负载均衡
Redis Cluster 使用 16384 个哈希槽（slot）分片，节点间分配 slot 范围，扩容时只迁移部分 slot
Nginx 负载均衡 upstream 配置 ip_hash 或一致性哈希，保证同一客户端请求路由到同一后端（会话保持）
分布式缓存路由 Memcached 客户端（ketama 算法）、Dubbo 负载均衡均采用一致性哈希，缩小扩缩容时的缓存雪崩范围

分布式高频面试题

Q1 CAP 理论是什么？分布式系统为什么不能同时满足三者？ +
  • C（一致性）：所有节点同一时刻读到相同数据；A（可用性）：每个请求都能收到响应；P（分区容忍性）：网络分区时系统仍可运行。
  • P 在分布式系统中不可放弃——网络故障是客观存在的，真正的选择只在 C 和 A 之间。
  • 网络分区时：选 C（CP）则拒绝写请求直到节点同步，牺牲可用性；选 A（AP）则立即返回可能过期的数据，牺牲一致性。
  • 典型 CP：ZooKeeper、etcd（选主期间不可用）；典型 AP：Eureka（自我保护，不剔除实例）、Cassandra。
Q2 BASE 理论是什么？和 ACID 有什么区别？ +
  • BA（基本可用）：系统故障时允许降级，核心功能仍可用（如双十一限流降级、展示兜底数据）。
  • S（软状态）：允许中间状态存在，节点间数据同步有延迟，不要求实时一致。
  • E（最终一致性）：无新更新时，经过一段时间所有副本最终达到一致。
  • 与 ACID 对比：ACID 强调强一致性（事务内立即一致），适合金融/订单；BASE 强调高可用优先，适合社交/推荐等高并发场景。
  • BASE 不是"随便不一致"，需通过对账、重试补偿、版本号等机制保证最终收敛到一致。
Q3 分布式事务有哪些解决方案？各自适用什么场景？ +
  • 2PC：强一致，性能差（同步阻塞），业务侵入低，适合 DB 层 XA 协议；缺点是单点故障和数据不一致风险。
  • TCC（Try-Confirm-Cancel）：最终一致，性能好，业务侵入性强（需实现 3 个接口）；适合资金、库存等核心业务；需处理空回滚/悬挂/幂等三大异常。
  • Saga：将长事务拆成本地事务序列，每步立即提交，失败时逆向补偿；适合订单、物流等长流程业务。
  • 本地消息表：业务操作与消息写入同一本地事务，定时任务扫描发送 MQ；对业务侵入最小，适合跨服务异步通知。
  • 事务消息（RocketMQ）：生产者发送半消息，本地事务执行后提交/回滚，Broker 定期回查；适合生产者与消费者解耦场景。
Q4 2PC 的原理是什么？有哪些缺陷？ +
  • 同步阻塞：Prepare 阶段参与者持有锁等待协调者指令，高并发下锁竞争严重，吞吐量极低。
  • 单点故障：协调者宕机后，参与者将永久阻塞并持锁无法释放，系统僵死。
  • 数据不一致：Phase 2 协调者发出部分 Commit 后宕机，已提交和未提交的参与者数据出现分歧。
  • 过于保守：任一参与者返回 No，整个事务回滚；网络抖动可能导致误判，降低成功率。
  • 3PC 通过增加 CanCommit 阶段和超时机制部分缓解单点问题，但实现复杂，生产中极少使用。
Q5 TCC 的原理是什么？和 2PC 相比有什么优缺点？ +
  • 无全局锁：TCC 的 Try 阶段只预留资源不加全局 DB 锁，Confirm/Cancel 完成实际提交，并发性能远优于 2PC。
  • 业务侵入性：TCC 需要为每个服务实现 Try/Confirm/Cancel 三个接口，开发成本高；2PC 由 DB 层（XA 协议）自动处理，对业务透明。
  • 三大异常必须处理：
    • 空回滚：Try 未收到（网络超时），Cancel 先到达，需判断 Try 是否执行过再决定是否回滚
    • 悬挂：Cancel 先执行，Try 才到达，需拒绝后来的 Try（事务已结束）
    • 幂等：Confirm/Cancel 网络重试，需保证重复执行结果相同
  • 实践中推荐用 Seata TCC 模式，框架层处理幂等和空回滚，减少手动实现负担。
Q6 如何用 Redis 实现分布式锁？有哪些常见问题？ +
  • 加锁：SET lock_key ${uuid} NX PX 30000——NX 保证只有一个客户端能设置成功，PX 设置过期时间防止死锁，value 存 UUID 标识持锁方。
  • 释放锁：必须用 Lua 脚本原子执行"判断 value 相等 + DEL"，不能直接 DEL——否则可能误删其他线程的锁（A 锁过期，B 获锁，A 此时 DEL 会删 B 的锁）。
  • Redisson：封装了 SET NX + Lua 删除 + 看门狗续期，支持可重入锁、公平锁、读写锁，生产推荐使用。
  • 常见问题：锁过期业务未完成（看门狗解决）、主从切换锁丢失（RedLock 解决）、无法重入（Redisson 用 hash 结构记录重入次数）。
Q7 Redisson 的看门狗机制是什么？如何防止锁提前过期？ +
  • 看门狗（WatchDog）：调用 lock() 时如果不指定 leaseTime，Redisson 默认 leaseTime = 30s，同时启动后台 WatchDog 线程。
  • 续期机制：WatchDog 每隔 leaseTime / 3（默认 10s） 检查锁是否仍被持有，若是则重置过期时间为 30s，防止业务未完成时锁自动过期。
  • 停止续期：调用 unlock() 后，Redisson 删除锁并取消 WatchDog 定时任务，不再续期。
  • 注意：若指定了 leaseTime（如 lock(10, TimeUnit.SECONDS)），看门狗不会启动，锁在指定时间后自动过期，需自行估算业务执行时间。
  • JVM 进程崩溃时 WatchDog 线程也消失，锁会在 30s 后自动过期，不会死锁。
Q8 ZooKeeper 如何实现分布式锁？和 Redis 分布式锁有什么区别？ +
  • 实现步骤：① 在 /locks/ 下创建临时顺序节点；② 获取所有子节点并排序；③ 若自己是序号最小的节点则获锁；④ 否则 Watch 前一个节点（非 Watch 所有节点）。
  • 自动释放：持锁方宕机后会话断开，ZK 自动删除临时节点，触发后继节点的 Watch，不会死锁。
  • 羊群效应优化：Watch 前驱节点而非所有节点，避免节点释放时唤醒所有等待者（N-1 次无效唤醒）。
  • Redis vs ZK 对比：Redis 性能高（AP），ZK 可靠性强（CP）；Redis 主从切换可能锁丢失，ZK 临时节点无此问题；ZK 吞吐低，不适合高并发锁竞争。
Q9 Nacos、Eureka、ZooKeeper 作为注册中心有什么区别？如何选型？ +
  • Eureka（AP）：Spring Cloud 早期标配，自我保护机制（15min 内心跳丢失 >15% 不剔除）保障可用性，但已停止维护，新项目不推荐。
  • Nacos（AP/CP 可切换）：阿里开源，服务注册默认 AP（高可用），配置中心用 CP（强一致）；支持推送 + 健康检查，生态完善，国内首选。
  • ZooKeeper（CP）：Leader 选举期间服务不可用，服务发现场景下可用性差，适合做分布式锁/配置协调而非注册中心。
  • Consul（CP）：支持多数据中心，内置 DNS 服务发现，适合云原生和 K8s 场景。
  • 选型建议：国内 Spring Cloud 项目首选 Nacos；K8s 场景可用 Consul 或直接使用 etcd；强一致配置管理用 etcd/ZK。
Q10 Snowflake 算法的原理是什么？有哪些缺陷？ +
  • 结构（64 bit）：1位符号位（固定0）+ 41位时间戳（ms，可用约69年）+ 10位机器ID（5位datacenter + 5位workerId，最多1024台）+ 12位序列号（同毫秒内最多4096个ID）。
  • 特点：本地生成无网络依赖，每毫秒可生成 4096 × 1024 ≈ 420万个 ID；趋势递增（时间戳在高位），对 B+树友好。
  • 时钟回拨问题：NTP 校正导致系统时间倒退，若直接使用当前时间戳会生成重复 ID。
  • 解决方案：① 回拨量小（<5ms）时 spin 等待追上；② 回拨量大时抛异常由上层重试；③ 美团 Leaf 借用扩展位标记回拨，继续生成不重复 ID。
  • 实践中需保证机器ID不重复，可从 ZK/DB 动态分配；K8s 环境中 Pod 重启可能导致机器ID复用，需额外处理。
Q11 如何实现接口限流？令牌桶和漏桶算法有什么区别？ +
  • 令牌桶：以固定速率向桶中放令牌（容量=burst），请求消费令牌；桶满时令牌积累，可应对突发流量（允许瞬间通过 burst 个请求）。适合：API 限流、允许短暂突发的场景（Guava RateLimiter）。
  • 漏桶：请求进入漏桶排队，桶以固定速率漏出（处理）；桶满则丢弃新请求，强制平滑输出，无论输入多大，输出速率恒定。适合：保护下游固定处理速率的服务。
  • 核心区别：令牌桶允许突发（积累令牌），漏桶绝对平滑（固定输出）；接口保护通常用令牌桶（兼顾弹性和保护），消息队列消费速率控制用漏桶。
  • 固定窗口问题：窗口边界处可能放过双倍流量（前一窗口末尾 + 后一窗口开头）；滑动窗口通过细分子窗口解决此问题，Sentinel 默认使用。
Q12 什么是服务熔断？熔断器的三种状态是什么？ +
  • CLOSED（关闭）：正常工作状态，请求正常通过，同时统计最近时间窗口内的成功/失败率；当失败率超过阈值（如50%）或慢调用比例过高时，切换到 OPEN。
  • OPEN（打开）：熔断打开，所有请求直接快速失败（执行 fallback 逻辑），不再调用下游，防止级联雪崩；持续一段时间（如5秒）后自动进入 HALF-OPEN。
  • HALF-OPEN（半开）：探测恢复阶段，放行少量请求（如10%）到下游；若成功率达标则恢复 CLOSED，若仍有大量失败则回到 OPEN。
  • 熔断的意义：下游服务故障时快速失败，避免请求堆积耗尽线程池资源，防止级联雪崩（一个服务故障拖垮整个调用链）。
  • Sentinel vs Resilience4j：Sentinel 适合国内微服务，规则配置丰富；Resilience4j 轻量，适合 Spring Boot 项目，两者均支持三状态熔断器。
Q13 一致性哈希算法是什么？如何解决数据倾斜问题？ +
  • 哈希环：将 [0, 2³²-1] 的哈希空间首尾相连构成环，节点和 key 都映射到环上；key 沿顺时针方向找到第一个节点即为目标节点。
  • 优势：新增/删除节点只影响该节点到前驱节点之间的 key（约 1/N），其余 key 映射不变，极大减少缓存失效范围（传统取模则约 (N-1)/N 失效）。
  • 数据倾斜问题：节点少时分布不均，某节点可能承载大量数据；解决方案：虚拟节点——每个物理节点映射 150+ 虚拟节点均匀散布在环上，负载均衡。
  • 应用场景：Redis Cluster（16384槽的变种实现）、Memcached ketama 算法、Nginx 一致性哈希负载均衡、Dubbo 集群路由。
Q14 分布式系统中如何保证接口幂等性？ +
  • 唯一请求 ID：客户端生成全局唯一 requestId，服务端查 Redis/DB 判断是否已处理；已处理则直接返回上次结果。适合：支付、下单。
  • Token 机制：请求前先获取一次性 Token（存 Redis），提交时携带 Token，服务端原子判断+删除 Token（Lua）；Token 不存在则拒绝，防止表单重复提交。
  • 数据库唯一索引：在业务唯一字段（如 order_no）上建唯一索引，重复插入抛 DuplicateKeyException 并捕获返回成功。适合写入类幂等。
  • 状态机：通过状态流转（如 PENDING→PAID）保证幂等，已流转则不再重复处理；UPDATE ... WHERE status='PENDING' 乐观锁控制。
  • 乐观锁版本号：更新时携带 version，UPDATE ... WHERE id=? AND version=?，并发更新时只有一个成功。
Q15 什么是分布式链路追踪？TraceId 如何实现全链路透传？ +
  • 核心概念：TraceId 标识整条请求链路（全局唯一），SpanId 标识链路中的单个操作（父子关系组成调用树），一个 Trace 包含多个 Span。
  • MDC 透传：利用 MDC.put("traceId", traceId) 将 TraceId 存入线程本地变量，日志框架自动打印；跨线程需手动传递（线程池场景用 TransmittableThreadLocal）。
  • 跨服务透传：HTTP 调用时将 TraceId 放入请求头（如 X-Trace-Id），下游服务通过拦截器/Filter 提取并放入 MDC；MQ 场景放入消息属性。
  • Skywalking：字节码增强（Java Agent），无侵入，自动采集 TraceId/SpanId 并上报；Zipkin 需要业务代码集成（Spring Cloud Sleuth）。
  • 链路追踪解决分布式系统下日志分散的问题，结合 ELK/Jaeger 可以按 TraceId 聚合完整调用链，快速定位性能瓶颈和错误根因。
