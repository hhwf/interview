分布式系统核心知识

分布式系统将单体应用拆分为多个独立服务，带来水平扩展能力的同时，也引入了网络分区、数据一致性、服务协调等新挑战。 掌握 CAP/BASE 理论、分布式事务、分布式锁等核心原理，是设计高可用系统和通过大厂面试的必备基础。

CAP & BASE 理论

CAP 定理指出分布式系统不能同时满足一致性（C）、可用性（A）、分区容忍性（P），只能三选二。BASE 是对 ACID 的弱化，通过牺牲强一致性换取高可用与高性能。

CAP 三要素

C — 一致性 Consistency：所有节点在同一时刻看到相同的数据。写操作完成后，所有后续读都能读到最新值（强一致性）。
A — 可用性 Availability：每个请求都能收到响应（不一定是最新数据），系统不会拒绝服务。
P — 分区容忍性 Partition tolerance：网络分区（节点间通信中断）发生时，系统仍能继续运行。分布式系统中 P 几乎不可放弃。
面试要点：CAP 定理的核心是：分布式系统在网络分区（P）发生时，只能在一致性（C）和可用性（A）中选一个。实践中 P 不可避免，所以真正的选择是 CP 还是 AP。CP 代表：ZooKeeper、etcd（优先一致性，分区时拒绝服务）。AP 代表：Eureka、Cassandra（优先可用性，允许短暂不一致）。BASE 理论是 AP 系统的理论基础：基本可用 + 软状态 + 最终一致。

为什么只能三选二？

网络分区发生（P 必须保留）— 节点 A 与节点 B 无法通信

CP vs AP — 典型系统选型

类型代表系统分区时行为适用场景
CPZooKeeper、etcd、HBase、Consul拒绝服务，保证数据一致分布式锁、选主、配置中心（数据不能错）
APEureka、Cassandra、DynamoDB、CouchDB继续服务，可能返回旧数据服务注册、购物车、DNS（可用性优先）
CA单机关系型数据库（MySQL 单节点）不存在网络分区问题单体应用（无法水平扩展）

BASE 理论

BASE 是 AP 系统的设计指导原则，牺牲强一致性换取高可用：

BA — 基本可用 Basically Available：系统出现故障时，允许损失部分可用性（如延迟增加、降级返回默认值），但核心功能仍可用。
S — 软状态 Soft state：允许系统中存在中间状态，数据在节点间同步存在延迟，不要求节点时刻一致。
E — 最终一致性 Eventually consistent：系统保证在没有新的更新操作后，经过一段时间，所有副本最终达到一致状态。
维度ACID（传统数据库）BASE（分布式系统）
一致性强一致性（事务内立即一致）最终一致性（允许短暂不一致）
可用性牺牲可用性保证一致性优先可用性
适用金融、订单（数据强一致要求）社交、推荐、购物车（高并发优先）
面试要点：实际系统中 P 不可放弃，真正的选择是 CP 还是 AP。Nacos 默认 AP 模式但可切换 CP；ZooKeeper 是 CP；Eureka 是 AP。"最终一致性"不等于"随便不一致"，需要通过补偿机制（重试、对账）保障。

分布式事务

跨服务 / 跨库操作无法依赖本地事务，需要 2PC、TCC、Saga、本地消息表等方案在一致性与可用性之间取得平衡。

主流方案对比

方案一致性性能业务侵入适用场景
2PC强一致低（阻塞）低数据库层（XA 协议）
TCC最终一致高高（需实现 3 接口）资金、库存等核心业务
Saga最终一致高中（需编写补偿）长流程业务（订单、物流）
本地消息表最终一致高低跨服务异步通知

面试要点：分布式事务的选型原则：能不用就不用（业务合并 / 接受最终一致 / 转为幂等补偿）。需要强一致时：Seata AT（无侵入，自动 undo log）适合大多数场景；TCC（强侵入，需手写三步）适合对一致性要求极高的金融场景；本地消息表 / 事务消息适合跨服务最终一致；Saga 适合长流程多步骤。面试核心：必须能画出具体的事务执行和补偿流程。

分布式锁

多节点并发写同一资源时需要分布式锁协调，主流方案包括基于 Redis（Redisson 看门狗）和 ZooKeeper（临时顺序节点）两种路线。

为什么需要分布式锁？

单机环境下 synchronized / ReentrantLock 只在同一 JVM 进程内生效，多实例部署后进程间无法共享内存，需要借助第三方存储（Redis、ZK）实现跨进程互斥。

Redis 方案

  // 1. 加锁（原子操作）
  SET lock_key ${uuid} NX PX 30000
      └── NX: 不存在时才设置   PX: 30000ms 自动过期

  // 2. 释放锁（Lua 脚本保证原子性）
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])   -- 只删自己的锁
  else
      return 0
  end
  -- value 存 uuid，防止误删其他线程的锁
面试要点：分布式锁选型：Redis 锁（高性能，但需注意锁超时、主从切换导致的非互斥）适合高并发场景；ZooKeeper 锁（强一致，基于 ephemeral + sequential node）适合对一致性要求高的场景。Redisson 的 WatchDog 解决锁续期问题（自动续 leaseTime，任务完成后取消）。RedLock 算法的争议：Antirez 提出，Martin Kleppmann 认为 GC Pause 等场景下仍有安全问题，实践中慎用。
SET NX PX 单条命令原子加锁，NX 防重入，PX 设置过期时间防死锁
Lua 脚本删锁 get + del 非原子，必须用 Lua 脚本保证"判断+删除"原子性，避免误删他人的锁
Redisson 看门狗 默认 leaseTime=30s，后台 WatchDog 每 10s（leaseTime/3）自动续期；调用 unlock() 后停止续期并删除锁
RedLock 向 N 个独立 Redis 实例加锁，超过半数（N/2+1）成功即视为获锁成功，提升可靠性（了解即可）

Raft 共识算法

Raft 是一种易于理解的分布式共识算法，etcd、TiKV、Nacos CP 模式、Sentinel 选举均基于它。相比 Paxos，Raft 通过 Leader 唯一化将共识问题拆解为三个独立子问题：Leader 选举、日志复制、安全性保证。

三种节点角色

Leader 同一时刻集群只有一个 Leader。接收所有客户端写请求、复制日志到 Follower、定期发心跳防止其他节点超时选举
Follower 被动接收 Leader 的 AppendEntries 和心跳。Election Timeout 内未收到心跳则转为 Candidate 发起选举
Candidate 选举过渡态。广播 RequestVote RPC，获多数票成为 Leader；收到更高 term 的消息退回 Follower
面试要点：Raft 三个核心问题：① 领导选举：term + 随机选举超时，超时先投自己，获多数票成 Leader；② 日志复制：Leader 接收请求，并发复制到 Follower，超半数确认后 commit 并回复客户端；③ 安全性：只有日志最新的节点才能当选 Leader（防止已提交日志丢失）。与 Paxos 的区别：Raft 更易理解，明确分离选举/日志/快照三个子问题。

① Leader 选举

Follower Election Timeout 到期（随机 150–300 ms，避免同时超时）
↓
转为 Candidate，currentTerm++，先投票给自己
↓
并发广播 RequestVote RPC（携带自身 term 和最后日志的 index/term）
↓
获多数票 (> N/2)
成为 Leader，立即广播心跳占位
收到同 term 其他 Leader
退回 Follower
选举超时（分票）
随机等待后重新选举

投票安全规则：只投给日志"至少一样新"的 Candidate——先比较最后日志的 term，term 相同再比 index。这保证了新 Leader 一定包含所有已提交的日志（至少在多数节点上存在的日志 = 已提交）。

③ 安全性保证

Leader 完整性 新 Leader 必定包含所有已提交日志。已提交 = 写入多数节点，选举要赢多数票 → 交集至少 1 个节点有已提交日志 → 投票规则拒绝日志落后的 Candidate
日志匹配 两份日志在同一 (index, term) 位置相同，则该位置前所有条目完全一致（AppendEntries 的一致性检查递归保证）
任期（Term） 单调递增的逻辑时钟。节点收到更大 term 的消息立即退回 Follower 并更新 term，旧 Leader 发出的过期命令被识别并丢弃

Raft vs Paxos 一句话对比

Paxos 解决单值共识（Single-Decree Paxos），理论严谨但工程化复杂；Multi-Paxos 没有规范实现，各家实现差异大（Google Chubby、Spanner 均做了大量扩展）
Raft 直接面向复制状态机和日志，Leader 唯一简化了并发问题；工程实现明确（etcd、TiKV、CockroachDB、Nacos、HashiCorp Consul 均采用）；代价是 Leader 成为写瓶颈

分布式 ID

分布式环境下数据库自增 ID 无法保证全局唯一，需要 Snowflake、号段模式、Redis incr 等方案生成全局唯一、趋势递增的 ID。

分布式 ID 的核心要求

全局唯一 同一业务场景下 ID 不重复，是基本要求
趋势递增 MySQL B+ Tree 的叶节点顺序写入时性能最优，随机 ID（UUID）导致页分裂，写性能下降
高性能 每秒生成百万级 ID，不能成为系统瓶颈
高可用 ID 生成服务不能单点，需容灾和降级方案
面试要点：Snowflake ID 结构：1 bit 符号位 + 41 bit 时间戳（ms，约 69 年）+ 10 bit 机器 ID + 12 bit 序列号（每 ms 最多 4096 个）。时钟回拨是最大风险：需要处理（等待 / 拒绝请求 / 使用备用序列号）。百度 UidGenerator / 美团 Leaf 等均是 Snowflake 的改进版，主要解决时钟回拨和机器 ID 分配问题。号段模式（DB 预取一段 ID）适合对时间无序要求不高的场景。

时钟回拨问题与解决方案

问题 NTP 时间校正导致系统时钟倒退，若用当前时间戳生成 ID 会出现重复
方案1：抛异常 检测到时钟回拨直接抛异常，由上层业务重试（简单直接，适合回拨量小的场景）
方案2：等待追上 回拨量小（<5ms）时 spin 等待时钟追上，回拨量大时抛异常
方案3：借用序列号 美团 Leaf 做法：回拨时用扩展位标记，借用未来时间的序列号空间生成 ID

限流与熔断

限流（令牌桶 / 漏桶 / 滑动窗口）保护服务不被流量压垮，熔断（CLOSED→OPEN→HALF-OPEN 状态机）在下游异常时快速失败，防止级联雪崩。

四种限流算法对比

算法原理突发流量适用场景
固定窗口统计固定时间窗口内的请求数，超过阈值拒绝窗口边界处可放双倍流量简单限流，精度要求不高
滑动窗口将窗口细分为多个子窗口，滚动统计，避免边界问题处理较好，无边界突刺接口级限流（Sentinel 使用）
漏桶请求入桶，桶以固定速率出水；桶满则丢弃不允许，强制平滑输出平滑流量，保护下游固定速率处理
令牌桶按固定速率向桶中放令牌；请求需取到令牌才能通过允许（桶内令牌积累可应对突发）允许突发但有上限（RateLimiter）
面试要点：限流算法对比：① 令牌桶：允许突发流量（桶内有令牌就放行），适合接口限流；② 漏桶：严格匀速输出，适合平滑流量；③ 滑动窗口：比固定窗口更精确，防止边界突刺。熔断状态机：Closed（正常）→ Open（快速失败）→ Half-Open（放少量请求探测）→ Closed。Sentinel vs Hystrix：Sentinel 基于滑动窗口计数，支持 QPS / 并发数多种规则，功能更丰富；Hystrix 已停维护，推荐迁移 Sentinel。

一致性哈希

一致性哈希将节点和数据都映射到同一哈希环，节点变动只影响相邻数据，结合虚拟节点解决数据倾斜，广泛用于分布式缓存和负载均衡。

面试要点：一致性哈希解决了取模分片的扩容迁移量大问题：增减节点时只影响相邻节点的数据，迁移量 ≈ 1/N（N 为节点总数）。虚拟节点（每个物理节点映射多个虚拟节点）解决数据倾斜问题，使负载更均匀。实际生产中，Consistent Hash 广泛用于 Redis Cluster 的槽位分配、分库分表路由、Nginx upstream 负载均衡。
