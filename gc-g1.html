G1 Garbage Collector
深度解析

深入剖析 G1 GC 的核心设计——Region 分区模型、SATB 并发标记、Remembered Set 跨区引用、Collection Set 选择策略、Evacuation 复制回收、Young GC 并行机制，以及完整 GC 流程与调优参数。

● G1 Garbage Collector

G1 —— Garbage First

G1 是 JDK 9+ 的默认收集器，目标是在 有限停顿时间内 尽可能回收垃圾最多的区域， 兼顾吞吐量与低延迟，适合 4GB–64GB 的中大型堆。

堆分区模型（Region）

G1 将整个堆划分为大小相等的 Region（1~32MB，始终为 2 的幂）， 数量约为 2048 个。Region 的角色不固定——同一块物理内存可以在不同 GC 周期中充当 Eden / Survivor / Old / Humongous，这是 G1 灵活调度的基础。 超过 Region 50% 大小的对象直接分配在连续的 Humongous Region 中。

并发标记 — SATB 算法

SATB（Snapshot-At-The-Beginning）是 G1 并发标记的核心正确性保障。 它建立在三色标记算法之上，通过"标记开始时对引用关系拍快照" 加上写前屏障来解决并发标记期间应用线程修改引用可能导致的漏标问题。

三色不变式：黑色对象不能直接引用白色对象—— 因为黑色对象已完成扫描，不会再被访问，若它指向白色对象，该对象将永远无法被标记为灰色，最终被错误回收。
面试要点：SATB（Snapshot At The Beginning）是 G1 并发标记的核心算法。并发标记期间，应用线程修改对象引用时，写屏障会将旧引用值记入 SATB 队列，确保「快照时刻活着的对象」不会被漏标（避免黑色对象指向白色对象且灰色对象失去该引用的场景）。与增量更新（CMS/G1 Final Remark 用）的区别：SATB 记录删除边，增量更新记录新增边。

漏标的两个必要条件（缺一不可）：

条件② 所有通往该白色对象的灰色路径被删除（B.next = null）
破坏任意一个条件，漏标就不会发生。SATB 选择破坏条件②。
③ SATB 快照解法——记录被删除的引用
SATB 的核心思路：标记开始时所有可达对象构成一个逻辑快照， GC 保证快照中的每个对象本轮都不会被错误回收。 当应用线程删除一条引用时，写前屏障把被删除的旧引用值记录下来—— 这样即使灰色路径断开，目标对象依然能在 Remark 阶段被重新扫描。

⚠
浮动垃圾（Floating Garbage）——SATB 的代价：
并发标记期间若对象 X 变为不可达，但快照中 X 曾经是可达的，SATB 会把旧引用记录下来确保 X 不被漏标。 结果是：X 在本轮 GC 被认为存活，不会被回收，成为"浮动垃圾"，等待下一次 GC 清除。 通常浮动垃圾占堆的 0~5%，是 SATB 换取正确性和短暂 Remark 停顿的代价。

Remembered Set（RSet）

G1 每次只回收部分 Region，但需要知道"从其他 Region 是否有引用指向本 Region 的对象"。 RSet 就是这个索引：每个 Region 维护一个 Card Table 集合， 记录哪些外部 Card（512 字节内存块）持有指向本 Region 的引用。 维护由 写后屏障（Post-Write Barrier） + Dirty Card Queue 异步完成。

面试要点：RSet（Remembered Set）记录「哪些外部 Region 的对象引用了本 Region」，使 GC 时只扫描 RSet 而不必扫描全堆。代价：每次引用写操作都要更新 RSet（通过 dirty card queue → Refinement 线程异步维护），有写屏障开销。RSet 本身也占内存（典型 1–5% 堆空间）。大 Region 的 RSet 条目太多时触发 humongous evacuation failure。
⚠
RSet 的代价：每个 Region 的 RSet 占用额外内存（通常堆的 1%~20%）； 每次写操作都触发写后屏障更新 RSet；Dirty Card Queue 消耗 CPU。 Region 越多、跨 Region 引用越密集，RSet 开销越高。

Collection Set（CSet）与 Garbage First

并发标记后，G1 已知每个 Region 的 存活率。 "Garbage First" 名字正来源于此——按 垃圾占比 × 回收价值 排序， 优先回收垃圾最多的 Region，在停顿时间预算内最大化回收效率。 被选入 CSet（Collection Set）的 Region，其所有存活对象将被复制走，Region 整体释放。

Evacuation（疏散复制）

Evacuation 是 G1 真正回收内存的步骤，全程 STW。 CSet 中所有存活对象被并行复制（Parallel Copy）到空 Region， 原 Region 整体清空——这就避免了碎片化，G1 始终使用整块 Region。 若 to-space 不足，则发生 Evacuation Failure，退化为类 Full GC。

Young GC 机制详解

Young GC 是 G1 最频繁执行的 GC 类型，全程 STW、多线程并行。 触发条件是 Eden Region 被填满，目标是回收所有 Young Region（Eden + Survivor）， 将存活对象复制到新 Survivor Region 或晋升至 Old Region。 整个过程以 Evacuation（疏散复制） 为核心，没有标记位清扫，只有对象复制。

💡
TLAB（Thread-Local Allocation Buffer）： 每个线程持有一小块独占的 Eden 内存，对象在 TLAB 内分配无需任何锁，速度接近栈分配。 TLAB 用尽时才从 Eden 申请新的 TLAB（需短暂锁定），只有 Eden Region 全满时才触发 Young GC。 大对象（> TLAB 剩余空间一半）直接在 Eden Region 中 CAS 分配，绕过 TLAB。

G1 完整 GC 流程

G1 的 GC 生命周期由三种模式交替构成，以老年代占堆比例作为主要驱动信号。 Young GC 在 Eden 填满时持续发生；当 Old 占比触达 IHOP 阈值， 下一次 Young GC 会"捎带"启动并发标记（Initial Mark 搭车）； 标记完成后连续执行若干轮 Mixed GC 逐步消化老年代； 只有内存耗尽且 GC 跟不上分配速率时，才退化为代价极高的 Full GC。

⚠
Full GC 退化（应极力避免）
以下两种情况会触发 Full GC，G1 退化为单线程 Serial Old，对整个堆做压缩整理，停顿可达秒级：
  • ▸Evacuation Failure：Evacuation 时 Free Region 不足，存活对象无处复制。根因：晋升速率过快 / 大对象过多 / 堆占用率长期偏高。
  • ▸Concurrent Mode Failure：并发标记期间 Old 代已被填满，GC 来不及回收——此时触发 Full GC。根因：IHOP 设置过高 / 晋升速率远超并发标记速度。
规避手段：降低 -XX:InitiatingHeapOccupancyPercent（提前触发标记）、增加 -XX:G1ReservePercent（预留更多空 Region）、增大 -Xmx。

G1 GC 日志关键标志（-Xlog:gc*）：
  • GC pause (G1 Evacuation Pause) (young) → 正常 Young GC，STW 应 <200ms
  • GC pause (G1 Evacuation Pause) (mixed) → Mixed GC，Old Region 正在被回收
  • (to-space exhausted) → Evacuation Failure！立即检查 Free Region 数量和晋升速率
  • Full GC (Ergonomics) → G1 降级为 Serial Old Full GC，停顿可达秒级
  • GC concurrent-mark-start → 并发标记已启动，Old 代占比触达 IHOP 阈值
