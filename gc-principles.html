垃圾收集器原理

深入剖析三款 JVM 垃圾收集器的核心设计——G1 的分区回收与停顿控制、ZGC 的染色指针与亚毫秒停顿、CMS 的历史局限与设计演进，理解它们如何在不同约束下解决同一个问题。

● G1 Garbage Collector

G1 —— Garbage First

G1 是 JDK 9+ 的默认收集器，目标是在 有限停顿时间内 尽可能回收垃圾最多的区域， 兼顾吞吐量与低延迟，适合 4GB–64GB 的中大型堆。

堆分区模型（Region）

G1 将整个堆划分为大小相等的 Region（1~32MB，始终为 2 的幂）， 数量约为 2048 个。Region 的角色不固定——同一块物理内存可以在不同 GC 周期中充当 Eden / Survivor / Old / Humongous，这是 G1 灵活调度的基础。 超过 Region 50% 大小的对象直接分配在连续的 Humongous Region 中。

并发标记 — SATB 算法

SATB（Snapshot-At-The-Beginning）是 G1 并发标记的核心正确性保障。 它建立在三色标记算法之上，通过"标记开始时对引用关系拍快照" 加上写前屏障来解决并发标记期间应用线程修改引用可能导致的漏标问题。

Remembered Set（RSet）

G1 每次只回收部分 Region，但需要知道"从其他 Region 是否有引用指向本 Region 的对象"。 RSet 就是这个索引：每个 Region 维护一个 Card Table 集合， 记录哪些外部 Card（512 字节内存块）持有指向本 Region 的引用。 维护由 写后屏障（Post-Write Barrier） + Dirty Card Queue 异步完成。

Collection Set（CSet）与 Garbage First

并发标记后，G1 已知每个 Region 的 存活率。 "Garbage First" 名字正来源于此——按 垃圾占比 × 回收价值 排序， 优先回收垃圾最多的 Region，在停顿时间预算内最大化回收效率。 被选入 CSet（Collection Set）的 Region，其所有存活对象将被复制走，Region 整体释放。

Evacuation（疏散复制）

Evacuation 是 G1 真正回收内存的步骤，全程 STW。 CSet 中所有存活对象被并行复制（Parallel Copy）到空 Region， 原 Region 整体清空——这就避免了碎片化，G1 始终使用整块 Region。 若 to-space 不足，则发生 Evacuation Failure，退化为类 Full GC。

Young GC 机制详解

Young GC 是 G1 最频繁执行的 GC 类型，全程 STW、多线程并行。 触发条件是 Eden Region 被填满，目标是回收所有 Young Region（Eden + Survivor）， 将存活对象复制到新 Survivor Region 或晋升至 Old Region。 整个过程以 Evacuation（疏散复制） 为核心，没有标记位清扫，只有对象复制。

Work Stealing 机制
每个 GC 线程维护一个本地 对象处理队列（Task Queue）。复制一个对象后，其所有子引用被推入队列。当本地队列为空时，线程随机从其他线程的队列尾部窃取（steal）工作，保持所有核心饱和运行，消除线程间等待。
GCLAB 的作用
类似 TLAB，每个 GC 线程有独立的 GCLAB（来自目标 Survivor/Old Region）， 复制对象时直接 bump-pointer 分配，无竞争、无锁。 GCLAB 填满才向 Region 申请新块。
GC 线程数 默认 = min(8, CPU 核数 × 5/8)，可通过 -XX:ParallelGCThreads=N 调整。线程数越多停顿越短，但超过核数反而会因上下文切换退化。

G1 完整 GC 流程

G1 的 GC 生命周期由三种模式交替构成，以老年代占堆比例作为主要驱动信号。 Young GC 在 Eden 填满时持续发生；当 Old 占比触达 IHOP 阈值， 下一次 Young GC 会"捎带"启动并发标记（Initial Mark 搭车）； 标记完成后连续执行若干轮 Mixed GC 逐步消化老年代； 只有内存耗尽且 GC 跟不上分配速率时，才退化为代价极高的 Full GC。

以下两种情况会触发 Full GC，G1 退化为单线程 Serial Old，对整个堆做压缩整理，停顿可达秒级：

● Z Garbage Collector

ZGC —— 亚毫秒停顿

ZGC 以 亚毫秒级停顿 为核心目标，通过染色指针和读屏障 将几乎所有 GC 工作移到并发阶段，STW 仅做极少量的根标记。 适合对延迟极敏感的大堆（GB~TB 级）场景。

染色指针（Colored Pointer）

ZGC 的关键创新：将 GC 元数据直接编码进对象指针的 高位比特。 64 位系统实际寻址只用低 44 位（最大 16TB），剩余高位被 ZGC 借用存储 标记状态、重映射状态、Finalizable 状态。 这样 GC 无需额外的 Mark Bitmap，读指针即知对象 GC 状态。

● CMS Garbage Collector

CMS —— Concurrent Mark Sweep

CMS 是 G1 的前身，JDK 5u3 引入、JDK 9 废弃、JDK 14 正式移除。 它以并发清除老年代为核心目标，大幅降低老年代 GC 停顿， 但代价是内存碎片、Concurrent Mode Failure 与更高的 CPU 消耗。 理解 CMS 的局限是理解为什么设计 G1 的关键。

概览与传统分代堆布局

CMS 使用传统固定分代布局：Young 区（Eden + Survivor × 2）+ Old 区（单块连续内存）。 Old 区由 free-list 管理空闲块，分配时从列表中查找合适的空间——这是碎片化的根源。 与 G1 的可变 Region 模型形成鲜明对比。

从 GC Roots 出发，标记所有直接可达的老年代对象。
工作量 = GC Root 数量，停顿极短（毫秒级）。
面试要点：CMS 四阶段：Initial Mark（STW，标 GC Root 直接引用）→ Concurrent Mark（并发，不 STW）→ Remark（STW，修正并发期间变化，增量更新写屏障记录新增引用）→ Concurrent Sweep（并发清除，不压缩）。Remark 是最长 STW，可用 -XX:+CMSScavengeBeforeRemark 先触发 Minor GC 减少 Remark 扫描量。
② Concurrent Mark（并发）
从 Initial Mark 的结果出发，DFS 遍历整个老年代对象图。
与应用线程并发，可能漏标（需 Remark 修正）。
耗时最长，通常占整个 CMS GC 80% 以上时间。
③ Concurrent Preclean（并发）
扫描并发标记期间被写屏障标记为 Dirty 的 Card Table 条目。
提前处理部分引用变更，减轻 Remark 的工作量（减少 STW 时间）。
④ Remark（STW · 较长⚠）
使用增量更新（Incremental Update）处理并发标记期间的所有引用变更。
需要重扫所有被"重新变灰"的黑色对象的子树。
停顿时间随堆大小增长，远比 G1 的 Remark 更长（10ms ~ 数百ms）。
⑤ Concurrent Sweep（并发）
遍历老年代，将未标记对象占用的空间归还到 free-list（空闲链表）。
注意：只清除，不整理内存，碎片化从此产生。
⑥ Concurrent Reset（并发）
重置 CMS 的内部数据结构（Card Table、Bitmap 等），为下一轮 GC 做准备。
■ STW   ■ 并发（与应用线程同时运行）   ■ 并发（辅助/重置）
💡
CMS 触发时机：Old Gen 占用率达到阈值（-XX:CMSInitiatingOccupancyFraction，默认 92%）时启动并发标记周期。
默认阈值 92% 非常高，几乎没有余量——推荐调低到 70~75%，给并发阶段留出缓冲，避免 Concurrent Mode Failure。

卡表（Card Table）

Young GC 需要找出所有从 Old Gen 指向 Young Gen 的引用作为 GC Root。 若每次都全量扫描 Old Gen 则开销极大。 卡表将堆划分为 512 字节的小块（Card），用一个字节数组记录每块是否被修改， Young GC 时只扫描被标记为脏（dirty）的 Card，大幅缩小扫描范围。

维度 CMS 卡表 G1 RSet
记录方向 记录「谁被修改了」（哪个 Card 含脏引用） 记录「谁引用了我」（哪些外部 Card 指向本 Region）

适用场景 固定分代，只需区分 Young/Old 跨代引用 Region 可变，需精确追踪任意跨 Region 引用
G1 的 RSet 底层同样依赖卡表实现（写后屏障将修改的 Card 记入 Dirty Card Queue）， 再由 Refinement 线程将 Card 信息汇总到目标 Region 的 RSet 中——可以理解为 RSet 是卡表的精确化升级版。
Card 大小
固定 512 字节。4GB 堆 → 卡表仅 8MB，内存开销极小。
写后屏障
赋值后执行，计算 card_table[addr >> 9] = dirty，开销极低。
与增量更新的关系
卡表负责「记位置」（哪个 Card 脏了）；增量更新负责「修正标记」（Remark 时重扫灰色对象子树）。两者协作完成 Remark STW。
Young GC 加速
Minor GC 时仅扫描脏 Card，避免全量扫描 Old Gen，是分代 GC 高效运行的基础设施。

增量更新（Incremental Update）

CMS 使用写后屏障 + 增量更新解决并发标记漏标问题。 当一个黑色对象新增对白色对象的引用时，写后屏障将该黑色对象重新标记为灰色， Remark 阶段重新扫描其所有子引用——这正是 CMS Remark 停顿较长的根本原因。

SATB（G1）vs 增量更新（CMS）对比
维度 SATB（G1 使用） 增量更新（CMS 使用）
屏障时机 写前屏障，记录被覆盖的旧引用 写后屏障，记录新赋入的新引用目标
破坏漏标条件 破坏条件②：旧引用被删除时保留，防止灰色路径断开 破坏条件①：黑色对象新增白色引用后，将黑色重置为灰色重新扫描
Remark 停顿 短——只处理 SATB 队列（通常远小于全堆） 长——需重扫所有变灰的黑色对象的完整子树
浮动垃圾 较多（快照中存活对象本轮不回收） 较少（实时追踪新引用，更精准）
实现复杂度 相对简单，只记录一个旧值 复杂，需管理"重新变灰"的黑色对象集合
G1 选择 SATB 而非增量更新的核心原因：更短的 Remark STW 停顿，这与 G1 的低停顿目标高度一致。
面试要点：CMS 的并发阶段（Concurrent Mark / Concurrent Sweep）与应用线程同时运行，会与应用线程抢占 CPU（默认使用 (CPU数+3)/4 个线程），在 4 核以下机器上影响较明显。-XX:CMSInitiatingOccupancyFraction（默认 68%）控制何时触发 CMS，设太高会导致 Concurrent Mode Failure，设太低浪费 CPU。

三大核心问题

CMS 三大核心问题
① Concurrent Mode Failure（CMF）——最严重
触发：并发清除阶段，应用线程分配速度 > CMS 回收速度，Old Gen 在 GC 结束前被填满。
后果：CMS 中止，JVM 退化为 Serial Old 单线程 Full GC（STW + 整堆标记压缩），停顿时间可达秒级。
预防：调低 -XX:CMSInitiatingOccupancyFraction 到 70~75%（提前触发 CMS）；增大堆；优化应用分配热点。
面试要点：CMS 核心问题：① 浮动垃圾（并发标记/清除期间新产生的垃圾本轮无法回收）；② 碎片化（Mark-Sweep 不压缩，大对象分配失败时触发 Full GC）；③ Concurrent Mode Failure（老年代在并发阶段满了，退化 Serial Old Full GC，停顿极长）。这三个问题推动了 G1 的设计，G1 通过 Region + Evacuation 同时解决碎片和 CMF。
② Promotion Failure（晋升失败）
触发：Young GC 尝试将存活对象晋升到 Old Gen，但 Old Gen 的 free-list 找不到足够大的连续空间（即使 Old Gen 总空闲空间足够）。
后果：同样退化为 Full GC（Serial Old），这是内存碎片直接导致的灾难。
预防：定期触发 Full GC 整理（-XX:+UseCMSCompactAtFullCollection，已废弃）；最根本的解决是迁移到 G1。
③ 内存碎片（Memory Fragmentation）——慢性病
根因：Concurrent Sweep 只是将死亡对象标记为空闲归还 free-list，从不移动对象、从不整理内存（标记-清除算法本质）。
演化：随着应用运行，Old Gen 中出现越来越多的"小空洞"，大对象分配越来越难，free-list 查找开销增大，最终引发 Promotion Failure 或 CMF。
G1 的解法：Evacuation 阶段将存活对象复制到空 Region，回收后的 Region 整块释放，天然无碎片。这是 G1 相对 CMS 最根本的改进之一。
CMSInitiatingOccupancyFraction
Old Gen 触发阈值，默认 92%（过高）。推荐 70~75%，给并发阶段留余量。
+UseConcMarkSweepGC
启用 CMS。JDK 9 废弃，JDK 14 移除。现代应用请使用 G1 或 ZGC。
ConcGCThreads
并发 GC 线程数。过高会抢占应用 CPU，过低会使并发标记落后于分配速率。
历史地位
JDK 5u3 引入 → JDK 9 废弃 → JDK 14 正式移除。G1 成为默认收集器（JDK 9+）。
