G1 面试题

覆盖堆结构、并发标记、跨区引用、回收流程等核心考点。

← G1 GC 原理
G1·01 G1 的堆分区模型与传统的分代堆（Eden/Survivor/Old 固定区域）有什么本质区别？Humongous Region 的分配阈值是多少？ +
核心区别
传统分代堆各代占据物理上连续且固定的地址区间，Young/Old 区大小只能通过参数静态配置。G1 将堆切分为若干大小相等的 Region（1~32 MB，2 的幂），每个 Region 的"角色"（Eden / Survivor / Old / Humongous）在 GC 周期间可以变换，GC 可以根据停顿时间目标动态调整 Young 区 Region 数量（5%~60%），无需停机重配置。
  • 传统堆：角色固定，碎片化后只能 Full GC 整理；G1：Region 回收后整体释放为 Free，天然无碎片。
  • 传统堆：Young GC 必须扫描整个 Young 区；G1：Young GC 只需处理被选入 CSet 的 Eden/Survivor Region。
Humongous 阈值
对象大小 > Region 大小 × 50% 时直接进入连续的 Humongous Region（视为 Old 代）。例如 Region = 4 MB，则 >2 MB 的对象为大对象。大对象在 Full GC 或并发标记后才会被回收，频繁分配大对象是 G1 停顿的常见隐患。
G1·02 请解释 SATB（Snapshot-At-The-Beginning）算法的核心思想，以及"写前屏障"在其中的作用。SATB 会引入什么副作用？ +
SATB 核心思想
并发标记开始时，对堆的引用关系做一次逻辑"快照"。标记过程中保证：快照时存活的对象，本轮不会被错误回收。由此绕开"三色标记"的并发修改正确性问题。
  • 写前屏障（Pre-Write Barrier）：应用线程在改写引用字段 之前，将旧引用值压入线程本地的 SATB 队列。Remark 阶段 GC 会处理队列，把这些旧引用标记为存活，防止漏标。
  • 写前屏障只记录"被覆盖掉"的旧值，与 G1 的写后屏障（用于维护 RSet）共同存在，目的不同。
副作用：浮动垃圾
并发标记期间若某对象被应用线程置为不可达，但它在快照中已被"保活"，则本轮不会回收——成为浮动垃圾（Floating Garbage），下次 GC 才能清除。这是 SATB 的核心代价，也是 G1 实际内存回收效率略低于理论值的原因之一。
G1·03 G1 的 Remembered Set（RSet）存储的是什么方向的引用？维护 RSet 的机制是什么？RSet 会带来哪些开销？ +
引用方向
RSet 记录的是"外部 Region → 本 Region"的引用，即：Region B 的 RSet 里存的是"哪些外部 Region 的哪些 Card 持有指向 B 内对象的引用"。回收 Region B 时，只需扫描 RSet 中列出的 Card，而无需扫描全堆。
  • 写后屏障（Post-Write Barrier）：应用线程写入引用字段之后，将修改过的内存块（Card，512 字节）标记为 Dirty，放入线程本地的 Dirty Card Queue。
  • 后台 Refinement 线程异步从队列取出 Dirty Card，扫描其中的跨 Region 引用，更新目标 Region 的 RSet。
开销
  • 内存：RSet 本身占用堆的 1%~20%，跨 Region 引用密集时显著增大。
  • CPU：每次写操作触发写后屏障；Refinement 线程持续消耗 CPU。
  • Region 数量越多、对象关联越复杂，RSet 开销越高——这是 G1 在超大堆（>64GB）上性能下降的重要原因。
G1·04 G1 为什么叫"Garbage First"？CSet 是如何选取的？默认停顿时间目标是多少？ +
名字由来
并发标记完成后，G1 知道每个 Region 的存活率，可计算出"回收此 Region 能释放多少内存（垃圾量）"。G1 将 Region 按垃圾占比从高到低排序，优先回收垃圾最多的 Region——在相同 STW 时间内最大化释放内存，故名 Garbage First。
  • CSet 构建过程：先把所有 Young Region强制加入（Young GC 必选），再按垃圾密度从高到低追加 Old Region，直到预测总停顿时间触达 -XX:MaxGCPauseMillis（默认 200 ms）为止。
  • CSet 中所有存活对象将被 Evacuation 阶段复制走，Region 整体释放为 Free。
关键参数：-XX:MaxGCPauseMillis=200（停顿预算）、-XX:G1HeapRegionSize（手动指定 Region 大小）。
G1·05 Evacuation 阶段是 STW 还是并发执行？如果复制目标空间不足（to-space 耗尽）会发生什么？ +
全程 STW
Evacuation 是 G1 真正回收内存的步骤，全程 Stop-The-World，使用多线程并行复制（Parallel Copy）以缩短停顿时长。CSet 内所有存活对象被复制到空 Region，原 Region 整块释放——这保证了回收后无碎片。
  • Evacuation Failure：当 Free Region 不足以承接所有存活对象时触发。G1 无法完成复制，就地保留原对象并打标，本次 GC 降级为 Full GC（Serial Old），对整个堆做压缩整理，代价极高（秒级停顿）。
  • 常见根因：Young/Old 晋升速率过快、大对象频繁分配、堆占用率长期偏高。
  • 预防手段：增大堆（-Xmx）、调小 -XX:InitiatingHeapOccupancyPercent（提前触发并发标记），或增加 -XX:G1ReservePercent（预留更多空 Region）。
G1·06 G1 有哪三种 GC 模式？各自的触发条件和回收范围是什么？Mixed GC 是如何逐步清理老年代的？ +
  • Young GC：Eden Region 被分配满时触发，只回收 Young Region（Eden + Survivor），纯 STW。频率最高，也是最主要的停顿来源。
  • 并发标记（Concurrent Marking）：老年代占堆比例达到 -XX:InitiatingHeapOccupancyPercent（默认 45%）时，在 Young GC 的 Initial Mark 阶段"搭车"启动。大部分工作与应用并发，不独立产生停顿。标记完成后 G1 掌握每个 Old Region 的存活率，为 Mixed GC 做准备。
  • Mixed GC：并发标记完成后触发，回收所有 Young Region + 部分高垃圾 Old Region。G1 连续执行若干轮 Mixed GC（默认最多 8 轮，由 -XX:G1MixedGCCountTarget 控制），逐步消化老年代，避免一次性 STW 太久。Mixed GC 结束条件：Old Region 存活率普遍超过 -XX:G1HeapWastePercent，回收性价比不足。
只有当以上三种模式都无法跟上分配速率时，才会退化为 Full GC（单线程 Serial Old，应当极力避免）。
G1·07 TAMS 指针是什么？G1 如何保证并发标记期间新分配的对象不被误回收？ +
TAMS（Top-At-Mark-Start）
每个 Region 都维护两个 TAMS 指针：prevTAMS 和 nextTAMS，用于区分"标记开始前的对象"与"并发标记期间新分配的对象"。
  • Initial Mark（STW）：将每个 Region 的 nextTAMS 快照为当前 top 指针位置。
  • 并发标记阶段：应用线程继续在 top 指针之上（nextTAMS 到新 top 之间）分配新对象。这些"TAMS 之上"的对象被 G1 视为隐式存活，无需标记，不会被回收。
  • Remark/Cleanup（STW）：标记结束，nextTAMS 晋升为 prevTAMS，准备下一轮；已确认存活的对象进入存活位图。
要点：TAMS 机制与 SATB 协同——SATB 屏障保证"快照时已存活对象"不漏标；TAMS 保证"快照后新分配对象"不被误回收。两者缺一不可。
G1·08 Young GC 的并行疏散是如何运作的？GCLAB 的作用是什么？什么情况下触发 Evacuation Failure？ +
并行疏散流程
Young GC 为纯 STW，由多个并行 GC 线程同时扫描 CSet（所有 Eden + Survivor Region）：
  • 每个 GC 线程从 Free Region 列表中领取一块私有缓冲区——GCLAB（GC Local Allocation Buffer）——用于存放复制过来的存活对象，避免多线程竞争同一 Region 的 top 指针。
  • 存活对象按年龄计数：年龄 < 晋升阈值 → 复制到 Survivor；年龄 ≥ 阈值 → 晋升到 Old Region 的 GCLAB。
  • 复制完成后，原 CSet Region 整体释放为 Free，不需要整理碎片。
Evacuation Failure
当 Survivor 和 Old 的 Free Region 耗尽（to-space exhausted），GC 线程无法再分配 GCLAB，疏散中断。G1 的处理：
  • 已复制的对象保留在新位置；未来得及复制的对象原地保留，其指针修复为自引用（self-forwarded）。
  • 触发后续一轮并发标记，最终可能升级为 Full GC。
  • 预防手段：增大 -XX:G1ReservePercent（默认 10%）预留更多 Free Region 作为应急缓冲。

ZGC 面试题

聚焦染色指针、多重映射、读屏障和并发周期等 ZGC 专属机制。

← ZGC 原理
ZGC·01 ZGC 染色指针（Colored Pointer）的核心思想是什么？各元数据位的含义是什么？为什么只能在 64 位 JVM 上使用？ +
核心思想
将 GC 元数据直接编码进对象引用指针的高位比特，使 GC 通过读指针即可获知对象当前的 GC 状态，无需维护独立的 Mark Bitmap 或额外数据结构。
  • Marked0 / Marked1（bit 43/42）：两位交替使用，区分"本轮已标记"与"上轮已标记"，避免每次 GC 都清空全量 bitmap。
  • Remapped（bit 44）：对象已完成地址重映射（迁移后的新指针），该位置位表示指针有效、无需修复。
  • Finalizable（bit 45）：对象只剩 Finalizer 持有，可触发 finalize()。
  • 低 42 位（最大 4TB）存储实际对象地址。
为何只支持 64 位
32 位指针所有 32 位均用于寻址（上限 4 GB），没有多余高位可存储元数据。染色指针依赖 64 位宽度中"用不完"的高位，因此 ZGC 仅在 64 位 JVM 上可用。
ZGC·02 ZGC 为什么需要多重虚拟内存映射？它是如何实现的？对物理内存消耗有影响吗？ +
问题根源
染色指针的高位不同（Marked0 / Marked1 / Remapped），使同一个对象在不同 GC 阶段具有三个不同的虚拟地址。若 OS 把虚拟地址当作独立位置处理，则访问这三个地址将映射到不同的物理页——对象数据就"消失"了。
  • 解法：ZGC 通过 mmap 将同一块物理内存同时映射到三个独立的虚拟地址空间（Marked0 空间 / Marked1 空间 / Remapped 空间），访问任意虚拟地址都读写同一份物理数据。
  • 对物理内存的影响：几乎没有。多重映射只是页表条目的增加，物理内存只有一份。虚拟地址空间消耗是物理堆的约 3 倍（Linux /proc/pid/maps 会显示更大的虚拟内存），但不占用额外物理 RAM，属 OS 层面的 mmap 技巧。
ZGC·03 ZGC 读屏障（Load Barrier）的工作原理是什么？它如何保证并发重定位的正确性？与 G1 写后屏障相比开销有什么不同？ +
工作原理
每当应用线程从堆中加载一个对象引用（obj = *ptr）时，JIT 编译器在此处注入读屏障代码：
  • 检查指针的 Remapped 位是否已置位。
  • 若已置位（已重映射）→ 直接返回，零额外开销。
  • 若未置位（对象已被 GC 迁移，但此指针尚未更新）→ 查询转发表获取对象新地址，就地修复 *ptr = 新地址，再返回。
这保证了：即使 GC 并发移动了对象，应用线程读到的引用始终有效，实现"对应用透明"的并发重定位。
  • 与 G1 写后屏障的对比：G1 写后屏障在写操作时触发，每次引用字段赋值都有开销；ZGC 读屏障在每次读取引用时触发，由于读操作频率远高于写操作，理论上开销更大（约 4%~8% 吞吐损耗）。
  • JDK 21 Generational ZGC 通过分代减少全堆扫描频率，实际读屏障触发次数大幅降低，吞吐损耗已接近 G1。
ZGC·04 ZGC 一次完整 GC 周期有几次 STW？分别在什么时机、做什么事？为什么 ZGC 的停顿时间基本不随堆大小增长？ +
3 次 STW，均 <1ms
  • Pause Mark Start：标记 GC Roots（线程栈、全局变量），激活读屏障。工作量 = GC Root 数量，与堆大小无关。
  • Pause Mark End：处理少量漏标引用，完成最终标记，统计各 Region 存活率。
  • Pause Relocate Start：移动 Roots 直接引用的对象，建立转发表初始条目。
三次 STW 之间的所有耗时工作（Concurrent Mark、处理软弱引用、选 Relocation Set、并发复制对象、修复指针）均与应用线程并发执行。

为何与堆大小无关
三次 STW 均只做"遍历 GC Roots"这一步。GC Root 数量 ≈ 线程数 × 栈帧深度，这是一个有界常数，与堆中对象数量完全无关。堆从 4 GB 扩到 4 TB，Root 数量不变，STW 时间几乎不变——这是 ZGC 实现亚毫秒停顿的根本原因。
ZGC·05 ZGC 的并发重定位（Concurrent Relocate）阶段，旧指针是何时被批量修复的？如果 GC 周期结束时仍有旧指针未修复会怎样？ +
延迟修复策略（Lazy Remap）
ZGC 不在 Relocate 阶段立即修复全部旧指针。并发重定位期间：
  • GC 线程将存活对象复制到新地址，并在转发表（Forwarding Table）中记录"旧地址 → 新地址"映射。
  • 应用线程若读到旧指针，读屏障自动查转发表完成修复（按需、就地修复）。
当一轮 GC 结束时，转发表中仍可能存在未被任何读屏障触及的旧指针（那些字段还没被读过）。ZGC 的做法是：将 Remap 工作推迟到下一轮 GC 的 Concurrent Mark 阶段合并执行——标记时顺便把遇到的旧指针修复，并回收上一轮的转发表。这样每个指针最多被修复一次，开销均摊到两轮 GC 之间。
ZGC·06 Generational ZGC（JDK 21）相比传统 ZGC 解决了什么问题？核心设计变化是什么？ +
传统 ZGC 的痛点
传统 ZGC 每轮都标记整个堆，绝大多数短命对象（Eden 分配后立即成为垃圾）也要经历完整的标记-重定位流程，读屏障（Load Barrier）触发频繁，导致吞吐量比 G1 低 5~15%。

核心设计变化
JDK 21 引入分代后，ZGC 采用 Young / Old 双代模型：
  • Young Gen：存放新分配对象，GC 频率高，每轮只标记和重定位 Young Region——短命对象快速回收，大幅减少读屏障触发次数。
  • Old Gen：存放晋升的长寿对象，GC 频率低，与 Young GC 并发穿插执行。
  • 读屏障（Load Barrier）保留，针对分代场景扩展了跨代引用跟踪（类似 G1 的 RSet，但通过屏障动态维护）。
效果
  • 吞吐量提升 10~40%（典型场景），基本追平 G1。
  • 停顿仍保持亚毫秒（通常 <1 ms）。
  • 启用方式：JDK 21 使用 -XX:+ZGenerational；JDK 23+ 默认开启，无需手动指定。

← CMS 原理
CMS·01 CMS 的六个 GC 阶段是什么？哪些是 STW？为什么 Remark 阶段比 G1 的 Remark 停顿更长？ +
六阶段
  • ① Initial Mark（STW·短）：标记 GC Roots 直接可达对象，工作量与 Root 数量相关，停顿极短。
  • ② Concurrent Mark（并发）：DFS 遍历整个老年代对象图，耗时最长，占整个 CMS 周期 80% 以上。
  • ③ Concurrent Preclean（并发）：重扫并发期间标记为 Dirty 的 Card，减轻 Remark 压力。
  • ④ Remark（STW·较长⚠）：处理并发阶段所有引用变更，使用增量更新，需重扫所有"重新变灰"的对象子树。
  • ⑤ Concurrent Sweep（并发）：清除未标记对象，归还空间到 free-list。不移动对象，产生碎片。
  • ⑥ Concurrent Reset（并发）：重置内部数据结构。
Remark 更长的原因
G1 使用 SATB（写前屏障记录旧引用），Remark 只需处理 SATB 队列中少量条目，停顿极短（1~10ms）。
CMS 使用增量更新（写后屏障将黑色对象重置为灰色），Remark 需要重新扫描所有被重置为灰色的黑色对象的完整子树——工作量随并发阶段修改量增长，可达数十至数百毫秒。
CMS·02 什么是 Concurrent Mode Failure？触发条件是什么？发生时 JVM 会做什么？如何预防？ +
定义
CMS 并发清除阶段，应用线程仍在分配对象。若 Old Gen 在并发 GC 完成之前被填满，CMS 来不及完成回收，此时触发 Concurrent Mode Failure。

后果：CMS 中止当前并发周期，JVM 退化为 Serial Old 单线程 Full GC——Stop-The-World + 整堆标记压缩，停顿时间可达秒级，是 CMS 最严重的故障场景。

预防措施：
  • 调低 -XX:CMSInitiatingOccupancyFraction 到 70~75%（默认 92% 几乎没有余量）
  • 增大堆（-Xmx），减少 Old Gen 压力
  • 优化应用减少长生命周期对象晋升
  • 根本解决：迁移到 G1（G1 的 Mixed GC 逐步回收老年代，无 CMF 风险）
CMS·03 CMS 为什么会产生内存碎片？碎片化如何影响性能？有什么解决手段？ +
根因：CMS 使用标记-清除（Mark-Sweep）算法，Concurrent Sweep 阶段只是将死亡对象标记为空闲归还 free-list，从不移动存活对象、从不整理内存。

影响演化：
  • 随应用运行，Old Gen 出现越来越多大小不一的"空洞"
  • free-list 查找合适空间开销增大
  • 大对象分配时找不到足够大的连续空间 → Promotion Failure → Full GC
  • 碎片严重时即使 Old Gen 总空闲空间充足，仍无法满足分配请求
解决手段：
  • -XX:+UseCMSCompactAtFullCollection：Full GC 时顺带做一次内存整理（已在 JDK 9 废弃）
  • -XX:CMSFullGCsBeforeCompaction=N：每 N 次 Full GC 后做一次整理
  • 根本解决：换用 G1——G1 的 Evacuation 将存活对象复制到空 Region，回收后的 Region 整块释放，天然无碎片
CMS·04 CMS 使用增量更新（Incremental Update）而非 SATB，各有什么代价？G1 为什么选择 SATB？ +
增量更新（CMS）：
写后屏障——黑色对象新增对白色对象的引用后，将黑色对象重置为灰色。
代价：Remark 阶段需要重扫所有被重置为灰色的黑色对象的完整子树，工作量大，STW 较长；实现复杂（需管理"重新变灰"的对象集合）。

SATB（G1）：
写前屏障——引用被覆盖前，将旧引用值压入 SATB 队列。
代价：浮动垃圾较多（快照中存活的对象本轮不回收）；但 Remark 只需清空少量 SATB 队列，停顿极短。

G1 选择 SATB 的核心原因
G1 的核心目标是可预测的短暂停顿（默认 200ms 预算）。SATB 将 Remark 停顿控制在 1~10ms；若用增量更新，Remark 停顿不可预测，与 G1 设计目标冲突。浮动垃圾问题由 Mixed GC 多轮回收逐步消化。
CMS·05 CMS 在 JDK 14 被移除前，为什么还有人选它而不是 G1？CMS 和 G1 各自的适用场景是什么？ +
历史原因：
G1 在 JDK 6u14 引入（2009），JDK 9 才成为默认（2017）。在此之前，G1 仍处于实验/调优阶段，生产稳定性不如 CMS。JDK 9 之前的老项目若已针对 CMS 精心调优，迁移有风险和成本。

技术层面选 CMS 的场景（历史上）：
  • 堆较小（≤4GB），碎片化问题不严重，CMS 停顿比 G1 更可预测
  • 应用对 吞吐量 要求高但能接受偶发性长停顿（CMF 触发的 Full GC）
  • JDK 7/8 环境，G1 尚不成熟
G1 的优势（也是 CMS 被取代的原因）：
  • 无内存碎片（Evacuation 整块回收 Region）
  • Remark 更短（SATB vs 增量更新）
  • 可预测的停顿时间（MaxGCPauseMillis 目标驱动 CSet 选择）
  • Mixed GC 渐进回收老年代，无 CMF 风险
现状 JDK 14 已移除 CMS。现代应用：中大型堆用 G1（默认），超低延迟或超大堆用 ZGC（JDK 21+ Generational ZGC）。
● Interview · G1 vs ZGC

对比面试题

从停顿、吞吐、屏障、适用场景等维度综合比较两款收集器。

← G1 vs ZGC 对比
对比·01 G1 和 ZGC 的停顿时间模型有何根本差异？为什么 G1 的停顿会随堆变大而增长，而 ZGC 不会？ +
  • G1：Evacuation 阶段是全 STW，需要扫描并复制 CSet 内的所有存活对象。堆越大 → Old Region 越多 → CSet 候选对象越多 → Evacuation 耗时越长。虽然 G1 用停顿预算（200ms）限制 CSet 大小，但并发标记的 Remark 阶段也与堆中对象数量正相关。
  • ZGC：STW 只做 GC Roots 扫描，工作量由线程数 × 栈深度决定（有界常数），与堆大小完全解耦。对象移动全部在并发阶段完成，靠读屏障按需修复指针。
本质差异：G1 的 STW 工作量 = f(对象数)，ZGC 的 STW 工作量 = f(Root数)。
对比·02 ZGC 的吞吐量为什么通常低于 G1？ZGC 有哪些机制可以弥补这一差距？ +
根因：读屏障频率高于写屏障
ZGC 使用读屏障——几乎每次堆引用加载都会执行一次屏障检查。而 G1 使用写后屏障，只在引用字段被修改时触发，触发频率远低于读操作。读操作在 JVM 中远比写操作频繁，导致 ZGC 的屏障总体开销更高，约降低 5~15% 吞吐量。
  • Generational ZGC（JDK 21，-XX:+ZGenerational）：引入年轻代，大多数对象在年轻代快速回收，减少全堆并发标记/重定位的频率，读屏障触发次数大幅减少，吞吐量接近 G1。
  • JIT 深度内联屏障代码：现代 JVM 已将读屏障优化为极少指令，热点路径（Remapped 位已置位）几乎零开销。
对比·03 G1 和 ZGC 在"跨区引用追踪"上的机制分别是什么？ZGC 为何可以不用 RSet？ +
  • G1 — RSet（Remembered Set）：因为 G1 每次只回收部分 Region（CSet），必须知道"外部是否有引用指向 CSet 内的对象"。RSet 是每个 Region 维护的反向索引，记录哪些外部 Card 持有指向本 Region 的引用。没有 RSet，G1 就必须扫描全堆才能找到所有根——这会让 STW 时间无法控制。
  • ZGC — 无 RSet，依赖全堆并发标记：ZGC 每轮 GC 都执行全堆并发标记，从所有 GC Roots 出发遍历完整对象图。因为标记是并发的，不占用 STW 预算，所以不需要 RSet 作"精准索引"来减少扫描范围。代价是每轮 GC 都要标记全堆，内存开销低但 CPU 占用更持续。
简言之：G1 用 RSet 减少 STW 时的扫描量；ZGC 用并发标记代替 RSet，把扫描开销转移到并发阶段。
对比·04 G1 和 ZGC 各自的"灾难场景"是什么？如何诊断和规避？ +
G1 — Evacuation Failure → Full GC
  • 现象：GC 日志出现 Evacuation Failure，随后触发 Full GC (Ergonomics)，STW 可达秒级。
  • 根因：Free Region 不足以承载 CSet 内存活对象的复制；常由晋升速率过快、大对象频繁分配、堆占用率长期 >85% 引起。
  • 规避：增大堆 / 降低 -XX:InitiatingHeapOccupancyPercent（提前启动并发标记）/ 增加 -XX:G1ReservePercent。
ZGC — Allocation Stall（分配停顿）
  • 现象：GC 日志出现 Allocation Stall，应用线程被阻塞等待 GC 腾出空间，无传统意义的 Full GC。
  • 根因：GC 并发回收速度跟不上应用分配速率（GC 饥饿），内存耗尽。
  • 规避：增大堆 / 使用 Generational ZGC（JDK 21）提升回收效率 / 优化应用对象分配热点。
对比·05 在实际项目中如何选择 G1 还是 ZGC？请给出决策依据和典型场景。 +
  • 选 ZGC 的条件（满足任一）
    • 对 P99/P999 延迟有严格要求（<10ms 响应时间），如交易系统、实时竞价、游戏服务器。
    • 堆 >32 GB，G1 的 STW 停顿已不可接受。
    • JDK ≥ 15（ZGC 正式可用）且对吞吐量损耗 5~10% 可接受。
    • TB 级超大堆（只有 ZGC 能做到亚毫秒）。
  • 选 G1 的条件（满足任一）
    • JDK 版本 < 15，或需要稳定分代支持（JDK 21 前 ZGC 无分代）。
    • 堆 4~32 GB，对延迟要求不极端（可接受 100~200ms 停顿）。
    • 追求最大吞吐量，对读屏障开销敏感的计算密集型场景。
    • 团队对 G1 调优更熟悉，生产经验更丰富。
JDK 21+ Generational ZGC 已大幅缩小吞吐量差距，新项目在满足 JDK 版本的前提下，低延迟场景首选 ZGC。
对比·06 生产环境如何诊断 G1 / ZGC 的 GC 性能问题？两者的调优思路有何不同？ +
G1 诊断与调优
开启详细日志：-Xlog:gc*:file=gc.log:time,uptime,level,tags，重点关注：
  • (to-space exhausted) → Evacuation Failure，需增大 -XX:G1ReservePercent 或扩堆。
  • Full GC (Ergonomics) → G1 自适应降级，说明并发标记已跟不上分配速率，应调低 -XX:InitiatingHeapOccupancyPercent 提前触发标记。
  • 停顿超出 MaxGCPauseMillis → 缩小 Young Gen 比例（-XX:G1NewSizePercent）或增加 -XX:ConcGCThreads。
  • Old Region 碎片化 → 适当降低 -XX:G1HeapWastePercent 触发更多 Mixed GC。
ZGC 诊断与调优
同样开启 -Xlog:gc*，ZGC 特有的关键信号：
  • Allocation Stall（分配停顿）：应用线程因等待 GC 释放内存而被阻塞，是 ZGC 中最严重的问题信号。
  • 根因：GC 并发速度跟不上分配速率。首选方案：增大堆内存（-Xmx）；其次增加 -XX:ConcGCThreads 并发线程。
  • ZGC 几乎不需要手动调参：给够堆内存 + 开启 Generational ZGC（JDK 21+）通常已足够。
核心区别：G1 调优复杂，需要平衡 Young/Old 比例、停顿预算、Mixed GC 频率等多个旋钮；ZGC 的核心调优手段只有两个——堆大小和并发线程数，其余交给 GC 自适应。
⚡ 实战 线上频繁 Full GC 怎么排查？ +

频繁 Full GC 是 JVM 性能问题中最常见也最紧急的场景，需要按照"观察 → 定位 → 修复"三步走。

Step 1：确认 Full GC 频率与耗时
首先开启统一 GC 日志（JDK 9+ 统一日志框架）： 
-Xlog:gc*,gc+heap=debug:file=gc-%t.log:time,uptime,level,tags:filecount=10,filesize=64M
在日志中搜索 Full GC 关键字，关注每次 Full GC 的暂停时间（Pause Full）以及触发原因（如 Ergonomics、Allocation Failure、Metadata GC Threshold）。如果日志中出现 to-space exhausted，说明 G1 Evacuation 阶段预留空间不足，已退化为 Full GC。 Step 2：观察各代使用率变化
使用 jstat 以 1 秒间隔采样： 
jstat -gcutil <pid> 1000
重点关注列：O（老年代使用率）是否持续走高且不回落——这通常意味着存在内存泄漏；M（Metaspace 使用率）是否逼近上限——说明类加载器泄漏或 Metaspace 配置偏小；FGC（Full GC 累计次数）增长速度是否异常。如果 O 列长期 >90% 且 FGC 快速增长，基本可以判定为老年代空间不足或泄漏。 Step 3：导出堆快照并分析
在 Full GC 发生后尽快导出 heap dump： 
jmap -dump:live,format=b,file=heap.hprof <pid>
注意 live 参数会先触发一次 Full GC 再 dump，确保拿到的是存活对象。使用 MAT（Memory Analyzer Tool）或 VisualVM 打开 dump 文件，依次查看：
  • Leak Suspects Report：MAT 自动分析可疑泄漏点，快速缩小范围。
  • Dominator Tree：按 retained size 排序，找出占用内存最大的对象及其引用链。
  • Histogram：按类统计实例数和总大小，关注异常的集合类（如 HashMap、ArrayList）。
常见原因与处理
  • Metaspace 不足：动态类加载（CGLIB/反射/OSGi）频繁创建类导致 Metaspace 耗尽。处理：增大 -XX:MaxMetaspaceSize 和 -XX:MetaspaceSize，排查是否有不必要的动态代理。
  • 大对象直接晋升老年代：单次分配超过 -XX:PretenureSizeThreshold（G1 下默认为 Region 大小一半）的对象直接进入 Old Gen。处理：调大 -XX:PretenureSizeThreshold 或拆分为小对象分批处理。
  • 内存泄漏：ThreadLocal 未调用 remove()、静态集合无限增长、监听器/回调未注销。处理：修复代码，确保资源正确释放。
  • 老年代碎片化：G1 下 Mixed GC 未及时回收导致碎片。处理：降低 -XX:G1HeapWastePercent（默认 5%）促使更多 Mixed GC，或在极端情况下适当增大堆。
应急措施：线上紧急时可先通过 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/ 确保 OOM 时自动留 dump，然后重启服务恢复。后续根据 dump 分析结果做根因修复。
⚡ 实战 如何根据业务选型 G1 还是 ZGC？ +

GC 选型的核心是在延迟、吞吐量和堆大小三个维度之间权衡。以下是系统化的决策框架。

G1 — 通用均衡之选
G1（Garbage-First）是 JDK 9+ 的默认 GC，适用于堆大小 4~32 GB 的绝大多数场景。它通过 Region 分区和停顿目标（-XX:MaxGCPauseMillis，默认 200ms）在延迟与吞吐之间自适应调节。Young GC 和 Mixed GC 均为 STW，停顿时间与存活对象数量正相关。如果你的应用对 P99 延迟要求在百毫秒级别、堆不超过 32 GB、且团队已有成熟的调优经验，G1 是最稳妥的选择。典型场景：电商后台服务、微服务集群、批处理任务。 ZGC — 极致低延迟之选
ZGC 在 JDK 15 正式 production-ready，JDK 21 引入 Generational ZGC 后性能大幅提升。其核心优势是几乎所有阶段都并发执行，STW 停顿稳定在亚毫秒级（<1ms），且停顿时间不随堆大小增长。ZGC 适合堆 16 GB 到数 TB 的超大堆场景。典型场景：金融交易系统（P99 <10ms）、实时竞价引擎、游戏服务器、大规模缓存服务。 决策依据速查
  • SLA 要求 P99 延迟 <10ms → 选 ZGC。G1 在大堆下 Mixed GC 停顿可能超过 100ms，无法满足严格的延迟 SLA。
  • 堆 ≤16 GB 且吞吐优先 → 选 G1。G1 没有读屏障开销，在 CPU 密集型场景下吞吐量更高。
  • 堆 >32 GB → 必须选 ZGC。G1 在大堆下 STW 停顿不可控，Full GC 可能长达数秒。
  • JDK 版本 <15 → 只能用 G1（或 CMS，但 CMS 已在 JDK 14 移除）。ZGC 在 JDK 15 前为实验性功能，不建议上生产。
注意事项
  • CPU 开销：ZGC 的读屏障（Load Barrier）和并发标记/转移会额外消耗约 10~15% 的 CPU。如果服务器 CPU 已接近饱和且无法扩容，ZGC 反而可能导致吞吐下降和延迟抖动。
  • 内存开销：ZGC 需要额外的堆外内存存储染色指针映射和转发表，实际内存占用比 G1 高约 5~10%。
  • Generational ZGC（JDK 21+）：开启 -XX:+UseZGC -XX:+ZGenerational 后，ZGC 引入分代，大幅降低了 CPU 开销和 Allocation Stall 概率，在绝大多数场景下已接近甚至超越 G1 的吞吐水平。新项目如果 JDK 版本允许，强烈推荐直接启用。
  • 不适合 ZGC 的场景：CPU 密集型且资源紧张的环境（如嵌入式设备、低配容器），读屏障的持续开销会挤压业务线程的 CPU 时间。
实操建议：上线前用 jmh 或真实流量做 A/B 压测，分别跑 G1（-XX:+UseG1GC）和 ZGC（-XX:+UseZGC），对比 P99/P999 延迟、QPS 和 CPU 利用率。数据说话比理论分析更可靠。
