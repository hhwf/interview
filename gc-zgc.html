Java Virtual Machine · Garbage Collection

ZGC
亚毫秒停顿解析

深入剖析 ZGC 的核心创新——染色指针将 GC 元数据编码进指针高位、多重虚拟内存映射使并发重定位成为可能、读屏障在应用线程读取引用时透明修复指针，实现停顿时间与堆大小无关的亚毫秒级 GC。

● Z Garbage Collector

ZGC —— 亚毫秒停顿

ZGC 以 亚毫秒级停顿 为核心目标，通过染色指针和读屏障 将几乎所有 GC 工作移到并发阶段，STW 仅做极少量的根标记。 适合对延迟极敏感的大堆（GB~TB 级）场景。

染色指针（Colored Pointer）

ZGC 的关键创新：将 GC 元数据直接编码进对象指针的 高位比特。 64 位系统实际寻址只用低 44 位（最大 16TB），剩余高位被 ZGC 借用存储 标记状态、重映射状态、Finalizable 状态。 这样 GC 无需额外的 Mark Bitmap，读指针即知对象 GC 状态。

为什么 32 位系统无法用染色指针？ 32 位指针所有 32 位都用于寻址（最大 4GB），没有任何多余高位可存储元数据。 ZGC 因此只支持 64 位 JVM。

多重虚拟内存映射

染色指针引发一个问题：同一个对象在不同 GC 阶段，其指针高位不同（Marked0 / Marked1 / Remapped）， 导致同一物理地址对应多个不同的虚拟地址。ZGC 的解法是 将同一块物理内存映射到三个独立的虚拟地址空间， 访问任意一个虚拟地址都能读到同一份物理数据，OS 层面透明。

GC 并发标记时使用 Marked0/Marked1 空间；重定位后使用 Remapped 空间。 物理内存只有一份，虚拟映射是操作系统层面的 mmap 技巧，几乎零开销。
面试要点：多重映射（Multi-Mapping）是 ZGC 实现染色指针的底层机制：将同一物理内存页映射到 3 个不同的虚拟地址空间（Marked0 / Marked1 / Remapped），应用程序访问不同颜色的指针都能访问到同一物理页，只是触发不同的读屏障逻辑。这使得 ZGC 可以在不 STW 的情况下修改指针颜色。

读屏障（Load Barrier）

每当应用线程从堆中 加载一个对象引用 时，JIT 编译器注入的 读屏障代码 会检查指针的染色位。若对象已被 GC 移动但指针尚未更新， 屏障自动查询转发表，修复指针后再返回——应用线程完全感知不到对象已被移动。 这是 ZGC 并发重定位的核心保障。

面试要点：ZGC 读屏障（Load Barrier）在应用线程读取对象引用时触发（而非写时），检查指针是否需要更新（Remap）。这使 ZGC 能并发移动对象而不需要 STW：移动完成后，下次读该引用时读屏障将旧指针更新为新地址。代价是每次对象读都有额外检查开销（但 JIT 优化后约 10–20ns）。
⚡
性能影响：读屏障在每次引用加载时都执行，约带来 4%~8% 吞吐量损耗。 JDK 21 的 Generational ZGC 优化了屏障逻辑，通过分代减少全堆扫描频率，实际影响更小。

ZGC GC 并发周期

ZGC 一次完整 GC 包含 3 次极短 STW 和大量并发工作。 STW 只处理必须在安全点完成的少量任务（扫描根）， 所有耗时工作——标记、移动对象、修复指针——全部并发执行。

停顿短的根本原因：ZGC 三次 STW 均只做「遍历根引用」这一最小量工作—— 根引用数量有界（线程数 × 栈深），与堆大小无关。 堆从 4GB 增到 4TB，STW 时间几乎不变。
🚀
Generational ZGC（JDK 21，-XX:+ZGenerational）——吞吐量的飞跃：
传统 ZGC 每轮标记全堆，读屏障触发频繁，吞吐量比 G1 低 5~15%。JDK 21 引入分代后：
  • Young Gen 快速回收：大多数短命对象在年轻代收集，无需全堆并发标记，读屏障触发次数大幅减少。
  • 吞吐量接近 G1：Oracle 实测提升 10%~40%，大幅缩小与 G1 的差距。
  • 停顿仍保持亚毫秒：分代设计不影响 STW 上限，依然只扫根引用。
  • JDK 23+ 隐式启用：-XX:+UseZGC 默认即为分代模式；旧非分代行为需 -XX:-ZGenerational 显式关闭。
