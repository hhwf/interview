JMM & volatile & synchronized

Q1 volatile 的作用是什么？底层原理是什么？ +

两大作用：可见性 + 有序性（不保证原子性）

可见性：volatile 写强制将工作内存值刷回主内存，volatile 读强制从主内存重新加载，使其他线程的修改立即可见。

有序性：编译器在 volatile 写前后、读后插入内存屏障（StoreStore / StoreLoad / LoadLoad / LoadStore），禁止特定方向的指令重排序。

底层原理：JIT 将 volatile 写编译为带 lock 前缀的汇编指令（如 lock addl），触发 MESI 缓存一致性协议，将其他 CPU 缓存行置为 Invalid 状态，强制重读主内存。

不保证原子性：volatile int i; i++ 是 read-modify-write 三步，多线程下会丢失更新，需用 AtomicInteger。

典型使用场景：① 状态标志位（volatile boolean running）② DCL 双重检查锁单例中防止对象半初始化。

Q10 Java 内存模型（JMM）是什么？happens-before 规则有哪些？ +

JMM：Java 内存模型（Java Memory Model）是 JVM 规范的一部分，定义了多线程程序中变量的读写可见性规则，屏蔽不同 CPU 架构的内存模型差异，解决并发的三大问题：可见性、有序性、原子性。

工作内存 vs 主内存：每个线程有自己的工作内存（抽象，对应 CPU 缓存），只能操作主内存变量的副本。线程间通信必须经过主内存。

happens-before 8 条规则（A hb B 意味着 A 的写对 B 可见）：

  1. 程序顺序规则：同一线程内，前面操作 hb 后面操作
  2. 监视器锁规则：unlock hb 后续对同一锁的 lock
  3. volatile 规则：volatile 写 hb 后续对该变量的读
  4. 线程启动规则：Thread.start() hb 该线程的任何操作
  5. 线程终止规则：线程所有操作 hb 其他线程的 Thread.join() 返回
  6. 线程中断规则：interrupt() hb 检测到中断的代码
  7. 对象终结规则：构造函数结束 hb finalize() 开始
  8. 传递性：A hb B，B hb C → A hb C
Q2 synchronized 的锁升级过程是什么？ +

四个状态（单向升级，不可降级）：

① 无锁：对象初始状态，Mark Word 存 hashCode + GC 年龄。

② 偏向锁：第一个线程访问时，CAS 将线程 ID 写入 Mark Word，之后同一线程重入只检查 ID，无需 CAS，开销极低。JDK 15 默认禁用，JDK 18 彻底移除。

③ 轻量级锁：第二个线程竞争时，偏向锁撤销（STW），升级为轻量级锁。线程在栈帧创建 Lock Record，CAS 将 Mark Word 替换为 Lock Record 指针，失败则自旋重试（自适应自旋）。

④ 重量级锁：自旋超过阈值或有第三个线程等待时，膨胀为重量级锁，关联 OS Mutex。未获锁线程进入 Monitor 的 EntryList 阻塞（内核态切换），开销最大。

升级触发条件：偏向→轻量（竞争出现）；轻量→重量（自旋失败 或 等待线程数 > 1）。

Q3 synchronized 和 ReentrantLock 有什么区别？ +

相同点：都是可重入的互斥锁，都保证可见性、有序性、原子性。

主要区别：

① 实现层面：synchronized 是 JVM 关键字，基于 Monitor；ReentrantLock 是 Java API，基于 AQS。

② 额外能力（ReentrantLock 特有）：

  • 可中断等待：lockInterruptibly()
  • 超时等待：tryLock(timeout, unit)
  • 公平锁：new ReentrantLock(true)
  • 多条件变量：newCondition()，可精确唤醒

③ 锁释放：synchronized 自动释放（代码块结束/异常）；ReentrantLock 必须在 finally 中手动调用 unlock()，否则死锁。

④ 性能：JDK 6 后 synchronized 引入锁升级优化，低竞争下性能相当；高竞争下 ReentrantLock 略优（可中断减少无效等待）。

选用原则：优先 synchronized（简单、JVM 优化好）；需要超时/中断/多条件时用 ReentrantLock。

CAS & AQS

Q4 CAS 是什么？有什么缺点？如何解决 ABA 问题？ +

CAS（Compare-And-Swap）：比较并交换，是一种无锁原子操作。包含三个操作数：内存地址 V、预期值 expected、新值 newValue。当且仅当 V == expected 时，原子地将 V 改为 newValue，否则不修改。底层依赖 CPU 原子指令（x86: lock cmpxchg）。

三大缺陷：

① ABA 问题：值从 A→B→A，CAS 认为未变化但实际已被修改两次。
解决：AtomicStampedReference，附加 stamp 版本号，每次修改 stamp+1，CAS 同时比较值和版本号。

② 自旋开销：高并发下 CAS 长时间自旋，大量消耗 CPU。
解决：LongAdder 分段累加降低竞争；或限制自旋次数后升级为阻塞锁。

③ 只能保证单变量原子性：多个变量需要同时原子更新时 CAS 无法直接实现。
解决：将多个变量封装为对象，用 AtomicReference 整体 CAS；或改用 synchronized。

Q5 AQS 的原理是什么？哪些类基于 AQS 实现？ +

AQS（AbstractQueuedSynchronizer）：JUC 同步框架的核心，通过一个 volatile int state 表示同步状态，一个 CLH 变体双向链表队列管理等待线程，使用 LockSupport.park/unpark 阻塞/唤醒线程。

核心流程：线程调用 acquire() → 先 tryAcquire()（子类实现，CAS 修改 state）→ 失败则封装成 Node 入队 → LockSupport.park 阻塞。持锁线程 release() → tryRelease() 恢复 state → unpark 队头后继线程。

两种模式：

  • 独占模式：同时只有一个线程持锁，重写 tryAcquire / tryRelease
  • 共享模式：多线程可同时持锁，重写 tryAcquireShared / tryReleaseShared

基于 AQS 实现的类：

  • ReentrantLock：独占，state = 重入次数
  • ReentrantReadWriteLock：读共享/写独占，state 高16位=读计数，低16位=写重入数
  • Semaphore：共享，state = 剩余许可数
  • CountDownLatch：共享，state = 计数，归零后所有 await 线程被唤醒
  • ThreadPoolExecutor 的 Worker：独占，用于判断线程是否在执行任务

线程池

Q6 ThreadPoolExecutor 有哪些核心参数？各自的作用是什么？ +

7 个核心参数：

  • corePoolSize：核心线程数，常驻不销毁（除非设置 allowCoreThreadTimeOut）
  • maximumPoolSize：最大线程数，队列满后才扩展到此上限
  • keepAliveTime + unit：非核心线程空闲超时时间，超时后销毁
  • workQueue：任务等待队列。有界（ArrayBlockingQueue）防止 OOM；无界（LinkedBlockingQueue）注意堆积风险
  • threadFactory：线程工厂，可自定义线程名方便排查问题
  • handler：拒绝策略，队列满且线程数达上限时触发

执行流程：提交任务 → 线程数 < core 则创建核心线程 → 否则入队 → 队满则创建非核心线程（直到 max）→ 仍满则执行拒绝策略。

四种拒绝策略：AbortPolicy（抛异常）/ CallerRunsPolicy（调用者执行，实现背压）/ DiscardPolicy（静默丢弃）/ DiscardOldestPolicy（丢最老任务）。

Q7 线程池的拒绝策略有哪些？分别适用什么场景？ +

触发时机：workQueue 已满 且 线程数已达 maximumPoolSize。

四种内置策略：

  • AbortPolicy（默认）：抛出 RejectedExecutionException。适合必须感知任务被拒绝的场景，调用方可捕获异常进行补救。
  • CallerRunsPolicy：由提交任务的线程自己执行该任务。不丢任务，但会阻塞提交线程（如 HTTP 线程），天然实现背压，减缓任务提交速度。
  • DiscardPolicy：静默丢弃新任务，无任何通知。适合对任务完整性不敏感的统计/日志类场景。
  • DiscardOldestPolicy：丢弃队头最旧的任务，重新尝试提交新任务。适合新任务优先级高于旧任务的场景（如实时行情推送）。

自定义策略：实现 RejectedExecutionHandler 接口，可将拒绝任务写入数据库/MQ 以便后续重试，适合不允许丢失的核心业务场景。

Q8 如何合理设置线程池大小？CPU 密集型和 IO 密集型有何不同？ +

CPU 密集型任务：任务主要消耗 CPU（计算、加解密、压缩）。线程过多会导致频繁上下文切换，反而降低性能。

公式：N = CPU 核心数 + 1（+1 是为了在偶发页缺失等停顿时仍能保持 CPU 满负荷）

IO 密集型任务：任务大量时间在等待 IO（数据库查询、网络请求、文件读写），线程等待期间 CPU 空闲，可配置更多线程充分利用 CPU。

公式：N = CPU 核心数 × (1 + 等待时间 / 计算时间)。实践中常用 2 × CPU 核心数 作为起点，再根据监控调整。

混合型任务：拆分为独立的 CPU 密集线程池和 IO 密集线程池，互不干扰。

实践建议：公式只是起点，最终要通过压测 + 监控（线程活跃数、队列深度、CPU 利用率、RT）动态调整。推荐使用支持动态调参的线程池（如美团 DynamicTp）。

并发工具类

Q11 CountDownLatch、CyclicBarrier、Semaphore 分别适用什么场景？ +

CountDownLatch（倒计时门闩）：

  • 用法：new CountDownLatch(N)，工作线程完成后调 countDown()，主线程 await() 等待 N 归零
  • 特点：一次性，不可重置；基于 AQS 共享模式
  • 场景：主线程等待多个子任务全部完成（如并行查询后汇总结果）

CyclicBarrier（循环屏障）：

  • 用法：new CyclicBarrier(N, barrierAction)，所有线程调 await() 到达屏障点后一起继续，可触发 barrierAction
  • 特点：可重置/复用（cyclic），基于 ReentrantLock + Condition
  • 场景：多线程分阶段计算（每轮结束后同步，再开始下一轮，如并行数值模拟）

Semaphore（信号量）：

  • 用法：new Semaphore(permits)，acquire() 消耗许可，release() 归还许可
  • 特点：控制同时访问资源的线程数量；基于 AQS 共享模式
  • 场景：限流（如数据库连接池限制最大并发连接数）、资源池控制

核心区别：CountDownLatch 是"等待事件完成"；CyclicBarrier 是"等待所有参与者到齐"；Semaphore 是"限制并发数量"。

Q12 ThreadLocal 的原理是什么？如何避免内存泄漏？ +

原理：每个 Thread 对象内部持有一个 ThreadLocalMap（Entry 数组），key 是 ThreadLocal 实例的弱引用，value 是线程本地值。调用 ThreadLocal.get/set 时，实际操作的是当前线程的 ThreadLocalMap。

内存泄漏原因：

  • ThreadLocalMap 的 Entry key 是弱引用，GC 时 ThreadLocal 对象可被回收，key 变为 null
  • 但 value 是强引用，只要线程存活（如线程池复用），value 就无法被 GC 回收
  • Thread → ThreadLocalMap → Entry → value，形成强引用链，导致内存泄漏

避免内存泄漏：使用完后必须调用 ThreadLocal.remove() 手动清除 Entry，尤其是在线程池场景下。

ThreadLocalMap 的补救机制：get/set 时会扫描 key 为 null 的 Entry 并清理（expungeStaleEntry），但不保证及时，不能替代显式 remove()。

典型应用：数据库连接、用户登录信息（Token/UserId）、Spring 事务的 ConnectionHolder、MDC 日志追踪。

Q13 什么是死锁？产生的四个必要条件是什么？如何预防和检测？ +

死锁：两个或多个线程互相持有对方需要的锁，均在等待对方释放，导致永久阻塞。

四个必要条件（缺一不可）：

  1. 互斥：资源同时只能被一个线程持有
  2. 持有并等待：线程持有已有资源，等待获取其他资源
  3. 不可剥夺：已持有的资源不能被强制剥夺，只能主动释放
  4. 循环等待：线程间形成环形等待链（T1 等 T2，T2 等 T1）

预防（破坏必要条件）：

  • 破坏持有并等待：一次性申请所有资源，或申请新资源前释放已有资源
  • 破坏循环等待：对所有锁统一编号，按序申请（如先锁 A 再锁 B，不允许反向）
  • 破坏不可剥夺：使用 tryLock(timeout) 超时失败后释放已持有的锁

检测：

  • jstack <pid> 打印线程堆栈，搜索 "deadlock" 关键字
  • JConsole / VisualVM 的线程面板可直接检测死锁
  • 代码层：ThreadMXBean.findDeadlockedThreads()
Q14 ReentrantReadWriteLock 的适用场景和注意事项？ +

适用场景：读多写少，如本地缓存、配置中心、路由表等。读操作并发执行，写操作独占，比全程 synchronized 吞吐量高。

读写规则：读-读不互斥；读-写互斥；写-写互斥。

锁降级（支持）：持有写锁时可以再获取读锁，然后释放写锁（写锁降级为读锁）。常用于"更新缓存后立刻读取"，保证不被其他写线程打断。

锁升级（不支持，会死锁）：持有读锁时不能直接获取写锁。因为写锁需要等所有读锁释放，而当前线程持有读锁又在等写锁 → 死锁。

写饥饿问题：读锁持续到来时，写锁可能长期无法获取。非公平模式下更严重；使用公平模式（new ReentrantReadWriteLock(true)）可缓解，但降低吞吐量。

更高性能替代：极高并发读场景可用 StampedLock 的乐观读模式，但不支持重入和 Condition，使用更复杂。

Q15 CompletableFuture 相比 Future 有哪些改进？ +

Future 的缺陷：

  • get() 是阻塞调用，无法在结果就绪时自动回调
  • 不支持链式组合多个异步任务
  • 无法手动完成（无法从外部设置结果）
  • 异常处理不便（只能在 get() 处 try-catch）

CompletableFuture 的改进（JDK 8+）：

  • 异步回调：thenApply / thenAccept / thenRun — 结果就绪后自动触发，非阻塞
  • 链式组合：thenCompose（串行）/ thenCombine（并行合并两个结果）
  • 聚合等待：allOf() 等全部完成 / anyOf() 等任意一个完成
  • 异常处理：exceptionally() / handle() 在链中优雅处理异常
  • 手动完成：complete(value) / completeExceptionally(ex)
  • 自定义线程池：各回调方法可传入 Executor，避免使用默认 ForkJoinPool.commonPool

注意事项：默认使用 ForkJoinPool.commonPool()，IO 密集型任务应传入自定义线程池，否则会阻塞公共池影响其他任务。get() 仍会阻塞，生产中应优先用 join() 或回调，避免嵌套阻塞。

并发容器

Q9 ConcurrentHashMap 的实现原理是什么？为什么不允许 null key？ +

ConcurrentHashMap（JDK 8）：Node 数组 + CAS + synchronized

  • 数据结构：Node 数组 + 链表（长度 ≥ 8 且数组 ≥ 64 转红黑树），查询最坏 O(logN)
  • 锁粒度：桶头节点级别。桶为空用 CAS 插入；桶非空用 synchronized 锁桶头节点，并发度 = 桶数量
  • 读操作：Node 的 value / next 均为 volatile，读无需加锁
  • size()：baseCount + CounterCell[] 分段累加（类似 LongAdder），是估算值
  • 扩容：支持多线程协助 transfer（helpTransfer），桶 hash == MOVED 时触发

不支持 null key / null value（HashMap 支持）：多线程下无法区分"key 不存在"和"key 对应 null"。

Q19 HashMap 的底层结构是什么？扩容机制是怎样的？为什么线程不安全？ +

底层结构：数组 + 链表 + 红黑树

  • Node<K,V>[] table，每个桶挂一条链表；JDK 8 起链表长度 ≥ 8 且数组长度 ≥ 64 时树化为红黑树，查询从 O(n) 降为 O(logN)；树节点数 ≤ 6 时退化回链表
  • 桶下标计算：hash(key) = h ^ (h >>> 16)（扰动函数）后 & (capacity - 1)，要求容量必为 2 的幂

扩容机制：默认初始容量 16，负载因子 0.75，size > capacity × 0.75 时触发扩容，新容量为旧容量的 2 倍。JDK 8 优化：无需重新计算 hash，通过 (hash & oldCap) == 0 判断节点留在原下标还是移动到「原下标 + oldCap」，均摊了扩容开销。

为什么非线程安全：

  • JDK 7：扩容用头插法迁移节点，多线程并发扩容可能形成环形链表，导致 get() 死循环、CPU 100%
  • JDK 8：改为尾插法，不会死循环，但并发 put 仍可能互相覆盖数据，并发 resize() 可能丢失节点
  • size++ 非原子操作，并发下计数不准确

解决方案：多线程场景使用 ConcurrentHashMap，而非 Collections.synchronizedMap() 或自行加锁。

虚拟线程（JDK 21）

Q16 JDK 21 虚拟线程是什么？和平台线程有什么区别？有哪些使用陷阱？ +

虚拟线程（Virtual Thread，JEP 444）：由 JVM 调度的轻量级线程，挂载在少量平台线程（Carrier Thread）上运行。当虚拟线程阻塞于 IO 时，自动从载体线程卸载，载体线程可立即执行其他虚拟线程，从而用少量 OS 线程支撑百万级并发。

核心差异：平台线程由 OS 调度，内存约 1 MB；虚拟线程由 JVM 调度，内存仅几 KB，创建成本极低，适合 IO 密集型场景。

三大使用陷阱：

  • Pinning（钉住）：虚拟线程在 synchronized 块内发生 IO 阻塞时，无法卸载，会一直占用载体线程，退化为平台线程行为。解决：将 IO 密集临界区的 synchronized 换成 ReentrantLock（JDK 24 已修复）。
  • 不要池化：虚拟线程创建极廉价，池化反而浪费。应使用 Executors.newVirtualThreadPerTaskExecutor()，每任务一个虚拟线程。
  • ThreadLocal 内存压力：百万虚拟线程各自持有 ThreadLocal 会消耗大量堆内存。推荐改用 JDK 21 的 Scoped Values（JEP 446），不可变、生命周期明确。

不适合场景：CPU 密集型计算（虚拟线程无法在计算途中卸载，不带来吞吐提升）。

Spring Boot 接入：spring.threads.virtual.enabled=true（Spring Boot 3.2+），Tomcat/Jetty 请求处理线程自动切换为虚拟线程。

⚡ 实战题

⚡ 实战 线上 CPU 100%，怎么定位到具体线程和代码行？ +

排查步骤（经典四步法）：

第一步：找到高 CPU 的 Java 进程 PID。使用 top 命令查看系统进程，按 P 键按 CPU 使用率排序，找到占用最高的 Java 进程，记下其 PID。也可以使用 jps -l 直接列出 Java 进程。
top -b -n 1 | head -20

第二步：找到该进程中高 CPU 的线程。使用 top -Hp <pid> 列出该进程下所有线程的 CPU 占用，找到 CPU 最高的线程 TID（Thread ID）。
top -Hp 12345
例如线程 TID 为 8801。

第三步：将线程 ID 转换为 16 进制。jstack 输出中的线程 nid 是 16 进制的，因此需要将 10 进制 TID 转换：
printf '%x' 8801
结果为 2261。

第四步：导出线程栈并搜索。使用 jstack 导出线程堆栈，搜索该 nid：
jstack 12345 | grep -A 30 'nid=0x2261'
输出中即可看到具体的方法调用栈和代码行号，例如 com.example.OrderService.processOrder(OrderService.java:127)。

常见原因：

  • 死循环：最常见，如 while 条件永远为 true、HashMap 并发 resize 导致链表环（JDK 7）。
  • 频繁 Full GC：GC 线程占用大量 CPU。通过 jstat -gcutil <pid> 1000 查看 GC 频率，若 Old 区频繁打满则需排查内存泄漏。
  • 正则回溯：复杂正则表达式在特定输入上触发指数级回溯，线程栈中可见 java.util.regex.Pattern$xxx.match。

Arthas 快捷方式：如果线上已挂载 Arthas，一条命令即可定位：
thread -n 3
直接显示 CPU 使用率最高的 3 个线程及其完整调用栈，省去手动 grep 的步骤。profiler start 还可以生成火焰图，直观展示 CPU 热点。

⚡ 实战 线程池任务堆积了怎么处理？ +

一、紧急处理（止血）：

任务堆积意味着队列持续增长，延迟变大，甚至触发 RejectedExecutionException。首先快速扩容：如果线程池参数无法动态调整，最快的方法是新建一个临时线程池，将新提交的任务路由到临时池处理，避免继续堆积。

二、根因分析（定位）：堆积的根本原因只有两类——消费变慢或生产变快。

  • 任务执行时间变长：检查任务内部是否有慢查询、外部 RPC 超时、锁竞争。通过 jstack 查看线程池线程的调用栈，若大量阻塞在 IO 或等待锁，说明下游服务变慢。
  • 提交速率突增：上游流量激增（如营销活动、爬虫），导致任务提交速率超过消费速率。
  • 拒绝策略是否合理：默认的 AbortPolicy 会直接抛异常，生产环境应考虑 CallerRunsPolicy（让提交线程自己执行，自动限流）或自定义策略（写日志/发消息队列稍后重试）。

三、监控指标（可观测性）：线程池必须接入监控，核心指标包括：

  • executor.getActiveCount() — 当前活跃线程数，接近 maximumPoolSize 说明线程资源紧张。
  • executor.getQueue().size() — 队列中等待的任务数，持续增长即为堆积信号。
  • executor.getCompletedTaskCount() — 已完成任务总数，配合时间窗口可算出吞吐量（TPS）。

建议将以上指标暴露到 Prometheus/Grafana，设置队列长度告警阈值（如队列使用率 > 80%）。

四、长期方案（根治）：

  • 动态线程池：接入 Hippo4j 或 DynamicTp 等开源框架，支持运行时动态调整核心线程数、最大线程数、队列容量，无需重启。
  • 合理设置拒绝策略：生产环境推荐 CallerRunsPolicy（反压限流）或自定义策略将溢出任务写入 MQ 异步重试，避免任务丢失。
  • 慢任务异步化：将耗时任务拆分为多步异步流水线，使用 CompletableFuture 编排，避免单个长耗时任务占住线程。
  • 核心参数经验值：CPU 密集型 corePoolSize = N + 1，IO 密集型 corePoolSize = 2N（N 为 CPU 核心数），但最终应以压测数据为准。
