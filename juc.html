并发编程核心知识

Java 并发编程涵盖内存可见性、原子操作、锁机制、线程池和并发容器等核心主题。 理解底层原理（JMM、AQS、CAS）是写出高性能、线程安全代码的基础，也是面试必考方向。

Java 内存模型（JMM）

JMM 定义了多线程程序中变量的读写规则，解决可见性、有序性和原子性三大并发问题。

可见性 一个线程修改共享变量后，另一个线程不能立即看到最新值。原因：每个线程有自己的工作内存（CPU 缓存），不直接操作主内存。
有序性 JVM 和 CPU 会对指令重排序优化，单线程下结果一致，多线程下可能产生意外行为（如双重检查锁的经典 bug）。
原子性 非 long/double 的基本类型读写是原子的，但复合操作（如 i++）不是。JMM 通过 synchronized / CAS 保证复合操作的原子性。

面试要点：JMM 不是具体实现，是规范。volatile 和 synchronized 都是 JMM 规范在 Java 语言层面的实现手段。happens-before 是 JMM 向程序员暴露的最简洁的内存可见性契约。

volatile 原理

volatile 通过内存屏障保证可见性与有序性，但不保证复合操作的原子性。

可见性 写 volatile 变量时强制将工作内存中的值刷回主内存；读 volatile 变量时强制从主内存重新加载，使其他线程的写立即可见。
有序性 通过插入内存屏障禁止特定方向的指令重排序，解决如双重检查锁（DCL）中对象半初始化的问题。
不保证 不保证原子性。volatile int i; i++ 等价于 read → modify → write 三步，多线程下仍会丢失更新，需用 AtomicInteger。

内存屏障插入规则

位置屏障类型作用
volatile 写之前StoreStore禁止上面的普通写与 volatile 写重排
volatile 写之后StoreLoad禁止 volatile 写与下面的 volatile 读重排（最重屏障）
volatile 读之后LoadLoad禁止 volatile 读与下面的普通读重排
volatile 读之后LoadStore禁止 volatile 读与下面的普通写重排

经典案例：双重检查锁（DCL）单例

  // 错误写法（没有 volatile，指令重排可能导致其他线程拿到半初始化对象）
  private static Singleton instance;

  // 正确写法
  private static volatile Singleton instance;

  public static Singleton getInstance() {
      if (instance == null) {              // 第一次检查（无锁）
          synchronized (Singleton.class) {
              if (instance == null) {      // 第二次检查（加锁）
                  instance = new Singleton();
                  // new 分三步：1.分配内存 2.初始化对象 3.赋值引用
                  // 无 volatile 时 2 和 3 可能被重排 → 其他线程看到非 null 但未初始化的对象
              }
          }
      }
      return instance;
  }

CAS 与原子类

CAS（Compare-And-Swap）是乐观锁的底层实现，依赖 CPU 原子指令。java.util.concurrent.atomic 包基于 CAS 提供无锁原子操作。

CAS 原理

  CAS(V, expected, newValue)
    if V == expected:
        V = newValue   // 原子操作，由 CPU 指令（x86: CMPXCHG）保证
        return true
    else:
        return false   // 失败则自旋重试

  // AtomicInteger.incrementAndGet() 内部实现
  do {
      current = get();       // 读取当前值
  } while (!compareAndSet(current, current + 1));  // CAS 失败则重试

CAS 的三大缺陷

ABA 问题 变量从 A→B→A，CAS 认为没有变化但实际已被修改。解决：AtomicStampedReference 附加版本号（stamp），每次修改 stamp+1。
自旋开销 高并发竞争下 CAS 长时间自旋，大量消耗 CPU。解决：限制自旋次数，或升级为重量级锁（synchronized）。
只能保证单变量 CAS 只能保证一个共享变量的原子操作。多变量需用 AtomicReference 封装对象，或用 synchronized。

自定义同步器只需重写

  // 独占锁需重写：
  protected boolean tryAcquire(int arg)    // 尝试获取锁
  protected boolean tryRelease(int arg)    // 尝试释放锁

  // 共享锁需重写：
  protected int  tryAcquireShared(int arg) // 返回负数=失败，0=成功但无剩余，正数=成功且有剩余
  protected boolean tryReleaseShared(int arg)
面试要点：AQS 用 LockSupport.park/unpark 阻塞/唤醒线程（不需要持有监视器，比 wait/notify 更灵活）。CLH 队列是虚拟的，节点代表线程，通过 CAS 入队保证线程安全。公平锁与非公平锁的区别仅在 tryAcquire 中是否先检查队列。

常用锁

ReentrantLock、ReentrantReadWriteLock、StampedLock 分别适用于不同的并发场景，各有性能与功能取舍。

锁类型底层可重入公平锁读写分离适用场景
synchronizedMonitor / 对象头✅❌❌简单互斥，JDK 内置优化

ReentrantLock 相比 synchronized 的额外能力

StampedLock 三种模式

写锁writeLock() → 返回 stamp，操作完后 unlockWrite(stamp)
读锁readLock() → 悲观读，与写互斥
乐观读tryOptimisticRead() → 不加锁，读后用 validate(stamp) 检查是否被写过；失败则升级为悲观读锁。适合极短的读操作。
  long stamp = lock.tryOptimisticRead();
  // 读取数据（可能被写线程修改）
  double x = this.x, y = this.y;
  if (!lock.validate(stamp)) {       // 期间有写操作，升级为读锁
      stamp = lock.readLock();
      try { x = this.x; y = this.y; }
      finally { lock.unlockRead(stamp); }
  }
注意：StampedLock 不支持重入，不支持 Condition，且 stamp 用完后不能复用。线程被中断时可能永久阻塞，需用 tryConvertToWriteLock 等带超时的 API。
面试要点：读写锁适合读多写少，但写锁等待期间读锁积压会导致写饥饿。StampedLock 乐观读性能最佳但使用复杂。日常推荐优先 synchronized，需要高级特性用 ReentrantLock，读多写少用 ReadWriteLock。

线程数配置经验公式

CPU 密集型N = CPU 核心数 + 1（+1 防止偶发页缺失等停顿，充分利用 CPU）
IO 密集型N = CPU 核心数 × (1 + 等待时间 / 计算时间)，实践中常用 2 × CPU 核心数
混合型拆分为 CPU 密集和 IO 密集两个独立线程池，避免相互影响

BlockingQueue 系列ReentrantLock + Condition中中生产者-消费者模型

HashMap 底层结构（数组 + 链表 + 红黑树）

存储结构 Node<K,V>[] table 数组，每个桶（bucket）挂一条链表；JDK 8 起当链表过长时转为红黑树（TreeNode），加速查找
树化阈值 单个桶的链表长度 ≥ 8 且数组容量 ≥ 64 时树化为红黑树；数组容量不足 64 时优先扩容而非树化
退化阈值 扩容重新分布后，红黑树节点数 ≤ 6 时退化回链表，避免小数据量下维护红黑树的额外开销
寻址方式 hash(key) = h ^ (h >>> 16)（扰动函数，让高位也参与运算，减少碰撞），桶下标 = hash & (table.length - 1)（要求数组长度必为 2 的幂，等价于取模但更快）

HashMap 扩容机制

默认参数初始容量 16，负载因子 0.75，扩容阈值 threshold = capacity × loadFactor（12）
触发条件size > threshold 时触发 resize()，新容量为旧容量的 2 倍（保持 2 的幂）
JDK 8 优化扩容时无需重新计算 hash，利用 (hash & oldCap) == 0 判断节点去向：为 0 留在原下标，非 0 则移到「原下标 + oldCap」，天然分裂成 lo / hi 两条链表
扩容时节点迁移逻辑（无需重新计算 hash）
(hash & oldCap) == 0
新下标 = 原下标
留在 lo 链表（位置不变）
(hash & oldCap) != 0
新下标 = 原下标 + oldCap
移到 hi 链表（位置后移）
示例：hash = 21，oldCap = 16
21 = 0b10101，16 = 0b10000
21 & 16 = 16 ≠ 0 → 迁移到新下标 (21 % 16) + 16 = 5 + 16 = 21

HashMap 的线程安全问题

JDK 7：扩容死循环 JDK 7 用头插法迁移节点，多线程并发扩容时，若线程 A 的 next 指针被线程 B 的迁移打断，可能形成环形链表，导致后续 get() 死循环，CPU 飙升到 100%
JDK 8：数据覆盖/丢失 JDK 8 改为尾插法，不再死循环，但并发 put 仍可能因竞争同一桶而互相覆盖（后完成的线程覆盖先完成线程的写入），或并发 resize() 导致部分节点丢失
size 计数不准 size++ 是非原子操作，多线程并发 put 会导致计数丢失，实际元素数与 size() 返回值不一致
根本原因 HashMap 所有方法均无任何同步措施（无 synchronized、无 CAS、无 volatile 修饰核心字段），设计上就是非线程安全容器
注意：多线程场景禁止直接使用 HashMap，应改用 ConcurrentHashMap；仅需简单同步包装可用 Collections.synchronizedMap()，但性能远不如 ConcurrentHashMap。

ConcurrentHashMap（JDK 8）实现原理

数据结构 Node 数组 + 链表 + 红黑树（链表长度 ≥ 8 且数组长度 ≥ 64 时树化，提升查询 O(n)→O(logN)）
锁粒度 桶头节点级别：桶为空用 CAS 插入；桶非空用 synchronized 锁住桶头节点。并发度 = 桶数量，远高于全表锁
size() 类似 LongAdder，baseCount + CounterCell[] 分段累加，是估算值（非强一致）
扩容 支持多线程协助迁移（helpTransfer），桶 hash == MOVED(-1) 时触发协助

CopyOnWriteArrayList 原理

读操作直接读 volatile 数组引用，无锁，不阻塞写线程
写操作加 ReentrantLock，复制整个数组，在新数组上修改，最后 volatile 写替换引用
弱一致性迭代器快照创建时的数组，写后新迭代器才能看到变化（不抛 ConcurrentModificationException）
内存代价每次写都复制全量数组，大数据量 + 频繁写时内存和 GC 压力大

BlockingQueue 主要实现

实现类有界底层结构特点
ArrayBlockingQueue✅ 固定容量数组一把锁（put/take 共用），公平可选
LinkedBlockingQueue可选（默认 Integer.MAX_VALUE）链表put/take 两把锁，吞吐量更高
SynchronousQueue0（不存储）—直接交换，CachedThreadPool 使用
PriorityBlockingQueue❌ 无界堆按优先级出队
DelayQueue❌ 无界堆元素到期后才可取，用于延迟任务
面试要点：ConcurrentHashMap 的 size() 是估算值（非强一致）。CopyOnWrite 适合"读远多于写"场景，不适合实时性要求高的场景。LinkedBlockingQueue 的两把锁（putLock / takeLock）比 ArrayBlockingQueue 的单锁吞吐量更高。

虚拟线程（Virtual Threads，JDK 21）

JEP 444 在 JDK 21 正式落地。虚拟线程是由 JVM 调度的轻量级线程，极大降低了 IO 密集型应用的线程开销，是对传统"一请求一线程"模型的根本性改进。

平台线程 vs 虚拟线程

维度平台线程（Platform Thread）虚拟线程（Virtual Thread）
调度者操作系统JVM（ForkJoinPool 作为载体线程池）
内存占用~1 MB（OS 栈）~几 KB（堆上动态扩展）
可创建数量千级（受 OS 限制）百万级
创建成本高（系统调用）极低
适用场景CPU 密集型IO 密集型（数据库、HTTP、文件）
与现有 API 兼容✅✅（Thread 子类，接口完全兼容）

创建方式

  // 方式 1：Thread.ofVirtual()
  Thread vt = Thread.ofVirtual().name("vt-1").start(() -> doWork());

  // 方式 2：ExecutorService（推荐，每个任务一个虚拟线程）
  try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
      executor.submit(() -> handleRequest());
  }

  // 方式 3：Thread.startVirtualThread()（快捷）
  Thread.startVirtualThread(() -> doWork());

使用注意事项

Pinning（钉住） 虚拟线程执行 synchronized 块内发生 IO 阻塞时，会"钉住"载体线程无法卸载，退化为平台线程行为。解决：IO 密集的临界区改用 ReentrantLock 替换 synchronized。JDK 24 已修复此限制。
不要池化 虚拟线程创建代价极低，不需要也不应该放入线程池复用（违背设计理念）。使用 newVirtualThreadPerTaskExecutor()，每任务创建一个。
ThreadLocal 慎用 百万级虚拟线程各自携带 ThreadLocal 会带来巨大内存压力。JDK 21 引入 Scoped Values（JEP 446）作为替代，不可变、作用域明确、性能更好。
不适合 CPU 密集 虚拟线程解决的是"线程在 IO 上等待"问题，CPU 密集型任务仍应使用平台线程池（如 ForkJoinPool）。

配套特性：结构化并发（Structured Concurrency，JEP 453）

  // 传统写法：子任务异常难以传播，容易泄露
  Future<User>  user  = executor.submit(() -> fetchUser(id));
  Future<Order> order = executor.submit(() -> fetchOrder(id));

  // 结构化并发：父任务生命周期管理所有子任务，任一失败则取消其他
  try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
      Subtask<User>  user  = scope.fork(() -> fetchUser(id));
      Subtask<Order> order = scope.fork(() -> fetchOrder(id));
      scope.join().throwIfFailed();        // 等待全部完成，任一失败抛异常
      return new Response(user.get(), order.get());
  }
面试要点：虚拟线程不是协程，是 Java 官方的"有栈协程"实现。它与现有 API（JDBC、synchronized、ThreadLocal）完全兼容，但部分场景有性能陷阱（Pinning）。Spring Boot 3.2+ 已支持一键开启虚拟线程：spring.threads.virtual.enabled=true。
