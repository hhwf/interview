JVM 内存与类加载核心知识

JVM 是 Java 程序的运行基石。理解运行时数据区的划分、类从加载到卸载的完整生命周期、以及 JIT 如何将热点代码编译为机器码， 对排查 OOM、调优性能、理解框架原理（如 Tomcat 类加载、SPI 打破委派）都至关重要。

运行时数据区

JVM 运行时数据区定义了程序执行期间的内存划分，分为线程私有和线程共享两大类。

  ┌─────────────────────────────────────────────────────────────────┐
  │                    JVM 运行时数据区                               │
  ├──────────────────────────┬──────────────────────────────────────┤
  │      线程私有             │          线程共享                     │
  ├──────────┬───────────────┼───────────────┬──────────────────────┤
  │ 程序计数器│  虚拟机栈      │     堆        │   方法区              │
  │  (PC)    │  (VM Stack)   │   (Heap)      │  (Metaspace, JDK8+) │
  ├──────────┼───────────────┼───────────────┼──────────────────────┤
  │ 当前执行  │ 栈帧:         │  对象实例      │  类元信息             │
  │ 字节码行号│ 局部变量表     │  数组          │  常量池              │
  │          │ 操作数栈      │               │  JIT 编译代码         │
  │          │ 动态链接      │  Young / Old  │                      │
  │          │ 返回地址      │               │  (PermGen, JDK7-)    │
  ├──────────┴───────────────┼───────────────┴──────────────────────┤
  │      本地方法栈           │        直接内存 (NIO DirectBuffer)    │
  │   (Native Method Stack)  │        不受 JVM 堆大小限制            │
  └──────────────────────────┴─────────────────────────────────────┘
程序计数器 线程私有。记录当前线程执行的字节码行号，用于线程切换后恢复执行位置。是唯一不会发生 OOM 的区域。
虚拟机栈 线程私有。每个方法调用创建一个栈帧，包含局部变量表、操作数栈、动态链接、返回地址。栈深度超过限制抛出 StackOverflowError；动态扩展失败抛出 OOM。
本地方法栈 线程私有。为 Native 方法（C/C++ 实现）服务，结构与虚拟机栈类似。HotSpot 将两者合并。
堆 线程共享。存放所有对象实例和数组，是 GC 的主要区域。分为 Young Generation（Eden + S0 + S1）和 Old Generation。-Xms 设初始大小，-Xmx 设最大大小。
方法区 线程共享。存储已加载类的元信息、常量池、静态变量、JIT 编译后的代码。JDK 7 用 PermGen（-XX:MaxPermSize），JDK 8+ 改为 Metaspace（使用本地内存，-XX:MaxMetaspaceSize 限制）。
直接内存 不属于 JVM 运行时数据区，但被频繁使用。NIO 的 DirectByteBuffer 通过 Unsafe.allocateMemory 在堆外分配，避免 JVM 堆与 Native 堆之间的数据拷贝，适合大缓冲区场景（Netty、Kafka）。

栈帧结构

  ┌───────────────────────────────┐
  │          栈帧 (Stack Frame)    │
  ├───────────────────────────────┤
  │  局部变量表 (Local Variables)  │  ← 编译期确定大小（Slot 数）
  │  操作数栈 (Operand Stack)     │  ← 字节码指令 push/pop
  │  动态链接 (Dynamic Linking)    │  ← 指向运行时常量池的方法引用
  │  返回地址 (Return Address)    │  ← 方法结束后回到调用点
  └───────────────────────────────┘

局部变量表以 Slot 为单位（32 bit），long 和 double 占 2 个 Slot。方法参数和局部变量都存放在此表。Slot 复用：一个局部变量超出作用域后，其 Slot 可被后续变量复用，减少内存占用。

对象内存布局

HotSpot 虚拟机中，对象在堆内存中的布局分为三个区域：对象头、实例数据、对齐填充。

  ┌──────────────────────────────────────────────────────────┐
  │                     对象内存布局                           │
  ├──────────────┬────────────────┬──────────────────────────┤
  │   对象头      │    实例数据      │    对齐填充              │
  │  (Header)    │  (Instance)    │    (Padding)            │
  ├──────────────┼────────────────┼──────────────────────────┤
  │ Mark Word    │  父类继承字段   │  补齐到 8 字节的整数倍    │
  │ (8 bytes/64) │  子类实例字段   │  空对象也需要 16 bytes   │
  │              │               │                          │
  │ Klass Pointer│  (按类型排列    │                          │
  │ (4/8 bytes)  │   减少空隙)    │                          │
  │              │               │                          │
  │ [数组长度]    │               │                          │
  │ (4 bytes)    │               │                          │
  └──────────────┴────────────────┴──────────────────────────┘

Mark Word（64 位 JVM，8 字节）

无锁unused(25) + hashCode(31) + GC 分代年龄(4) + 偏向标志(1) + 锁标志(2=01)
偏向锁线程 ID(54) + epoch(2) + 分代年龄(4) + 偏向标志(1) + 锁标志(2=01)
轻量级锁指向栈中锁记录的指针(62) + 锁标志(2=00)
重量级锁指向 Monitor 对象的指针(62) + 锁标志(2=10)
GC 标记空(62) + 锁标志(2=11)

Klass Pointer 与指针压缩

Klass Pointer 指向方法区中该对象的类元信息（Klass 结构）。64 位 JVM 默认开启压缩指针（-XX:+UseCompressedOops），将 8 字节指针压缩为 4 字节，可寻址 32 GB 堆空间（4 字节 × 8 对齐偏移）。压缩后对象头从 16 字节缩减为 12 字节，大幅降低内存开销。

对象大小计算：对象头(12) + 实例字段（对齐到 8 字节倍数）。空 Object 占 16 字节（12 头 + 4 padding）。开启 -XX:+UseCompactObjectHeaders（JDK 24+ Lilliput）可进一步将对象头压缩到 8 字节。

Class 文件结构

Class 文件是一组以 8 位字节为基础的二进制流，采用伪结构体方式组织，只有两种数据类型：无符号数和表。用 javap -c -v MyClass 可以查看完整的 Class 文件内容。

  ClassFile {
    u4   magic;                    // 魔数: 0xCAFEBABE
    u2   minor_version;            // 次版本号
    u2   major_version;            // 主版本号 (52=JDK8, 55=JDK11, 65=JDK21)
    u2   constant_pool_count;      // 常量池计数（从 1 开始）
    cp_info constant_pool[];       // 常量池表（字符串、类名、方法引用等）
    u2   access_flags;             // 访问标志（ACC_PUBLIC, ACC_FINAL...）
    u2   this_class;               // 当前类在常量池中的索引
    u2   super_class;              // 父类索引（Object 为 0）
    u2   interfaces_count;         // 接口数量
    u2   interfaces[];             // 接口索引数组
    u2   fields_count;             // 字段数量
    field_info fields[];           // 字段表（名称、描述符、访问标志、属性）
    u2   methods_count;            // 方法数量
    method_info methods[];         // 方法表（含 Code 属性 → 字节码指令）
    u2   attributes_count;         // 类属性数量
    attribute_info attributes[];   // 类属性（SourceFile, InnerClasses...）
  }

关键字段解读

常量池 Class 文件中最大的部分。存放字面量（字符串、final 常量值）和符号引用（类名、方法名、字段名）。运行时解析为直接引用（内存地址）。常量池也是反射、动态代理等机制的底层数据来源。
字段表 描述类中声明的变量。每个字段包含 access_flags、name_index、descriptor_index 和属性集合。字段描述符：I=int, J=long, Ljava/lang/String;=String 等。
方法表 描述类中声明的方法。方法的字节码指令存放在 Code 属性中，包含 max_stack（操作数栈最大深度）、max_locals（局部变量表 Slot 数）、字节码数组和异常表。
访问标志 ACC_PUBLIC(0x0001)、ACC_FINAL(0x0010)、ACC_SUPER(0x0020)、ACC_INTERFACE(0x0200)、ACC_ABSTRACT(0x0400)、ACC_ENUM(0x4000) 等位掩码组合。

实战工具：javap -c -p -v MyClass.class 反编译查看完整结构；ASM / ByteBuddy 可在运行时动态生成或修改 Class 文件（AOP、热部署的底层原理）。

类加载机制

类加载是将 Class 文件的二进制数据读入 JVM，并在方法区创建对应的 Class 对象的过程。完整生命周期包含加载、验证、准备、解析、初始化、使用、卸载 7 个阶段，其中验证、准备、解析统称为连接（Linking）。

  ┌──────────┐   ┌─────────────────────────────────┐   ┌────────┐   ┌──────┐
  │          │   │          连接 (Linking)            │   │        │   │      │
  │  加载    │──▶│  验证 ──▶ 准备 ──▶ 解析           │──▶│ 初始化 │──▶│ 使用 │
  │ (Loading)│   │ (Verify) (Prepare) (Resolve)     │   │(Init)  │   │      │
  │          │   │                                  │   │        │   │      │
  └──────────┘   └─────────────────────────────────┘   └────────┘   └──────┘
                                                                    │
                                                              ┌─────▼────┐
                                                              │   卸载   │
                                                              │(Unload)  │
                                                              └──────────┘
加载 通过全限定名获取二进制字节流（可从 Class 文件、JAR、网络、动态生成等来源），将字节流代表的静态存储结构转化为方法区的运行时数据结构，在堆中生成 java.lang.Class 对象作为方法区入口。
验证 确保 Class 文件格式正确、语义合法、不会危害 JVM。包括文件格式验证（魔数、版本号）、元数据验证（继承链、final 约束）、字节码验证（数据流/控制流分析）、符号引用验证（解析阶段的引用是否存在）。可用 -Xverify:none 跳过以加速启动。
准备 为类的静态变量分配内存并设零值。注意：final static 常量在编译时已确定值，准备阶段直接赋值（不取零值）。例：static int a = 123; 准备阶段 a=0，初始化阶段 a=123；static final int b = 123; 准备阶段 b=123。
解析 将常量池中的符号引用（名称字符串）替换为直接引用（内存地址/偏移量）。解析时机因 JVM 实现而异：可在类加载时一次性解析（静态解析），也可在首次使用时延迟解析（动态解析）。
初始化 执行类的 <clinit> 方法（编译器收集 static 语句块和静态变量赋值语句生成）。初始化是类加载的最后一步，至此才真正执行 Java 代码。JVM 保证 <clinit> 在多线程环境下的同步加锁。

主动引用（触发初始化）vs 被动引用（不触发）

主动引用① new 对象 / 读写静态字段 / 调用静态方法 ② 反射调用（Class.forName）③ 初始化子类时先初始化父类 ④ main 方法所在类 ⑤ MethodHandle 解析结果类
被动引用① 通过子类引用父类的静态字段 → 只初始化父类 ② 通过数组定义引用类 → 不触发初始化 ③ 引用类的常量（编译期已存入调用方常量池）→ 不触发初始化

双亲委派模型

双亲委派模型要求：类加载请求先委托父加载器处理，父加载器无法完成时才由自身加载。这不是强制约束，而是一种约定，通过组合复用而非继承实现。

  ┌──────────────────────────────────────────────────────┐
  │           Bootstrap ClassLoader (C++)                 │
  │     加载: JAVA_HOME/lib (rt.jar, java.lang.*)       │
  │     加载路径: sun.boot.class.path                    │
  └──────────────────────┬───────────────────────────────┘
                         │ parent
  ┌──────────────────────▼───────────────────────────────┐
  │          Extension ClassLoader (Java)                 │
  │     加载: JAVA_HOME/lib/ext 或 java.ext.dirs         │
  │     实现: sun.misc.Launcher$ExtClassLoader            │
  └──────────────────────┬───────────────────────────────┘
                         │ parent
  ┌──────────────────────▼───────────────────────────────┐
  │          Application ClassLoader (Java)               │
  │     加载: 用户类路径 (classpath / -cp / -classpath)   │
  │     实现: sun.misc.Launcher$AppClassLoader            │
  └──────────────────────┬───────────────────────────────┘
                         │ parent
  ┌──────────────────────▼───────────────────────────────┐
  │          自定义 ClassLoader (Java)                     │
  │     重写 findClass() + defineClass()                  │
  │     场景: 热部署、插件隔离、加密 Class                 │
  └──────────────────────────────────────────────────────┘

为什么需要双亲委派？

安全性 防止核心类被篡改。自定义一个 java.lang.String 不会被加载，因为 Bootstrap 加载器优先加载 JDK 自带的 String，保证 JRE 核心类库的完整性和安全性。
避免重复 父加载器已加载的类不会被子加载器重复加载，保证 Class 对象的全局唯一性（同一个类在 JVM 中只有一个 Class 对象）。

打破双亲委派的典型场景

SPI 机制 JDBC、SLF4J 等使用 SPI（Service Provider Interface）加载第三方实现。核心接口在 Bootstrap 加载，但实现类在 classpath。解决方式：引入线程上下文类加载器（Thread.setContextClassLoader），由父加载器反向委托子加载器加载实现。
Tomcat 每个 WebApp 独立类加载器（WebAppClassLoader），优先加载自己 WEB-INF/classes 和 lib 下的类（打破委派），实现不同应用间的类隔离。共享类（如 Spring）由公共父加载器加载。
OSGi / 热部署 OSGi 使用网状委派模型（非树形），每个 Bundle 有独立类加载器，按规则在 Bundle 间共享类。热部署（如 Spring DevTools）通过丢弃旧 ClassLoader 并创建新的来实现类的重新加载。

自定义 ClassLoader 最佳实践：重写 findClass()（保持双亲委派）而不是 loadClass()（打破委派）。loadClass 默认逻辑是先调 parent.loadClass()，找不到再调 findClass()；如果直接重写 loadClass 就跳过了委派逻辑。

JIT 编译与调优

JVM 执行程序有两种方式：解释执行（逐条翻译字节码）和JIT 编译执行（将热点字节码编译为本地机器码缓存）。JIT 编译器在运行时检测热点代码并编译，可大幅提升性能（通常 10~100 倍）。

C1 / C2 / 分层编译

C1 (Client) 编译速度快，优化程度低。适合简单方法。插入 profiling 数据收集点（分支计数、类型 profile），为 C2 提供优化依据。
C2 (Server) 编译速度慢，激进优化（内联、逃逸分析、标量替换、锁消除、循环展开）。适合热点方法。使用 C1 收集的 profiling 数据做精确优化。
分层编译 JDK 8 默认开启（-XX:+TieredCompilation）。5 个层级：Level 0 解释执行 → Level 1~3 C1 编译（逐步增加 profiling）→ Level 4 C2 编译。方法按调用频率在各层级间晋升。

热点代码检测

方法调用计数器 统计方法被调用次数。超过阈值（半衰期机制，默认 -XX:CompileThreshold=10000）触发 JIT 编译。长期未调用的方法计数器会衰减归零（半衰期由 -XX:+UseCounterDecay 控制）。
回边计数器 统计循环中的回边（向后跳转）次数。用于触发循环体的 OSR（On-Stack Replacement），在循环执行中途将解释执行替换为编译后的机器码。

关键 JIT 优化技术

方法内联 将方法体直接嵌入调用点，消除方法调用开销（栈帧创建/销毁、参数传递）。是最重要的优化手段。-XX:MaxInlineLevel=9 控制内联深度。final 方法更容易被内联（但现在非 final 方法也可以，通过 CHA 分析）。
逃逸分析 分析对象的引用是否逃逸出方法。未逃逸的对象可优化为：栈上分配（不用堆，方法结束自动回收）、标量替换（拆成基本类型，直接存寄存器）、锁消除（不存在竞争，移除 synchronized）。
去优化 JIT 编译时基于激进假设（如类型固定为某子类），运行时若假设被打破（加载了新子类），会丢弃编译结果回退到解释执行，称为去优化（Deoptimization）。频繁去优化会严重影响性能。

常用 JVM 调优参数

-Xms / -Xmx堆初始大小 / 最大大小。生产环境通常设置为相同值，避免堆动态扩缩引起 GC 停顿。
-Xmn新生代大小。通常设为堆的 1/3 ~ 1/4。
-Xss线程栈大小（默认 1MB）。StackOverflowError 时可适当调大。
-XX:MetaspaceSizeMetaspace 初始大小。设大避免频繁 Full GC 扩容。
-XX:+UseG1GC使用 G1 垃圾收集器（JDK 9+ 默认）。
-XX:+UseZGC使用 ZGC 垃圾收集器（JDK 15+），适合大堆低延迟场景。
-XX:MaxGCPauseMillisG1 目标最大 GC 停顿时间（默认 200ms）。
-Xlog:gc*JDK 9+ 统一日志框架输出 GC 日志（替代 -XX:+PrintGCDetails）。生产必开，用于事后分析。
-XX:+HeapDumpOnOOMOOM 时自动 dump 堆内存快照，配合 MAT 分析定位泄漏对象。

JVM 面试题（15 题）

覆盖运行时数据区、对象布局、Class 文件、类加载、双亲委派、JIT 编译等核心考点。

Q1 JVM 运行时数据区有哪些部分？各自的作用和线程归属？ +

线程私有：

  • 程序计数器（PC）：记录当前线程执行的字节码行号，线程切换后据此恢复执行位置。唯一不会 OOM 的区域。
  • 虚拟机栈：每个方法调用创建栈帧（局部变量表、操作数栈、动态链接、返回地址）。递归过深抛 StackOverflowError，扩展失败抛 OOM。
  • 本地方法栈：为 Native 方法服务，HotSpot 中与虚拟机栈合并。

线程共享：

  • 堆：存放所有对象实例和数组，GC 主战场。-Xms/-Xmx 控制大小。
  • 方法区（JDK 8+ 为 Metaspace）：存储类元信息、常量池、静态变量、JIT 编译代码。JDK 7 用 PermGen（受 -XX:MaxPermSize 限制），JDK 8+ 改为 Metaspace（使用本地内存）。

堆外：直接内存（NIO DirectByteBuffer），不受 -Xmx 限制但受物理内存限制。

Q2 堆和栈的区别是什么？为什么局部变量比对象快？ +

区别：

  • 栈：线程私有，自动回收（方法退出即释放），存取速度快（仅次于寄存器），空间小（默认 1MB）。存放基本类型值和对象引用。
  • 堆：线程共享，由 GC 回收，存取速度较慢，空间大。存放对象实例。

为什么局部变量快：局部变量（基本类型）直接存在栈帧的局部变量表中，CPU 可直接操作，无需经过 GC 管理。而 new 出来的对象在堆上分配，涉及 GC 的内存分配和回收开销。

逃逸分析的进一步优化：JIT 的逃逸分析可以判定对象不逃逸，将其优化为栈上分配或标量替换，使对象也能享受栈的速度。

Q3 Java 对象的内存布局是怎样的？对象头包含什么信息？ +

三部分：对象头 + 实例数据 + 对齐填充。

对象头（64 位 JVM，开启压缩指针，共 12 字节）：

  • Mark Word（8 字节）：存储 hashCode、GC 分代年龄、锁状态标志（无锁/偏向锁/轻量级锁/重量级锁/GC 标记）。不同锁状态下存储内容不同。
  • Klass Pointer（4 字节）：指向方法区中类元信息（Klass 结构），开启压缩指针时从 8 字节压缩为 4 字节。

数组对象头：额外增加 4 字节的数组长度字段，所以数组对象头共 16 字节。

对齐填充：JVM 要求对象起始地址必须是 8 字节的整数倍，不够时用 padding 补齐。空 Object 最小占 16 字节。

Q4 什么是压缩指针（Compressed Oops）？为什么能节省内存？ +

背景：64 位 JVM 中所有指针为 8 字节，相比 32 位 JVM 内存开销近乎翻倍，且降低缓存命中率。

原理：JVM 堆中对象的地址都是 8 字节对齐的（低 3 位为 0），因此可以用 4 字节的值乘以 8 来表示 32 GB 地址空间内的任何对象地址。将 8 字节指针压缩为 4 字节。

影响：对象头从 16 字节降至 12 字节；Klass Pointer 从 8 字节降至 4 字节；对象引用字段从 8 字节降至 4 字节。整体堆内存节省约 40%。

限制：堆超过 ~32 GB 时压缩指针失效（4 字节寻址不够），指针恢复 8 字节。因此生产中 -Xmx 通常不超过 28~30 GB。

开关：-XX:+UseCompressedOops（JDK 6u23+ 默认开启）。

Q5 类加载的完整过程是什么？每个阶段做什么？ +

7 个阶段：加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载。中间三步合称"连接"。

加载通过全限定名找到二进制字节流，解析为方法区的运行时结构，生成 Class 对象。
验证文件格式验证（魔数、版本）、元数据验证（继承链）、字节码验证（控制流/数据流）、符号引用验证。
准备为静态变量分配内存并赋零值。注意 final static 在准备阶段直接赋编译期常量值。
解析将符号引用替换为直接引用（内存地址）。可以 eagerly 或 lazily 执行。
初始化执行 <clinit> 方法（static 语句块 + 静态变量赋值）。这是真正执行 Java 代码的第一步。

面试关键：准备阶段赋零值 vs 初始化阶段赋显式值，final static 的特殊处理。

Q6 什么情况下会触发类的初始化？什么情况不会？ +

触发初始化（主动引用，6 种）：

  1. new 对象、读写静态字段（非 final）、调用静态方法
  2. 反射调用（Class.forName、Method.invoke）
  3. 初始化子类时，先初始化父类
  4. 虚拟机启动时的 main 方法所在类
  5. MethodHandle 解析结果的类
  6. 接口中 default 方法的实现类初始化时，接口本身先初始化

不触发初始化（被动引用）：

  • 通过子类引用父类的静态字段 → 只有父类初始化，子类不初始化
  • 通过数组定义引用类（SuperClass[] arr = new SuperClass[10]）→ 不触发
  • 引用编译期常量（static final int VAL = 123）→ 值已存入调用方常量池，不触发定义类的初始化
Q7 双亲委派模型是什么？为什么要打破它？有哪些打破的场景？ +

双亲委派：类加载请求先向上委托给父加载器，父加载器无法处理时才由自身加载。层次：Bootstrap → Extension → Application → 自定义。

为什么需要：保证核心类安全性（防止伪造 java.lang.String）和类的全局唯一性。

打破场景：

  • SPI：JDBC 驱动、SLF4J 实现等在 classpath 下，由 Bootstrap 加载的接口类需要反向委托子加载器加载实现。通过线程上下文类加载器（Thread.setContextClassLoader）解决。
  • Tomcat：每个 WebApp 有独立的 WebAppClassLoader，优先加载 WEB-INF 下的类（打破委派），实现应用隔离。
  • OSGi：网状委派模型，Bundle 间按规则共享类。
  • 热部署：Spring DevTools 丢弃旧 ClassLoader 并创建新的来重新加载类。

自定义 ClassLoader 建议：重写 findClass()（保持委派）而非 loadClass()（打破委派），除非明确需要打破。

Q8 PermGen 和 Metaspace 有什么区别？为什么 JDK 8 要改？ +

PermGen（JDK 7 及以前）：方法区的实现，在 JVM 堆内，大小由 -XX:MaxPermSize 限制（默认 82MB）。问题：大小固定、难以预测、容易报 PermGen space OOM、GC 回收效率低。

Metaspace（JDK 8+）：

  • 从 JVM 堆中移出，使用本地内存（Native Memory）
  • 默认不限大小（受物理内存限制），可通过 -XX:MaxMetaspaceSize 设置上限
  • 自动扩容/缩容，减少 OOM 风险
  • 字符串常量池从方法区移到堆中（JDK 7 就已移动）
  • 类元数据存储在 Klass 结构中，由 Metaspace 分配器管理

改动原因：PermGen 大小难以预估（太小 OOM，太大浪费），且与 GC 的并发/并行回收不兼容。Metaspace 使用本地内存后，GC 可以独立管理堆内存，类元数据的回收也更高效。

Q9 JIT 编译的分层编译是怎么回事？C1 和 C2 有什么区别？ +

分层编译（Tiered Compilation，JDK 8 默认开启）：将编译分为 5 个层级，方法按调用频率逐步晋升：

Level 0解释执行，无 profiling
Level 1C1 编译，无 profiling（简单方法不会再晋升）
Level 2C1 编译，轻量 profiling（C2 队列满时的过渡）
Level 3C1 编译，完整 profiling（收集分支计数、类型 profile）
Level 4C2 编译，利用 profiling 数据做激进优化

C1（Client Compiler）：编译快，优化少，适合简单方法。主要做基本优化（常量折叠、方法内联）并插入 profiling 代码。

C2（Server Compiler）：编译慢，优化激进。做方法内联、逃逸分析、标量替换、锁消除、循环展开、死代码消除等高级优化。使用 C1 收集的 profiling 数据做精确决策。

JDK 21+ GraalVM Native Image：提前编译（AOT）为本地二进制，启动快、内存省，但编译时间长，适合云原生/Serverless 场景。

Q10 什么是逃逸分析？能带来哪些优化？ +

逃逸分析：JIT 编译器分析对象的引用范围（是否逃逸出方法或线程）。如果对象只在方法内使用（不逃逸），可以做以下优化：

  • 栈上分配：对象直接在栈帧上分配，方法结束自动回收，无需 GC 参与。（注意：HotSpot 目前不直接支持栈上分配，而是通过标量替换间接实现。）
  • 标量替换：将对象拆解为基本类型（标量），直接存储在寄存器或栈上。例如 Point 对象拆为 x、y 两个 int，完全消除对象创建。
  • 锁消除：如果对象不会逃逸且不存在竞争，JIT 可移除该对象上的 synchronized 锁。典型例子：StringBuffer.append() 在局部变量上使用时，锁被消除。

开关：-XX:+DoEscapeAnalysis（JDK 6u23+ 默认开启）。

面试注意：不要说"HotSpot 支持栈上分配"，准确说法是"通过标量替换达到类似栈上分配的效果"。

Q11 JVM 中对象的创建过程是什么？ +

完整流程：

  1. 类加载检查：JVM 遇到 new 指令时，检查该类是否已加载、解析、初始化，如果没有则先执行类加载。
  2. 分配内存：根据对象大小在堆中分配空间。分配方式取决于堆内存是否规整：
    • 指针碰撞（Bump the Pointer）：堆内存规整（Serial/ParNew 带 Compact），移动指针即可。需要 CAS + 失败重试或 TLAB 保证线程安全。
    • 空闲列表（Free List）：堆内存不规整（CMS），在列表中找足够大的空间。
  3. 内存空间初始化为零值：保证对象的字段不赋初值也能使用（Java 规范要求的默认值）。
  4. 设置对象头：GC 分代年龄、hashCode（延迟计算）、锁信息、Klass Pointer 等。
  5. 执行 <init> 方法：即构造函数，按程序代码进行初始化（赋显式值、执行构造代码块）。

TLAB（Thread Local Allocation Buffer）：每个线程预分配一块 Eden 空间，对象分配时直接在 TLAB 中 bump pointer，无需竞争，大幅提升分配速度。

Q12 什么是 StackOverflowError 和 OOM？分别在什么情况下发生？ +

StackOverflowError：

  • 虚拟机栈中栈帧数量超过限制
  • 典型场景：无终止条件的递归、方法调用链过深
  • 调大 -Xss 可以容纳更多栈帧，但本质应修复代码

OutOfMemoryError（OOM）：

Java heap space堆内存不足（对象过多/过大），或内存泄漏
GC overhead limit exceededGC 耗时超 98% 但回收不到 2% 内存
Metaspace加载的类过多（大量动态代理/反射）
Direct buffer memoryNIO 直接内存超限
Unable to create new native thread操作系统无法创建更多线程（受 ulimit/内存限制）

排查思路：-XX:+HeapDumpOnOutOfMemoryError 生成 dump，用 MAT / VisualVM 分析大对象和 GC Roots 引用链，定位泄漏源。

Q13 Class.forName() 和 ClassLoader.loadClass() 有什么区别？ +

Class.forName(className)：

  • 加载类并执行初始化（执行 static 代码块）
  • 默认使用调用者的类加载器
  • 典型场景：JDBC 驱动注册（Class.forName("com.mysql.jdbc.Driver") 触发驱动的 static 块注册到 DriverManager）

ClassLoader.loadClass(className)：

  • 只执行加载阶段，不触发初始化
  • 使用指定的类加载器实例
  • 典型场景：Spring 延迟加载（Lazy Loading）、框架内部的类扫描（只需要知道类存在，不需要立即初始化）

核心区别：是否触发类的初始化（static 块是否执行）。Class.forName(name, false, classLoader) 可以指定不初始化，效果类似 loadClass。

Q14 什么是强引用、软引用、弱引用、虚引用？各自的使用场景？ +
强引用 普通赋值（Object obj = new Object()）。只要强引用存在，GC 绝不回收，即使 OOM 也不回收。最常见的引用类型。
软引用 SoftReference 内存充足时不回收，即将 OOM 时才回收。适合做内存敏感的缓存（图片缓存、页面缓存）。配合 ReferenceQueue 可感知回收。
弱引用 WeakReference 下次 GC 必定回收（无论内存是否充足）。适合生命周期短、可重建的关联关系。典型应用：ThreadLocalMap 的 key、WeakHashMap。
虚引用 PhantomReference 随时可能回收，get() 永远返回 null。唯一作用：配合 ReferenceQueue 跟踪对象被回收的时机，用于释放堆外资源（如 DirectByteBuffer 的 Cleaner）。

面试延伸：ThreadLocal 内存泄漏的本质——key 是弱引用被 GC 回收后变 null，但 value 是强引用不会被回收，必须手动 remove()。

Q15 线上服务应该配置哪些 JVM 参数？为什么 -Xms 和 -Xmx 要设成一样？ +

生产环境推荐参数：

-Xms = -Xmx避免堆动态扩缩。扩容时需要暂停 JVM 申请内存，缩容时释放内存，都可能引起不必要的停顿。
-XX:MetaspaceSize = -XX:MaxMetaspaceSize同理，避免 Metaspace 动态扩容触发 Full GC。
-XX:+HeapDumpOnOutOfMemoryErrorOOM 时自动 dump，事后用 MAT 分析。配合 -XX:HeapDumpPath 指定路径。
-Xlog:gc*:file=gc.log:time,uptime,level,tagsGC 日志，事后分析 GC 频率、停顿时间、回收效率。生产必开。
-XX:+UseG1GC (或 +UseZGC)选择合适的 GC。G1 适合通用场景，ZGC 适合大堆低延迟。
-XX:MaxGCPauseMillis=200G1 目标停顿时间，让 GC 自适应调整。

为什么 -Xms = -Xmx：如果初始堆小于最大堆，当应用内存增长触发扩容时，JVM 需要向操作系统申请额外内存并可能触发 Full GC 来整理堆空间，导致不可预期的停顿。设成一样后堆大小固定，消除这种不确定性。

经验法则：8GB 堆起步，G1 GC，GC 日志 + HeapDump 全开。调优前先分析 GC 日志，不要凭感觉调参。
