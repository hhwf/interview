JVM 内存与类加载核心知识

JVM 是 Java 程序的运行基石。理解运行时数据区的划分、类从加载到卸载的完整生命周期、以及 JIT 如何将热点代码编译为机器码， 对排查 OOM、调优性能、理解框架原理（如 Tomcat 类加载、SPI 打破委派）都至关重要。

运行时数据区

JVM 运行时数据区定义了程序执行期间的内存划分，分为线程私有和线程共享两大类。

程序计数器 线程私有。记录当前线程执行的字节码行号，用于线程切换后恢复执行位置。是唯一不会发生 OOM 的区域。
虚拟机栈 线程私有。每个方法调用创建一个栈帧，包含局部变量表、操作数栈、动态链接、返回地址。栈深度超过限制抛出 StackOverflowError；动态扩展失败抛出 OOM。
本地方法栈 线程私有。为 Native 方法（C/C++ 实现）服务，结构与虚拟机栈类似。HotSpot 将两者合并。
堆 线程共享。存放所有对象实例和数组，是 GC 的主要区域。分为 Young Generation（Eden + S0 + S1）和 Old Generation。-Xms 设初始大小，-Xmx 设最大大小。
方法区 线程共享。存储已加载类的元信息、常量池、静态变量、JIT 编译后的代码。JDK 7 用 PermGen（-XX:MaxPermSize），JDK 8+ 改为 Metaspace（使用本地内存，-XX:MaxMetaspaceSize 限制）。
直接内存 不属于 JVM 运行时数据区，但被频繁使用。NIO 的 DirectByteBuffer 通过 Unsafe.allocateMemory 在堆外分配，避免 JVM 堆与 Native 堆之间的数据拷贝，适合大缓冲区场景（Netty、Kafka）。

局部变量表以 Slot 为单位（32 bit），long 和 double 占 2 个 Slot。方法参数和局部变量都存放在此表。Slot 复用：一个局部变量超出作用域后，其 Slot 可被后续变量复用，减少内存占用。

堆的 Young Generation（Eden + S0 + S1）和 Old Generation 是 GC 的主战场。 G1 将堆拆分为等大的 Region 并按垃圾密度优先回收；ZGC 用染色指针实现亚毫秒并发标记—— 这些机制的完整原理见独立页面，包含 SATB 算法、RSet 跨区引用、染色指针与读屏障、 以及 G1 vs ZGC 选型建议。

Klass Pointer 与指针压缩

Klass Pointer 指向方法区中该对象的类元信息（Klass 结构）。64 位 JVM 默认开启压缩指针（-XX:+UseCompressedOops），将 8 字节指针压缩为 4 字节，可寻址 32 GB 堆空间（4 字节 × 8 对齐偏移）。压缩后对象头从 16 字节缩减为 12 字节，大幅降低内存开销。

对象大小计算：对象头(12) + 实例字段（对齐到 8 字节倍数）。空 Object 占 16 字节（12 头 + 4 padding）。开启 -XX:+UseCompactObjectHeaders（JDK 24+ Lilliput）可进一步将对象头压缩到 8 字节。

面试要点：Class 文件是平台无关的字节码格式。面试常问：常量池（Constant Pool）的作用——存放字面量和符号引用，类加载时部分解析为直接引用。javap -verbose 可反编译 Class 文件查看字节码，是排查编译器行为（如 try-with-resources、lambda 脱糖）的利器。

关键字段解读

常量池 Class 文件中最大的部分。存放字面量（字符串、final 常量值）和符号引用（类名、方法名、字段名）。运行时解析为直接引用（内存地址）。常量池也是反射、动态代理等机制的底层数据来源。
字段表 描述类中声明的变量。每个字段包含 access_flags、name_index、descriptor_index 和属性集合。字段描述符：I=int, J=long, Ljava/lang/String;=String 等。
方法表 描述类中声明的方法。方法的字节码指令存放在 Code 属性中，包含 max_stack（操作数栈最大深度）、max_locals（局部变量表 Slot 数）、字节码数组和异常表。
访问标志 ACC_PUBLIC(0x0001)、ACC_FINAL(0x0010)、ACC_SUPER(0x0020)、ACC_INTERFACE(0x0200)、ACC_ABSTRACT(0x0400)、ACC_ENUM(0x4000) 等位掩码组合。

实战工具：javap -c -p -v MyClass.class 反编译查看完整结构；ASM / ByteBuddy 可在运行时动态生成或修改 Class 文件（AOP、热部署的底层原理）。

类加载机制

类加载是将 Class 文件的二进制数据读入 JVM，并在方法区创建对应的 Class 对象的过程。完整生命周期包含加载、验证、准备、解析、初始化、使用、卸载 7 个阶段，其中验证、准备、解析统称为连接（Linking）。

面试要点：类加载五阶段：加载→验证→准备→解析→初始化。准备阶段为静态变量分配内存并赋默认值（int=0，引用=null）；初始化阶段执行 （类初始化方法）赋真实值。触发初始化的 6 种场景（主动引用）：new、读写静态字段/方法、反射、初始化子类、启动类、MethodHandle。
加载 通过全限定名获取二进制字节流（可从 Class 文件、JAR、网络、动态生成等来源），将字节流代表的静态存储结构转化为方法区的运行时数据结构，在堆中生成 java.lang.Class 对象作为方法区入口。
验证 确保 Class 文件格式正确、语义合法、不会危害 JVM。包括文件格式验证（魔数、版本号）、元数据验证（继承链、final 约束）、字节码验证（数据流/控制流分析）、符号引用验证（解析阶段的引用是否存在）。可用 -Xverify:none 跳过以加速启动。
准备 为类的静态变量分配内存并设零值。注意：final static 常量在编译时已确定值，准备阶段直接赋值（不取零值）。例：static int a = 123; 准备阶段 a=0，初始化阶段 a=123；static final int b = 123; 准备阶段 b=123。
解析 将常量池中的符号引用（名称字符串）替换为直接引用（内存地址/偏移量）。解析时机因 JVM 实现而异：可在类加载时一次性解析（静态解析），也可在首次使用时延迟解析（动态解析）。
初始化 执行类的 <clinit> 方法（编译器收集 static 语句块和静态变量赋值语句生成）。初始化是类加载的最后一步，至此才真正执行 Java 代码。JVM 保证 <clinit> 在多线程环境下的同步加锁。

主动引用（触发初始化）vs 被动引用（不触发）

主动引用① new 对象 / 读写静态字段 / 调用静态方法 ② 反射调用（Class.forName）③ 初始化子类时先初始化父类 ④ main 方法所在类 ⑤ MethodHandle 解析结果类
被动引用① 通过子类引用父类的静态字段 → 只初始化父类 ② 通过数组定义引用类 → 不触发初始化 ③ 引用类的常量（编译期已存入调用方常量池）→ 不触发初始化

双亲委派模型

双亲委派模型要求：类加载请求先委托父加载器处理，父加载器无法完成时才由自身加载。这不是强制约束，而是一种约定，通过组合复用而非继承实现。

Bootstrap ClassLoader (C++ 实现)
加载 JAVA_HOME/lib（rt.jar、java.lang.* 等核心类库）
加载路径: sun.boot.class.path
↓ parent
Extension ClassLoader (Java 实现)
加载 JAVA_HOME/lib/ext 或 java.ext.dirs
实现: sun.misc.Launcher$ExtClassLoader
↓ parent
Application ClassLoader (Java 实现)
加载用户类路径（classpath / -cp / -classpath）
实现: sun.misc.Launcher$AppClassLoader
↓ parent
自定义 ClassLoader (Java 实现)
重写 findClass() + defineClass()
场景: 热部署、插件隔离、加密 Class 文件加载
面试要点：双亲委派模型的好处：避免同一类被不同类加载器加载（保证 java.lang.Object 唯一）。打破场景：① SPI（JDBC Driver 需 BootstrapClassLoader 加载接口、AppClassLoader 加载实现）→ 线程上下文类加载器；② Tomcat（同一 JVM 隔离不同 WebApp）→ 各 WebAppClassLoader 独立；③ OSGi 模块热部署。JDK 9 模块化后改为模块化路径优先。

为什么需要双亲委派？

安全性 防止核心类被篡改。自定义一个 java.lang.String 不会被加载，因为 Bootstrap 加载器优先加载 JDK 自带的 String，保证 JRE 核心类库的完整性和安全性。
避免重复 父加载器已加载的类不会被子加载器重复加载，保证 Class 对象的全局唯一性（同一个类在 JVM 中只有一个 Class 对象）。

打破双亲委派的典型场景

SPI 机制 JDBC、SLF4J 等使用 SPI（Service Provider Interface）加载第三方实现。核心接口在 Bootstrap 加载，但实现类在 classpath。解决方式：引入线程上下文类加载器（Thread.setContextClassLoader），由父加载器反向委托子加载器加载实现。
Tomcat 每个 WebApp 独立类加载器（WebAppClassLoader），优先加载自己 WEB-INF/classes 和 lib 下的类（打破委派），实现不同应用间的类隔离。共享类（如 Spring）由公共父加载器加载。
OSGi / 热部署 OSGi 使用网状委派模型（非树形），每个 Bundle 有独立类加载器，按规则在 Bundle 间共享类。热部署（如 Spring DevTools）通过丢弃旧 ClassLoader 并创建新的来实现类的重新加载。

自定义 ClassLoader 最佳实践：重写 findClass()（保持双亲委派）而不是 loadClass()（打破委派）。loadClass 默认逻辑是先调 parent.loadClass()，找不到再调 findClass()；如果直接重写 loadClass 就跳过了委派逻辑。

JIT 编译与调优

JVM 执行程序有两种方式：解释执行（逐条翻译字节码）和JIT 编译执行（将热点字节码编译为本地机器码缓存）。JIT 编译器在运行时检测热点代码并编译，可大幅提升性能（通常 10~100 倍）。

C1 / C2 / 分层编译

C1 (Client) 编译速度快，优化程度低。适合简单方法。插入 profiling 数据收集点（分支计数、类型 profile），为 C2 提供优化依据。
C2 (Server) 编译速度慢，激进优化（内联、逃逸分析、标量替换、锁消除、循环展开）。适合热点方法。使用 C1 收集的 profiling 数据做精确优化。
分层编译 JDK 8 默认开启（-XX:+TieredCompilation）。5 个层级：Level 0 解释执行 → Level 1~3 C1 编译（逐步增加 profiling）→ Level 4 C2 编译。方法按调用频率在各层级间晋升。
面试要点：JIT 分层编译（Tiered Compilation，JDK 8+ 默认开启）：C0（解释）→ C1（简单优化）→ C2（激进优化）。常见优化：内联（Inlining）、逃逸分析（对象不逃逸则栈上分配 / 锁消除）、循环展开、分支预测。JIT 预热：启动初期大量 C0 解释执行，性能差；生产中可提前加载热方法。-XX:+PrintCompilation 可观察编译情况。
面试要点：JIT 分层编译（Tiered Compilation，JDK 8+ 默认开启）：C0（解释）→ C1（简单优化）→ C2（激进优化）。常见优化：内联（Inlining）、逃逸分析（对象不逃逸则栈上分配 / 锁消除）、循环展开、分支预测。JIT 预热：启动初期大量 C0 解释执行，性能差；生产中可提前加载热方法。-XX:+PrintCompilation 可观察编译情况。

热点代码检测

方法调用计数器 统计方法被调用次数。超过阈值（半衰期机制，默认 -XX:CompileThreshold=10000）触发 JIT 编译。长期未调用的方法计数器会衰减归零（半衰期由 -XX:+UseCounterDecay 控制）。
回边计数器 统计循环中的回边（向后跳转）次数。用于触发循环体的 OSR（On-Stack Replacement），在循环执行中途将解释执行替换为编译后的机器码。

关键 JIT 优化技术

方法内联 将方法体直接嵌入调用点，消除方法调用开销（栈帧创建/销毁、参数传递）。是最重要的优化手段。-XX:MaxInlineLevel=9 控制内联深度。final 方法更容易被内联（但现在非 final 方法也可以，通过 CHA 分析）。
逃逸分析 分析对象的引用是否逃逸出方法。未逃逸的对象可优化为：栈上分配（不用堆，方法结束自动回收）、标量替换（拆成基本类型，直接存寄存器）、锁消除（不存在竞争，移除 synchronized）。
去优化 JIT 编译时基于激进假设（如类型固定为某子类），运行时若假设被打破（加载了新子类），会丢弃编译结果回退到解释执行，称为去优化（Deoptimization）。频繁去优化会严重影响性能。

常用 JVM 调优参数

-Xms / -Xmx堆初始大小 / 最大大小。生产环境通常设置为相同值，避免堆动态扩缩引起 GC 停顿。
-Xmn新生代大小。通常设为堆的 1/3 ~ 1/4。
-Xss线程栈大小（默认 1MB）。StackOverflowError 时可适当调大。
-XX:MetaspaceSizeMetaspace 初始大小。设大避免频繁 Full GC 扩容。
-XX:+UseG1GC使用 G1 垃圾收集器（JDK 9+ 默认）。
-XX:+UseZGC使用 ZGC 垃圾收集器（JDK 15+），适合大堆低延迟场景。
-XX:MaxGCPauseMillisG1 目标最大 GC 停顿时间（默认 200ms）。
-Xlog:gc*JDK 9+ 统一日志框架输出 GC 日志（替代 -XX:+PrintGCDetails）。生产必开，用于事后分析。
-XX:+HeapDumpOnOOMOOM 时自动 dump 堆内存快照，配合 MAT 分析定位泄漏对象。
