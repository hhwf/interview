MQ 高频面试题

Q1 为什么使用 MQ？引入 MQ 有哪些缺点需要应对？ +
优点三点：
  • 解耦：服务 A 只发消息，B / C / D 各自订阅消费，互不依赖。新增消费方无需修改 A，B 故障不影响 A 主链路。
  • 异步：主流程发完消息即返回，耗时操作（发短信 / 发邮件 / 更新积分）异步处理，接口响应时间从数百 ms 降至个位数 ms。
  • 削峰：秒杀 / 大促洪峰流量先写入 MQ 缓冲，消费者按自身速率平稳消费，保护下游数据库。
缺点三点：
  • 可用性降低：MQ 挂掉整条链路受影响，需高可用集群部署。
  • 复杂度增加：消息丢失、重复消费、消息顺序都需单独处理，开发测试成本上升。
  • 一致性挑战：消费失败如何补偿、如何保证最终一致，需要完善的幂等和重试机制。
Q2 Kafka 为什么吞吐量能达到百万级？ +
五个关键设计共同作用：
  • 顺序写磁盘：消息追加写 .log 文件末尾，磁盘顺序写吞吐 ~600 MB/s，远高于随机写 ~100 MB/s，避免磁头寻道。
  • PageCache：写入 OS 页缓存即返回，OS 异步刷盘。Consumer 紧跟时直接从 PageCache 读，几乎零磁盘 IO。
  • sendfile 零拷贝：FileChannel.transferTo() 调用内核 sendfile，数据路径磁盘→内核缓冲区→网卡 DMA，仅 2 次拷贝 + 2 次上下文切换（传统 4 + 4 次）。
  • 批量压缩：Producer 聚合多条消息为 RecordBatch，支持 snappy / lz4 / zstd 压缩，大幅减少网络 IO 和存储占用。
  • 多 Partition 并发：Topic 拆分多个 Partition，多 Producer / Consumer 并行读写，横向扩展吞吐能力。
Q3 Kafka 如何保证消息不丢失（三段分析）？ +
  • 生产者端：acks=-1 + min.insync.replicas=2，确保至少 2 个 ISR 副本写入才返回成功；开启 retries 重试；使用幂等 Producer（enable.idempotence=true）防止重试导致重复。
  • Broker 端：replication.factor≥2 多副本保障；关闭 unclean.leader.election.enable=false，防止落后副本成为 Leader 后覆盖未同步数据。
  • 消费者端：关闭自动提交 enable.auto.commit=false，消费逻辑全部执行成功后再手动调用 commitSync()，避免消费失败但 offset 已推进。
答题技巧面试时按「生产→存储→消费」三段逐一展开，思路清晰有加分。
Q4 如何保证消息的顺序性？ +
核心思路：同一业务 Key 路由到同一 Partition，并由同一 Consumer 串行消费。
  • Producer 端：发送时指定 key = orderId，Kafka 按 hash(key) % numPartitions 确保相同 key 进同一 Partition；RocketMQ 通过 MessageQueueSelector 选固定 Queue。
  • Consumer 端：Consumer 实例数 ≤ Partition 数，每个 Partition 只分配一个 Consumer；Consumer 内部串行处理，不用线程池并发消费。
  • 注意：Rebalance 期间顺序可能短暂中断；发送失败重试可能乱序，需开启幂等 Producer 配合。
Q5 如何解决消息重复消费（幂等）？ +
MQ 语义是 at-least-once，重复消费不可避免，需在消费端实现幂等：
  • Redis SETNX：消费前 SET msgId EX 86400 NX，成功才处理；已存在则幂等返回。
  • DB 唯一约束：幂等表加联合唯一索引（business_key），INSERT 触发唯一冲突则 catch 忽略，简单可靠。
  • 业务状态机：消费前检查业务状态，只有目标前置状态才执行（如订单已支付则跳过支付消息）。
  • Kafka 幂等 Producer：enable.idempotence=true，用 PID + SequenceNumber 在 Broker 去重，解决生产者重试导致的重复写入。
Q6 消息积压了怎么处理？ +
分短期应急和长期根治两步走：
  • 扩容 Consumer：增加 Consumer 实例数（注意不能超过 Partition 数，否则多余实例空闲）；必要时临时增加 Partition 数再对应扩 Consumer。
  • 提升批量：调大 max.poll.records，每次 poll 更多消息批量处理，提高单实例消费速率。
  • 降级丢弃：对日志 / 埋点等非核心消息采样或直接丢弃，优先保证核心业务消息消费完。
  • 长期根治：排查消费端性能瓶颈（慢 SQL / 外部调用超时），优化消费逻辑，从根本上使消费速率 ≥ 生产速率。
加分点主动说明「Consumer 实例数 ≤ Partition 数」这个约束，很多候选人会忽略。
Q7 Kafka 的 ISR 是什么？HW 和 LEO 的区别？ +
  • AR（Assigned Replicas）：分区的全部副本集合 = ISR + OSR
  • ISR（In-Sync Replicas）：与 Leader 同步延迟在 replica.lag.time.max.ms（默认 30s）内的副本集合。
    Follower 落后超过阈值 → 踢出 ISR 进入 OSR；追上 Leader 后重新加入 ISR（动态收缩与扩张）
  • LEO（Log End Offset）：每个副本已写入的最大消息 offset + 1，即「下一条消息应写入的位置」，各副本独立维护。
  • HW（High Watermark）：ISR 中所有副本 LEO 的最小值。Consumer 只能消费 offset < HW 的消息——HW 是对外可见的「安全水位」，确保只有 ISR 全员确认的数据才对外可见。
  • 为什么需要 HW：防止数据回退。若 Follower 未同步时 Leader 宕机，新 Leader 可能缺少旧 Leader 已暴露给 Consumer 的消息，造成数据不一致。

Leader Election：新 Leader 必须从 ISR 中选出（由 Controller 决定）。unclean.leader.election.enable=false（默认）禁止 OSR 成为 Leader，确保不丢数据；设为 true 时 OSR 可被选为 Leader，可用性更高但可能丢失未同步的消息。

Leader Epoch（Kafka 2.0+）：HW 机制存在边缘 Bug——Follower 以 HW 为基准截断日志后重启，若 HW 未及时更新可能错误截断正确数据。Leader Epoch 用单调递增的纪元号替代 HW 截断，彻底解决该问题。

答题技巧按 ISR → LEO → HW 的顺序层层递进，最后主动补充 Leader Election 与 Leader Epoch，展示对 Kafka 可靠性机制的深度理解。
Q8 Kafka 的 acks 参数如何选择？ +
  • acks=0：不等待 Broker 确认，即发即忘。吞吐最高，但消息可能丢失。适用：日志收集、埋点上报等允许少量丢失的场景。
  • acks=1：等待 Leader 写入确认，不等 Follower 同步。Leader 宕机且 Follower 未同步时可能丢失。适用：一般业务消息，吞吐与可靠性折中。
  • acks=-1（all）：等待 ISR 全部副本写入后才确认。可靠性最高，配合 min.insync.replicas=2 防止 ISR 退化为单副本时失去意义。适用：订单、支付、金融等不允许丢失的场景。
Q9 Kafka Consumer Group 是什么？Rebalance 何时触发？ +
Consumer Group：同组内每个 Partition 只被一个 Consumer 消费（组内并行，互不重复）；不同 Group 各自独立消费全量消息（多订阅方互不干扰）。

Rebalance 触发时机：
  • Consumer 实例加入或离开（包括消费超时被踢出）
  • 订阅 Topic 的 Partition 数量变化
  • Consumer 心跳超时（超过 session.timeout.ms）
Rebalance 的问题：期间所有 Consumer 暂停消费（Stop The World），高并发下造成明显延迟。
优化：调大 session.timeout.ms 和 max.poll.interval.ms；Kafka 2.3+ 支持 Static Membership 减少不必要的 Rebalance。
Q10 RocketMQ 事务消息的原理是什么？ +
通过两阶段提交 + 事务回查，实现本地事务与消息投递的最终一致性：
  • ① Producer 发送半消息（Half Message）到 Broker，此时对 Consumer 不可见。
  • ② Broker 持久化半消息并返回 ACK。
  • ③ Producer 执行本地事务（如扣款、写订单）。
  • ④ 事务成功 → 发送 Commit，Broker 将消息标记为可见；失败 → 发送 Rollback，Broker 删除半消息。
  • ⑤ 若 Producer 宕机未发 Commit/Rollback，Broker 定时回查（checkLocalTransaction）本地事务状态，最多 15 次。
核心价值保证「本地事务成功 ↔ 消息对消费者可见」的最终一致性，解决分布式事务中消息与本地操作的原子性问题。
Q11 死信队列是什么？如何处理死信消息？ +
死信队列（DLQ）：消息消费失败超过最大重试次数后，自动路由到死信队列，不再重试，等待人工介入。
  • RocketMQ：自动创建 %DLQ%{ConsumerGroup} Topic；默认重试 16 次，每次间隔递增，最长 2 小时。
  • Kafka：无原生 DLQ，需业务代码 catch 消费异常后写入专用 .dead Topic。
处理策略：
  • 订阅 DLQ，消息进入即触发监控告警（钉钉 / Prometheus）。
  • 人工分析原因：消费代码 Bug、上游数据异常、依赖服务故障。
  • 修复后从 DLQ 重新投递到原 Topic 进行重放，Consumer 幂等处理完成补偿。
Q12 延迟消息如何实现？RocketMQ 和 Kafka 有什么区别？ +
  • RocketMQ（原生支持）：发送时调用 setDelayTimeLevel(N)，支持 18 个固定级别（1s / 5s / 10s / 30s / 1min … 2h）。内部将消息先写入 SCHEDULE_TOPIC_XXXX，定时线程每秒扫描，到期后投递到真实 Topic。
  • Kafka（不原生支持）：需业务层实现，常见方案：① 时间轮算法维护延迟队列；② 写入延迟 Topic + Consumer 定时轮询；③ 直接用 RocketMQ 处理延迟消息场景。
  • RabbitMQ：通过 x-message-ttl 设置消息过期时间，过期后路由到死信队列，DLQ 的 Consumer 即为延迟消费方。缺点：队头阻塞会影响后续消息的延迟精度。
Q13 Kafka 和 RocketMQ 存储架构的核心区别？ +
  • Kafka：每个 Partition 对应独立的 .log 文件。Partition 数量少时性能极致（纯顺序写）；但 Partition 增多时，多个文件交替写入会退化为随机写，吞吐明显下降。
  • RocketMQ：所有 Topic 消息统一追加写入 CommitLog（单文件，默认 1GB 分片），始终是顺序写。ConsumeQueue 是轻量级消费索引（每条固定 20 字节），消费时先查索引再按 offset 读 CommitLog。
  • 结论：RocketMQ 在 Topic / Queue 数量多时写入性能更稳定；Kafka 在少量 Partition 场景性能更极致，Topic 规模大时性能会有所下降。
Q14 如何设计一个可靠的消息投递系统（本地消息表方案）？ +
本地消息表是解决「分布式事务中消息与本地事务一致性」的经典方案：
  • ① 在同一个本地事务中执行业务操作（如扣款）+ 向本地消息表插入待发送记录（状态 = PENDING），两者原子提交。
  • ② 本地事务提交后，发送消息到 MQ，成功后将消息表记录标记为 SENT。
  • ③ 定时任务扫描消息表中超过 N 分钟仍为 PENDING 的记录进行重发，直到发送成功。
  • ④ 消费端做幂等处理，确保重复消息不产生副作用。
保证本地事务成功 → 消息最终一定发出（最终一致）；事务失败 → 消息表记录不存在，消息绝不发出。
缺点依赖本地 DB，消息表会成为写入热点；高频场景可用 RocketMQ 事务消息替代。
Q15 Kafka 的零拷贝（Zero-Copy）是如何实现的？ +
传统数据传输（4 次拷贝，4 次上下文切换）：
  • 磁盘 →（DMA）→ 内核缓冲区 →（CPU）→ 用户缓冲区 →（CPU）→ Socket 缓冲区 →（DMA）→ 网卡
sendfile 零拷贝（Linux 2.1+，2 次拷贝，2 次上下文切换）：
  • 磁盘 →（DMA）→ 内核缓冲区 →（CPU）→ Socket 缓冲区 →（DMA）→ 网卡
  • 完全跳过用户空间，减少 2 次数据拷贝和 2 次用户 / 内核态切换。
sendfile + DMA gather（Linux 2.4+，1 次拷贝）：
  • 磁盘 →（DMA）→ 内核缓冲区，网卡 DMA 直接从内核缓冲区读取，仅 1 次真正的数据拷贝。
Kafka 通过 Java 的 FileChannel.transferTo() 调用操作系统 sendfile，实现消息从磁盘到网卡的高效传输，是 Kafka 高吞吐的关键之一。
⚡ 实战 实战：消息积压了几百万条，怎么紧急处理？ +
① 确认积压原因（先诊断，再动手）

积压的本质是消费速度 < 生产速度，常见原因有三类：

  • 消费者挂了：消费组实例全部宕机或 rebalance 卡死，consumer lag 持续上涨但 consumer-fetch-rate 为 0。排查方式：kafka-consumer-groups.sh --bootstrap-server broker:9092 --describe --group my-group，查看每个 partition 的 LAG 列和 OWNER 列。
  • 消费速度跟不上：下游数据库 / 远程调用变慢，单条消息处理耗时从 5 ms 涨到 200 ms。可通过消费者日志、APM（SkyWalking / Arthas trace）定位瓶颈。
  • 网络 / broker 问题：消费者与 broker 之间网络抖动导致 fetch 超时频繁，日志中出现大量 FetchSessionHandler 错误。
② 紧急扩容消费者实例

最快的止血手段是增加消费者实例数，但消费者数不能超过 topic 的 partition 数——多余的实例只会处于 idle 状态。例如 topic 有 12 个 partition，最多部署 12 个实例并行消费：

# 快速查看 partition 数
kafka-topics.sh --bootstrap-server broker:9092 --describe --topic order-events
# 输出: PartitionCount: 12

# K8s 环境一键扩容
kubectl scale deployment order-consumer --replicas=12

如果 partition 数本身就少（比如只有 3 个），临时扩 partition 需要谨慎：新 partition 不会迁移旧数据，对已积压的消息无帮助，只对增量消息有效。

③ 临时方案：大 partition topic 转发

当 partition 数不够、又不能改原 topic 时，可以新建一个 partition 数更多的临时 topic，用少量消费者把积压消息快速转发过去，再用大量消费者并行消费新 topic：

# 1. 创建临时 topic，partition 数设为 64
kafka-topics.sh --create --bootstrap-server broker:9092 \
  --topic order-events-temp --partitions 64 --replication-factor 3

# 2. 临时转发消费者（伪代码）
while (msg = consumer.poll()) {
    producer.send("order-events-temp", msg.key(), msg.value());
    // 转发只做搬运，不做业务逻辑
}

这样可以用 64 个消费者实例并行处理，消化速度提升数倍。积压清零后下线临时 topic 和转发程序即可。

④ 消费端优化（提速）
  • 批量处理：将单条消费改为批量拉取（max.poll.records=500），批量写入数据库（JDBC batch insert），吞吐量可提升 5-10 倍。
  • 异步化：消费线程拿到消息后丢进本地线程池异步处理，主线程立即 commit offset 并拉取下一批。注意控制队列深度，防止 OOM。
  • 跳过非关键逻辑：紧急期间可临时关闭日志打印、指标上报、非核心缓存刷新等旁路操作，只保留核心业务写入。
⑤ 事后复盘 & 长效防护
  • 加监控：对 consumer lag 设置告警阈值（如 lag > 10000 持续 2 分钟触发 P1 告警），常用方案有 Burrow、Prometheus + kafka_exporter + Grafana。
  • 设置合理的 retention.ms（如 72 小时）和 retention.bytes，避免消息无限堆积导致磁盘打满。
  • 压测 & 容量规划：定期压测消费端极限吞吐量，确保峰值时消费速度至少是生产速度的 1.5 倍以上，留足安全余量。
⚡ 实战 实战：消费者处理消息时抛异常了，消息会丢吗？怎么处理？ +
① Kafka：取决于 offset 提交方式

这是面试中最容易踩坑的点。Kafka 本身不负责重试，消息丢不丢完全看 offset 有没有被提交：

  • 自动提交（enable.auto.commit=true）：消费者 poll 后会在后台线程按 auto.commit.interval.ms（默认 5 s）自动提交 offset。如果消息刚被 poll 进来、offset 已经提交、但业务处理抛了异常——这条消息就永久丢失了，因为下次 rebalance 或重启后会从已提交的 offset 之后开始消费。
  • 手动提交（enable.auto.commit=false）：业务处理成功后才调用 consumer.commitSync()。如果处理抛异常，offset 不提交，重启或 rebalance 后会从上次提交的 offset 重新拉取，消息不会丢，但可能重复消费。
// 推荐写法：手动提交 + try-catch
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        try {
            processMessage(record);              // 业务处理
            consumer.commitSync(                 // 成功才提交
                Map.of(new TopicPartition(record.topic(), record.partition()),
                       new OffsetAndMetadata(record.offset() + 1)));
        } catch (Exception e) {
            log.error("处理失败, offset={}", record.offset(), e);
            // 不提交 offset → 下次会重试；可加入重试计数器
        }
    }
}
② RocketMQ：内置重试机制

RocketMQ 对消费失败有开箱即用的重试机制，比 Kafka 友好得多：

  • PushConsumer 消费失败（返回 RECONSUME_LATER 或抛异常），broker 会自动将消息放入重试队列（%RETRY%{ConsumerGroup}），按梯度间隔重新投递：10s、30s、1min、2min、3min … 最多重试 16 次。
  • 超过最大重试次数后，消息进入死信队列（%DLQ%{ConsumerGroup}），不再自动投递。
  • 可通过 consumer.setMaxReconsumeTimes(5) 自定义最大重试次数；通过 setDelayLevelWhenNextConsume(3) 指定重试延迟级别。
// RocketMQ 消费示例
consumer.registerMessageListener((MessageListenerConcurrently) (msgs, context) -> {
    for (MessageExt msg : msgs) {
        try {
            processOrder(msg.getBody());
        } catch (BusinessException e) {
            // 业务异常：可重试，有补偿意义
            return ConsumeConcurrentlyStatus.RECONSUME_LATER;
        } catch (IllegalArgumentException e) {
            // 数据格式错误：重试也不会成功，记录日志后直接 ACK
            log.error("消息格式异常, msgId={}", msg.getMsgId(), e);
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;  // 跳过
        }
    }
    return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
});
③ 最佳实践总结
  • 关闭自动提交：Kafka 场景下务必使用手动提交 offset，确保"先处理、后提交"的语义。
  • 异常分级处理：业务异常（如下游超时、库存不足）→ 重试；系统异常 / 数据异常（如 JSON 解析失败、字段缺失）→ 记录并跳过，避免无限重试。
  • 死信队列兜底：为每个消费组创建对应的死信队列消费者或告警监控，定期人工检查死信消息，进行补偿或修复后重新投递。
  • 合理设置重试上限：重试次数过多会阻塞队列（尤其是 RocketMQ 的重试队列是串行消费的），建议根据业务 SLA 设定 3-5 次重试 + 指数退避。
④ 幂等性保障（重试的前提）

既然消息会被重复消费，幂等处理是必须的，常见方案：

  • 去重表：在数据库中创建 msg_dedup 表，以 msg_id 为唯一索引，消费前先 insert，insert 成功才执行业务，重复 insert 会抛唯一键冲突直接跳过。
  • Redis SETNX：以 msg_id 为 key 做 SETNX，设置过期时间略大于消息最大重试周期，O(1) 判断是否已处理。
  • 状态机：利用业务状态（如订单状态从 CREATED → PAID）做前置校验，重复消费时状态已经是 PAID 则直接返回成功。
  • 数据库乐观锁：update 语句带版本号 WHERE version = ?，重复更新时 version 不匹配则影响行数为 0，视为已处理。

总结一句话：先保证不丢（手动提交），再保证不重（幂等处理），最后兜底（死信队列 + 告警）。
