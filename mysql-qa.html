MySQL 高频面试题

Q1 一条 SELECT 语句的完整执行流程是什么？ +
  • 客户端建立 TCP 连接，连接器验证用户名密码，分配线程
  • 解析器：词法 + 语法分析，生成解析树；预处理器检查表/列是否存在，验证权限
  • 优化器：基于代价模型选择索引，确定 JOIN 顺序，生成执行计划
  • 执行器：按计划调用存储引擎接口逐行获取数据，返回结果集
注意：MySQL 8.0 已移除查询缓存（命中率低，缓存失效成本高）。
Q2 count(*)、count(1)、count(id) 哪个更快？ +
InnoDB 中 count(*) 和 count(1) 完全等价，优化器处理方式相同，都走最小二级索引扫描，无需读取行数据。

count(id) 需读取主键值并判断非 NULL，稍慢。

MyISAM 专门维护行数计数，count(*) 无 WHERE 时 O(1) 直接返回。
Q3 什么是幻读？InnoDB 如何解决？ +
幻读：同一事务中，两次相同范围查询返回的行数不同（另一个事务在中间插入了新行）。

InnoDB 在 RR 级别通过两种方式解决：

  • 快照读（普通 SELECT）：MVCC ReadView 在事务开始时固定，看不到后续插入的行
  • 当前读（SELECT FOR UPDATE / INSERT / UPDATE）：通过间隙锁（Gap Lock）锁住范围，阻止其他事务在间隙中插入
Q4 redo log 和 binlog 的区别？为什么需要两阶段提交？ +
redo log：InnoDB 引擎层，物理日志（记录页面改动），循环覆盖写，用于崩溃恢复（WAL 机制）。
binlog：Server 层，逻辑日志（SQL 或行变更），追加写不覆盖，用于主从复制和时间点恢复。

两阶段提交的必要性：若不用 2PC，可能出现 redo log 写了但 binlog 没写（或反之），导致主库和从库数据不一致，时间点恢复也无法对齐。2PC 通过 prepare/commit 两个状态确保两者的原子性。
Q5 int(10) 和 int(11) 有区别吗？VARCHAR(100) 和 VARCHAR(10) 性能一样吗？ +
int(10) vs int(11)：存储空间完全相同，都是 4 字节。括号内是显示宽度，仅配合 ZEROFILL 填充零，对存储范围和性能无任何影响。MySQL 8.0.17 后已废弃该语法。

VARCHAR(100) vs VARCHAR(10)：磁盘存储无差异，都只占实际数据长度 + 1~2 字节。但内存中排序和创建临时表时，MySQL 按最大声明长度分配内存，VARCHAR(100) 消耗内存是 VARCHAR(10) 的 10 倍，影响排序和临时表性能。
Q6 为什么不建议用 UUID 作主键？ +
InnoDB 聚簇索引按主键值有序存储数据页。UUID 是随机值，插入时目标页随机分散：
  • 目标页不在 Buffer Pool 中 → 频繁磁盘 IO
  • 插入位置随机 → 页分裂和碎片化严重，写性能差
  • UUID 16 字节，所有二级索引叶节点都存主键，空间占用大
推荐使用自增主键（顺序写，无页分裂）或雪花算法 ID（趋势递增，适合分布式）。
Q7 优化器什么时候会放弃索引选择全表扫描？ +
优化器基于代价模型（rows × cost_per_row），当预估索引扫描代价高于全表扫描时选择全表扫，通常出现在：
  • 索引返回行数估算超过总行数 20~30%（如区分度低的字段）
  • 统计信息过期（可用 ANALYZE TABLE 更新）
  • 查询条件导致索引无法有效过滤
可用 FORCE INDEX 强制走索引，但根本方法是优化查询条件或重新设计索引。
Q8 联合索引的最左匹配原则是什么？哪些情况会导致索引失效？ +
联合索引按从左到右的顺序构建 B+ Tree，查询条件必须从索引最左列开始连续匹配才能走完整联合索引。

常见索引失效场景：
  • 跳过最左列：WHERE b=1（index(a,b,c)）→ 走不到联合索引
  • 范围条件右侧截断：WHERE a=1 AND b>2 AND c=3，c 列不走索引
  • 对列做函数/运算：WHERE YEAR(create_time)=2024，函数导致索引失效（8.0.13+ 可建函数索引）
  • 隐式类型转换：varchar 字段用整数比较，MySQL 内部转换导致全表扫描
  • LIKE 前缀通配符：LIKE '%abc' 不走索引；LIKE 'abc%' 可走
  • NOT IN / != / IS NULL：通常走全表扫描（行数多时优化器放弃索引）
本质联合索引是多列拼接后的有序结构，跳过首列后顺序性消失，B+ Tree 无法二分定位。
Q9 什么是覆盖索引？为什么能避免回表，性能提升有多大？ +
回表：命中二级索引后，叶节点只存主键值，若 SELECT 的列不在该索引中，引擎必须拿主键再去聚簇索引查一次完整行——这额外一次随机 IO 就是回表。

覆盖索引：查询所需的所有列（SELECT + WHERE + ORDER BY）都包含在同一索引中，无需回表，直接从索引叶节点返回结果。EXPLAIN 中 Extra: Using index 表示命中覆盖索引。

性能差距：高并发场景可提升数倍到数十倍，尤其在机械盘/远程存储中，覆盖索引把随机 IO（聚簇索引随机页）变为顺序 IO（扫索引页）。

设计建议：高频查询按 (WHERE列, SELECT列) 建联合索引，或用 SELECT 主键, 关键列 代替 SELECT *。
Q10 MVCC 的实现原理是什么？ReadView 什么时候创建？RC 和 RR 的区别体现在哪？ +
核心结构：
  • 隐藏字段：每行记录有 DB_TRX_ID（最近修改的事务 ID）、DB_ROLL_PTR（指向 undo log 的回滚指针）
  • undo log 版本链：每次修改通过 roll_ptr 串成历史版本链，旧版本在 undo log 中保留
  • ReadView：快照读时创建，记录 4 个关键字段：活跃事务列表 m_ids、最小活跃事务 ID min_trx_id、下一个待分配 ID max_trx_id、当前事务 ID creator_trx_id
可见性判断：沿版本链找第一个可见版本：DB_TRX_ID < min_trx_id 已提交可见；在 m_ids 中则活跃不可见；≥ max_trx_id 是未来事务不可见。

RC vs RR 的本质区别：
  • RC：每次快照读都重新生成 ReadView → 能读到最新已提交 → 不可重复读
  • RR：同一事务内首次快照读生成 ReadView 后复用 → 读视图固定 → 可重复读
Q11 MySQL 死锁是如何产生的？InnoDB 如何检测死锁？出现死锁如何排查和预防？ +
成因：两个或多个事务互相持有对方所需的锁，形成循环等待。典型场景：事务 A 锁行 1 再锁行 2，事务 B 反序锁行 2 再锁行 1。

InnoDB 检测：维护等待图（Wait-for Graph），后台线程持续检测循环，发现死锁后回滚代价较小的事务（undo log 量少的），另一个继续执行。

排查方法：
  • SHOW ENGINE INNODB STATUS\G → LATEST DETECTED DEADLOCK 段，显示涉及的 SQL 和锁类型
  • 开启 innodb_print_all_deadlocks=ON，将死锁写入 error log，便于历史分析
预防手段：
  • 多个事务按相同顺序获取锁（消除循环等待）
  • 事务尽量短小，减少锁持有时间
  • 批量更新按主键排序，避免乱序加锁
  • 合理使用索引，避免间隙锁范围过大引入额外锁竞争
Q12 如何排查慢查询？EXPLAIN 输出中哪些字段最重要？ +
排查步骤：
  1. 开启慢查询日志：slow_query_log=ON，long_query_time=1
  2. 用 pt-query-digest 或 mysqldumpslow 聚合，找出频率最高/耗时最长的 SQL
  3. 对目标 SQL 执行 EXPLAIN 分析执行计划
EXPLAIN 关键字段：
  • type：访问类型，从优到差：system > const > eq_ref > ref > range > index > ALL。生产环境出现 ALL 需告警
  • key：实际使用的索引，NULL 表示未走索引
  • rows：预估扫描行数，越小越好
  • Extra：Using index（覆盖索引，好）；Using filesort（额外排序，需优化）；Using temporary（临时表，需优化）
  • filtered：条件过滤后剩余行比例，结合 rows 评估实际访问量
Q13 大表（千万行级）如何安全地加字段或加索引？Online DDL 的原理是什么？ +
直接 ALTER 的风险：锁全表阻塞所有读写，耗时与表大小成正比，生产环境几乎不可接受。

MySQL Online DDL（5.6+）：
  • INSTANT 算法（8.0.12+）：仅修改数据字典，不触碰数据页，加字段毫秒级完成，推荐优先使用
  • INPLACE 算法：在原表上操作，扫描全表构建索引，期间用 ROW_LOG 记录增量，最后短暂 MDL X 锁完成替换。允许并发 DML
  • COPY 算法：创建新表 → 全量复制 → 原子替换，期间阻止写操作，耗时最长
pt-online-schema-change（pt-osc）：创建影子表 → 触发器同步增量 → rename 原子替换。适合旧版 MySQL 或需更精细控制的场景。

生产实践：优先用 MySQL 8.0 INSTANT DDL；不支持时用 pt-osc 或 gh-ost，在业务低峰期执行，监控主从延迟和锁等待。
Q14 主从复制延迟的常见原因有哪些？如何监控和缓解？ +
复制流程：Master 写 binlog → IO 线程传到 Slave Relay Log → SQL 线程重放。

延迟原因：
  • 单线程重放（5.6 前）：Master 并发写，Slave SQL 线程串行重放，高 QPS 时严重落后
  • 大事务：单条 SQL 更新大量行，binlog 是一个大事务，重放耗时长
  • 从库硬件弱：磁盘 IO、CPU 性能不如主库
  • 网络抖动：主从之间带宽不足或延迟大
监控：SHOW REPLICA STATUS\G 中 Seconds_Behind_Source；更精确：用 pt-heartbeat 在主库写时间戳，从库计算真实延迟。

缓解手段：
  • MySQL 5.7+ 开启并行复制（replica_parallel_type=LOGICAL_CLOCK）
  • 拆分大事务，单次操作不超过万行
  • 对延迟敏感的读请求路由到主库（结合业务一致性要求判断）
Q15 InnoDB 的 change buffer 是什么？什么场景有效，什么场景无效？ +
作用：当写操作（INSERT/UPDATE/DELETE）的目标二级索引页不在 Buffer Pool 中时，InnoDB 不立即从磁盘读入页面，而是将变更记录在 change buffer 中，之后批量合并写入，把随机 IO 转为顺序 IO，提升写性能。

Merge 时机：查询该索引页时（读页的同时顺带 merge）、后台定时线程、数据库正常/异常关闭。

有效场景：写多读少（流水日志类）、索引选择性高（每次写的页不同，Buffer Pool 命中率低）

无效场景：
  • 唯一索引：写时必须读页验证唯一性，无法缓冲。change buffer 对唯一索引完全不起作用
  • 写后立即读：刚写的数据马上被查询，change buffer 未 merge，反而多一次 IO
实践意义这是"能用普通索引就不用唯一索引"的性能原因之一——唯一索引禁用了 change buffer 优化。
⚡ 实战 线上出现死锁，你的排查思路是什么？ +

第一步：确认死锁并获取详细信息。登录 MySQL 执行 SHOW ENGINE INNODB STATUS\G，找到 LATEST DETECTED DEADLOCK 段。这里会完整记录两个（或多个）事务各自持有什么锁、在等待什么锁，以及 InnoDB 选择回滚了哪个事务。重点看：

  • TRANSACTION 1 持有哪些行锁（如 X lock on record），在等待哪个资源
  • TRANSACTION 2 的持锁和等待情况
  • 等待图（Wait-for Graph）：如果两个事务互相持有对方需要的锁，就构成环形等待，InnoDB 会选择代价最小的事务进行回滚
-- 快速查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
-- 查看当前活跃事务
SELECT * FROM information_schema.INNODB_TRX WHERE trx_state = 'LOCK WAIT';

第二步：定位应用层代码。通过 INNODB STATUS 中记录的 SQL 语句，结合业务日志确定是哪些业务操作触发了死锁。如果是 Java 应用，可以使用 jstack <pid> 导出线程堆栈，搜索 DeadlockLoserDataAccessException 或 SQLTransientException 关键字，定位到具体的 Service 方法和 SQL 调用链。

第三步：代码层面修复。死锁的根因是多个事务以不同顺序竞争相同资源，修复策略包括：

  • 统一加锁顺序：比如操作多张表时，所有业务路径都按相同的表顺序加锁；操作多行时按主键排序后再更新
  • 缩短事务：将大事务拆成多个小事务，减少锁持有时间。避免在事务中做 RPC 调用或 IO 操作
  • 使用 tryLock 超时机制：在应用层对关键操作设置超时重试，如 Spring 的 @Retryable 配合退避策略
  • 降低隔离级别：如果业务允许，从 REPEATABLE READ 降到 READ COMMITTED，减少间隙锁（Gap Lock）冲突

第四步：持续监控。设置 innodb_print_all_deadlocks=ON 将所有死锁信息记录到 error log，配合告警系统（如 Prometheus + mysqld_exporter）监控 innodb_deadlocks 指标，及时发现趋势。

⚡ 实战 一条慢 SQL 告警了，怎么定位和优化？ +

第一步：从慢查询日志定位目标 SQL。确保线上已开启慢查询日志：

-- 查看当前配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 动态开启（无需重启）
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;       -- 超过 1 秒即记录
SET GLOBAL log_queries_not_using_indexes = ON;  -- 未走索引的也记录

拿到慢查询日志后，用 pt-query-digest slow.log 或 mysqldumpslow -s t slow.log 聚合分析，按总耗时、执行次数、单次平均耗时排序，找出最需要优化的 SQL。

第二步：EXPLAIN 实战解读。对目标 SQL 执行 EXPLAIN 或 EXPLAIN ANALYZE（MySQL 8.0.18+），重点关注以下字段：

  • type：访问类型。生产环境至少要达到 range 级别，如果出现 index（全索引扫描）或 ALL（全表扫描）则需优化
  • key：实际使用的索引。NULL 表示未走索引，需要检查是否缺少合适的索引，或者发生了隐式类型转换（如 varchar 列用数字查询）导致索引失效
  • rows：预估扫描行数。结合 filtered 百分比可以估算实际返回行数，如果 rows 很大而 filtered 很小，说明索引区分度不够
  • Extra：出现 Using filesort 说明需要额外排序（考虑加 ORDER BY 字段的联合索引）；Using temporary 说明使用了临时表（常见于 GROUP BY / DISTINCT，考虑调整索引覆盖分组字段）；Using index 是好消息，说明使用了覆盖索引

第三步：常见优化手段。

  • 覆盖索引：如果查询只需要索引列，确保 SELECT 的字段都在索引中，避免回表。例如 SELECT id, name FROM user WHERE status=1 可以建联合索引 (status, name)
  • 避免隐式转换：WHERE phone = 13800138000（phone 是 varchar）会导致全表扫描，改为 WHERE phone = '13800138000'
  • 拆分大事务：如果一个事务中包含多次远程调用和数据库操作，将非数据库操作（如发消息、调 RPC）移到事务外，缩短锁持有时间
  • 分页优化（延迟关联）：LIMIT 100000, 20 会扫描 100020 行再丢弃前 10 万行。优化方案是先通过覆盖索引取出主键 ID，再 JOIN 回主表获取完整数据： 
    -- 优化前
SELECT * FROM orders ORDER BY id LIMIT 100000, 20;
-- 优化后（延迟关联）
SELECT o.* FROM orders o
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 20) AS tmp
ON o.id = tmp.id;
  • 考虑查询改写：用 EXISTS 替代 IN 子查询、用 UNION ALL 替代 OR 条件（让优化器分别走不同索引）

第四步：验证优化效果。用 EXPLAIN ANALYZE 对比优化前后的执行计划和实际耗时，关注 actual time、rows、loops 等字段，确认优化生效后再上线。

⚡ 实战 主从延迟严重，业务读到了旧数据，怎么处理？ +

第一步：确认延迟程度。在从库执行 SHOW REPLICA STATUS\G（MySQL 8.0.22+）或 SHOW SLAVE STATUS\G，查看 Seconds_Behind_Master 字段。该值为 0 表示同步正常，大于 0 表示延迟秒数，NULL 表示复制线程异常中断。更精确的方式是使用 pt-heartbeat 工具，它在主库写入时间戳心跳行，从库读取后计算真实延迟，避免 Seconds_Behind_Master 在大事务场景下的误报。

-- 从库快速检查
SHOW REPLICA STATUS\G
-- 关注字段：
--   Seconds_Behind_Source: 延迟秒数
--   Replica_IO_Running:  IO 线程是否正常
--   Replica_SQL_Running: SQL 线程是否正常
--   Last_Error: 是否有复制报错

第二步：短期应急方案——关键业务强制读主库。对于一致性要求高的场景（如支付后查订单、修改后立即查看详情），在业务层路由这些请求到主库。具体做法：

  • 写后读主库：在写操作完成后，短时间内（如 3-5 秒）该用户的后续读请求强制路由到主库。可在 Redis 中标记 user:{uid}:force_master=1，TTL 设为预期延迟时间
  • GTID 等待从库追平：如果开启了 GTID，可以在从库执行 SELECT WAIT_FOR_EXECUTED_GTID_SET('uuid:N', timeout)，等从库重放到指定 GTID 后再返回结果。适合对一致性要求极高但流量不大的场景
-- 获取主库最新 GTID
-- 主库：SELECT @@GLOBAL.gtid_executed;
-- 从库等待追平：
SELECT WAIT_FOR_EXECUTED_GTID_SET('aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee:12345', 5);
-- 返回 0 表示追平成功，返回 1 表示超时

第三步：长期根治方案。

  • 并行复制（MTS）：MySQL 5.7+ 设置 replica_parallel_type=LOGICAL_CLOCK，开启多线程并行重放。MySQL 8.0 进一步优化为 WRITESET 依赖追踪，并行度更高。配合 replica_parallel_workers 设置并行线程数（建议 8-16）
  • 半同步复制：安装 semisync 插件，主库在收到至少一个从库的 ACK 后才提交事务，从库延迟从秒级降到毫秒级。代价是主库写入延迟增加约 0.5-1ms
  • 业务层读写分离策略调整：引入中间件（如 ShardingSphere-Proxy、ProxySQL）的读写分离模块，内置延迟检测和自动切流能力。当从库延迟超过阈值时自动将读流量切回主库
  • 从库硬件升级：确保从库的磁盘 IO（SSD）、CPU、内存不低于主库，避免因硬件瓶颈导致重放速度跟不上

第四步：建立监控和告警。通过 Prometheus + mysqld_exporter 监控 mysql_slave_status_seconds_behind_master 指标，设置分级告警：延迟 > 5s 为 Warning，> 30s 为 Critical，确保问题能被及时发现和处理。

⚡ 实战 分库分表后如何不停机迁移存量数据？有哪些关键风险点？ +

整体思路：全量同步 → 增量追平 → 双写验证 → 一键切换。核心目标是把停写窗口控制在 1 分钟以内。

第一步：全量同步（历史数据）。

  • 使用 DataX（阿里开源）或 mysqldump 将原库数据导入新分片库
  • 全量开始时记录原库当前的 binlog position（File + Position 或 GTID），作为后续增量的起点
  • 全量期间原库正常读写，不影响业务

第二步：增量同步（追实时变更）。

  • 部署 Canal 订阅原库 binlog，从第一步记录的 position 开始消费，实时将增量变更应用到新分片库
  • Canal 消费滞后量（lag）持续下降直到接近 0，说明增量追平

第三步：双写验证期（可选但推荐）。

  • 在应用层新增双写逻辑：写操作先写原库，异步写新库（原库为主，新库为辅）
  • 定期用 pt-table-checksum 或自研脚本抽样比对两库数据一致性
  • 发现不一致时排查 Canal 消费逻辑，修复后重新校验

第四步：切换（停写窗口）。

  1. 停止对原库的写入（可通过配置开关或流量染色实现）
  2. 等待 Canal 将剩余积压全部消费完（通常 <1 分钟）
  3. 最终一致性校验（行数核对 + 关键字段抽样）
  4. 修改应用配置，将流量切到新分片库，恢复写入

关键风险点：

  • 双写顺序与幂等：Canal 消费可能和双写并发，需要用 binlog 的 timestamp 或版本号保证新库不被旧数据覆盖
  • 分布式 ID 连续性：迁移前后分片键（如 user_id）的生成方式要统一，切换前后不能出现 ID 碰撞
  • 事务一致性：跨分片写操作在双写期间要么两边都成功，要么回滚原库后不写新库，避免数据不一致
  • 回滚方案：切换后保留原库只读 7 天，一旦发现问题可快速切回。下线前要确认原库 binlog 保留周期足够长
