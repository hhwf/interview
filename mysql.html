存储引擎：InnoDB vs MyISAM

生产环境几乎全用 InnoDB：事务、行级锁、崩溃恢复、MVCC 是高并发场景的必要条件。

InnoDB vs MyISAM 核心对比
特性InnoDBMyISAM
事务✅ 支持❌ 不支持
外键✅ 支持❌ 不支持
锁粒度行锁（默认）表锁
崩溃恢复✅ redo log 保证❌ 不支持
MVCC✅ 支持❌ 不支持
count(*)全表扫（无 WHERE 时走最小二级索引）O(1) 直接返回行数

适用场景OLTP、高并发读写只读/分析场景（已逐步淘汰）
面试要点：InnoDB 的 Buffer Pool 是性能核心，命中率应 > 99%。innodb_buffer_pool_size 建议设为可用内存的 60–80%。Change Buffer 缓存二级索引的写操作，Merge 时批量应用，减少随机 IO。面试区分：redo log（物理日志，崩溃恢复）vs binlog（逻辑日志，主从复制 / 数据恢复）。

索引原理

InnoDB 默认使用 B+ Tree 索引。非叶节点只存键，叶节点存完整数据（聚簇索引）或主键值（二级索引），叶节点之间以双向链表相连，支持高效范围扫描。

为什么是 B+ Tree 而非 B Tree / 红黑树？
维度B+ Tree（选用）B Tree红黑树
非叶节点存数据❌ 只存键，同页存更多键✅ 存键+数据，页内键少每节点存键+数据
树高（100W行）约 3 层约 4~5 层约 20 层
磁盘 IO 次数3 次（极少）4~5 次约 20 次
范围查询✅ 叶节点链表顺序扫描需要回溯父节点中序遍历，IO 多
面试要点：索引失效常见场景：① 最左前缀原则被破坏；② 在索引列上做函数/运算；③ 隐式类型转换（如字符串列传数字）；④ != / not in（小表可能走索引）；⑤ like '%开头'。覆盖索引可避免回表，是优化查询的重要手段。索引选择性越高越好——区分度低的列（如性别）不适合单独建索引。
聚簇索引 vs 二级索引（回表）
聚簇索引（主键）
叶节点存储完整行数据
一个表只有一个聚簇索引
按主键顺序物理排列
范围查询效率极高
二级索引（非主键）
叶节点存储主键值
查询时先找主键，再回主键索引取完整行
此过程称为回表
覆盖索引可避免回表
覆盖索引：查询列全部在索引中，无需回表。EXPLAIN 的 Extra 显示 Using index 即覆盖索引。
联合索引 — 最左前缀原则
INDEX (a, b, c)
✅ WHERE a = 1
✅ WHERE a = 1 AND b = 2
✅ WHERE a = 1 AND b = 2 AND c = 3
⚠ WHERE a = 1 AND b > 2 AND c = 3 — a,b 走索引，c 不走（范围后截断）
❌ WHERE b = 2 — 跳过 a，不走索引
❌ WHERE c = 3 — 跳过 a, b，不走索引
索引失效场景
❌ WHERE YEAR(create_time) = 2024 — 对索引列使用函数
✅ WHERE create_time >= '2024-01-01' — 改为范围条件
❌ WHERE phone = 13800138000 — phone 是 varchar，隐式类型转换
❌ WHERE name LIKE '%张%' — 前缀模糊，无法用 B+Tree
✅ WHERE name LIKE '张%' — 前缀匹配，可走索引
❌ 区分度极低（如 gender 字段） — 优化器放弃索引选全表扫

事务与 ACID

Atomicity 原子性
undo log 保证：要么全部提交，要么全部回滚
Consistency 一致性
由其他三个特性共同保证数据库从合法状态到合法状态
Isolation 隔离性
锁 + MVCC 保证并发事务互不干扰
Durability 持久性
redo log + fsync（WAL）：提交事务页面修改永久化
面试要点：ACID 中最难保证的是隔离性，隔离级别越高性能越低。默认 RR 级别通过 MVCC + Next-Key Lock 避免幻读。面试常问：可重复读下 select 是快照读（不加锁），select for update / lock in share mode 是当前读（加锁看最新版本）。事务中混用快照读和当前读会导致幻读问题。
四种隔离级别与并发问题
隔离级别脏读不可重复读幻读
READ UNCOMMITTED有有有
READ COMMITTED无有有
REPEATABLE READ（默认）无无部分解决 *
SERIALIZABLE无无无
* InnoDB RR 级别通过间隙锁（Gap Lock）阻止其他事务在范围内插入，解决了大部分幻读；但当前读（SELECT FOR UPDATE）仍可能出现幻读。

MVCC（多版本并发控制）

MVCC 让读操作不加锁，通过版本链 + ReadView实现不同隔离级别下的快照读，是 InnoDB 高并发的核心机制。

行数据隐藏字段 与 undo log 版本链
每行数据的隐藏字段
trx_id 最后修改该行的事务 ID
roll_ptr 指向 undo log 的指针（版本链入口）
版本链示意
当前行（trx_id=100） ←roll_ptr← 旧版本（trx_id=80） ←roll_ptr← 更旧版本（trx_id=60）
ReadView 可见性判断规则
ReadView 四个字段
m_ids 活跃（未提交）事务 ID 列表
min_trx_id m_ids 最小值
max_trx_id 下一个将分配的事务 ID
creator_trx_id 创建该 ReadView 的事务 ID
可见性判断（针对行的 trx_id）
可见 trx_id = creator_trx_id — 自己修改的
可见 trx_id < min_trx_id — 创建前已提交
不可见 trx_id >= max_trx_id — 创建后开启的事务
不可见 trx_id in m_ids — 活跃未提交
可见 min <= trx_id < max 且不在 m_ids 中 — 已提交
💡
RC vs RR 的本质区别：
RC（READ COMMITTED）：每次 SELECT 都创建新 ReadView → 能读到其他事务最新提交，存在不可重复读。
RR（REPEATABLE READ）：事务第一次 SELECT 时创建 ReadView，整个事务复用 → 解决不可重复读。

锁机制

锁的分类
维度类型说明
粒度表锁 / 行锁 / 页锁InnoDB 支持行锁，并发更高；MyISAM 只有表锁
模式共享锁 S / 排他锁 XS 兼容 S，X 不兼容任何锁
意向锁IS / IX（表级）协调行锁与表锁，申请行锁前先加意向锁
行锁算法记录锁 / 间隙锁 / 临键锁RR 级别默认用 Next-Key Lock
面试要点：行锁只在走索引时生效，未走索引会退化为表锁。Next-Key Lock = Record Lock + Gap Lock，区间为左开右闭。死锁：两个事务互等对方持有的锁；InnoDB 自动检测死锁并回滚代价小的事务。预防：固定加锁顺序、缩短事务、避免大范围锁。
Next-Key Lock = 记录锁 + 间隙锁（防幻读）
示例：索引值 (1, 5, 10)
Next-Key Lock 范围：(-∞,1] (1,5] (5,10] (10,+∞)
锁住区间内的间隙，阻止其他事务在间隙内插入行 → 防止幻读
⚠
死锁预防： 固定业务加锁顺序；缩小事务范围减少锁持有时间；设置 innodb_lock_wait_timeout； 尽量用索引访问减少间隙锁范围。InnoDB 默认开启死锁检测，检测到后回滚代价较小的事务。

日志体系

三大核心日志对比
日志redo logundo logbinlog
所属层级InnoDB 引擎InnoDB 引擎MySQL Server 层
日志内容物理日志（哪个页改了什么）逻辑日志（逆操作，INSERT→DELETE）逻辑日志（SQL 或行变更）
写入方式循环覆盖写（固定大小文件）段文件，随机写追加写，不覆盖
主要用途崩溃恢复（WAL 机制）事务回滚、MVCC 版本链主从复制、时间点恢复
面试要点：两阶段提交保证 redo log 与 binlog 一致：先 redo log prepare → 写 binlog → redo log commit。binlog 格式：statement（省空间，主从不一致风险）/ row（精确，推荐）/ mixed。面试常问：binlog 和 redo log 的区别——binlog 是 Server 层的逻辑日志，可用于主从复制；redo log 是 InnoDB 的物理日志，用于崩溃恢复，循环写。
两阶段提交（保证 redo log 与 binlog 一致）
1. InnoDB 写 redo log，状态置为 prepare
2. Server 层写 binlog 并 fsync
3. InnoDB 将 redo log 改为 commit 状态
崩溃恢复判断
redo prepare + binlog 存在 → 提交事务
redo prepare + binlog 缺失 → 回滚事务
⚠
双1配置（最安全）： innodb_flush_log_at_trx_commit = 1（每次提交刷 redo log 到磁盘） + sync_binlog = 1（每次提交刷 binlog 到磁盘）。 即使操作系统崩溃也不丢数据，代价是每次提交都有磁盘 IO。

SQL 优化

EXPLAIN 关键字段
type — 访问类型（从好到差）
system > const > eq_ref > ref > range > index > ALL
Extra — 重要提示
Using index 覆盖索引，无需回表（好）
Using filesort 文件排序，需要加索引优化
Using temporary 使用临时表，需要优化
Using where 回表后再过滤（可能需优化）
面试要点：EXPLAIN 重点关注：type（ref > range > index > ALL）、key（实际使用索引）、rows（预估扫描行数）、Extra（Using index = 覆盖索引，Using filesort / temporary = 需要优化）。慢查询排查：开启 slow_query_log → EXPLAIN 分析 → 考虑加索引或改写 SQL。大表分页用 WHERE id > 上次最大id LIMIT n 替代 LIMIT offset,n。
深分页优化
❌ SELECT * FROM t ORDER BY id LIMIT 1000000, 10;
扫描 100W 行才能定位，性能极差
✅ 游标分页：SELECT * FROM t WHERE id > :last_id ORDER BY id LIMIT 10;
✅ 子查询定位：SELECT * FROM t WHERE id >= (SELECT id FROM t ORDER BY id LIMIT 1000000, 1) LIMIT 10;

主从复制

主从复制流程
主库执行 DML/DDL → 写入 binlog
↓
从库 IO Thread 拉取主库 binlog → 写入 relay log
↓
从库 SQL Thread 读取 relay log → 重放 SQL → 数据同步
面试要点：主从复制延迟的常见原因：单线程回放大事务、网络抖动、从库性能不足。解决：MTS 并行复制（5.7+ 基于逻辑时钟）、控制大事务大小、从库升配。半同步复制：主库等至少一个从库 ACK 后返回，兼顾性能与一致性。读写分离时，强一致读需路由到主库或等待主从同步完成再读从。
binlog 三种格式
格式记录内容优点缺点
STATEMENT原始 SQL 语句日志体积小函数/触发器可能不一致
ROW行级前后变更值最安全，精确日志体积大
MIXED自动选择兼顾安全与大小少数场景仍可能不一致
💡
主从延迟常见原因：从库单线程回放（5.6前）、主库大事务、网络延迟。 解决：MySQL 5.7 MTS 并行复制（按 logical clock）；拆分大事务；半同步复制。

分库分表

单表数据量超过 2000 万行或单库 QPS 持续打满时，通常考虑分库分表。核心决策是选分片策略，代价是引入跨分片查询复杂度和数据迁移成本。

拆分策略

垂直分库
按业务拆分：订单库、用户库、商品库。降低单库复杂度，各业务独立扩容
水平分表
同一库内按 user_id / 时间等字段拆多张表，解决单表行数过多，索引膨胀
哈希取模
user_id % N，数据分布均匀，但扩容需要迁移数据（建议一致性哈希或预分更多分片如 1024 个逻辑分片）
范围分片
按时间或 ID 区间，天然支持范围查询，但有热点风险（最新分片写入集中）
面试要点：分库分表的核心难点：① 路由：取模（扩容痛）/ 一致性哈希（迁移少）/ 基因法（跨维度查询）；② 全局唯一 ID：Snowflake / 号段模式；③ 分布式事务：2PC / Saga；④ 跨分片查询：冗余宽表 / Elasticsearch 异构。扩容时双倍扩容仅需迁移约一半数据。

ShardingSphere 两种接入模式

JDBC 模式 以 JAR 包形式嵌入 Java 应用（ShardingSphere-JDBC），业务代码零改动，框架拦截 SQL 自动路由到目标分片。适合 Java 项目，低延迟（无网络跳转），读写分离 / 分库分表配置在 YAML 中声明
Proxy 模式 独立部署 MySQL 协议代理服务（ShardingSphere-Proxy），应用连接 Proxy 如同连接单个 MySQL，与语言无关。适合异构系统（PHP/Python/Go 等）或需要 DBA 统一管控的场景，额外引入一跳网络延迟

跨分片查询挑战与解决方案

跨分片 JOIN ① 全局表（广播表）：字典类小表（省市、类目）在每个分片全量冗余，JOIN 在本地完成。② 应用层聚合：分两次查询后在内存 JOIN，适合数据量不大的场景。③ 尽量避免：拆分时把强关联表放同一分片（同一 user_id 的订单+订单明细落同一分片）
ORDER BY + LIMIT 分页 每个分片返回 (offset+limit) 条，在内存归并排序后取 limit 条。深翻页（OFFSET 大）代价极高——改用游标分页（WHERE id > last_id）或 ES 检索 ID 再回库查数据
聚合 COUNT/SUM 各分片并行执行后在应用层累加。注意 AVG 要用 SUM/COUNT 分开聚合再计算，不能直接对各分片 AVG 取均值
复杂查询 非分片键条件（如按 status、city 查询）需要广播到所有分片，全量扫描代价高。通常引入 Elasticsearch 作为二级索引：写入时双写 ES，复杂条件查询先走 ES 拿到主键列表，再回 MySQL 批量 GET

⚠
