索引原理

InnoDB 默认使用 B+ Tree 索引。非叶节点只存键，叶节点存完整数据（聚簇索引）或主键值（二级索引），叶节点之间以双向链表相连，支持高效范围扫描。

为什么是 B+ Tree 而非 B Tree / 红黑树？

非叶节点存数据❌ 只存键，同页存更多键✅ 存键+数据，页内键少每节点存键+数据
树高（100W行）约 3 层约 4~5 层约 20 层
磁盘 IO 次数3 次（极少）4~5 次约 20 次
范围查询✅ 叶节点链表顺序扫描需要回溯父节点中序遍历，IO 多
聚簇索引 vs 二级索引（回表）
聚簇索引（主键）
叶节点存储完整行数据
一个表只有一个聚簇索引
按主键顺序物理排列
范围查询效率极高

叶节点存储主键值

❌ WHERE name LIKE '%张%' — 前缀模糊，无法用 B+Tree

❌ 区分度极低（如 gender 字段） — 优化器放弃索引选全表扫

redo log + fsync（WAL）：提交事务页面修改永久化
四种隔离级别与并发问题
隔离级别脏读不可重复读幻读
READ UNCOMMITTED有有有
READ COMMITTED无有有
REPEATABLE READ（默认）无无部分解决 *
SERIALIZABLE无无无
* InnoDB RR 级别通过间隙锁（Gap Lock）阻止其他事务在范围内插入，解决了大部分幻读；但当前读（SELECT FOR UPDATE）仍可能出现幻读。

MVCC（多版本并发控制）

MVCC 让读操作不加锁，通过版本链 + ReadView实现不同隔离级别下的快照读，是 InnoDB 高并发的核心机制。

行数据隐藏字段 与 undo log 版本链
每行数据的隐藏字段
trx_id 最后修改该行的事务 ID
roll_ptr 指向 undo log 的指针（版本链入口）
版本链示意
当前行（trx_id=100） ←roll_ptr← 旧版本（trx_id=80） ←roll_ptr← 更旧版本（trx_id=60）
ReadView 可见性判断规则
ReadView 四个字段
m_ids 活跃（未提交）事务 ID 列表
min_trx_id m_ids 最小值
max_trx_id 下一个将分配的事务 ID
creator_trx_id 创建该 ReadView 的事务 ID
可见性判断（针对行的 trx_id）
可见 trx_id = creator_trx_id — 自己修改的
可见 trx_id < min_trx_id — 创建前已提交
不可见 trx_id >= max_trx_id — 创建后开启的事务
不可见 trx_id in m_ids — 活跃未提交
可见 min <= trx_id < max 且不在 m_ids 中 — 已提交
💡
RC vs RR 的本质区别：
RC（READ COMMITTED）：每次 SELECT 都创建新 ReadView → 能读到其他事务最新提交，存在不可重复读。
RR（REPEATABLE READ）：事务第一次 SELECT 时创建 ReadView，整个事务复用 → 解决不可重复读。

MySQL 高频面试题

Q1 一条 SELECT 语句的完整执行流程是什么？ +
  • 客户端建立 TCP 连接，连接器验证用户名密码，分配线程
  • 解析器：词法 + 语法分析，生成解析树；预处理器检查表/列是否存在，验证权限
  • 优化器：基于代价模型选择索引，确定 JOIN 顺序，生成执行计划
  • 执行器：按计划调用存储引擎接口逐行获取数据，返回结果集
注意：MySQL 8.0 已移除查询缓存（命中率低，缓存失效成本高）。
Q2 count(*)、count(1)、count(id) 哪个更快？ +
InnoDB 中 count(*) 和 count(1) 完全等价，优化器处理方式相同，都走最小二级索引扫描，无需读取行数据。

count(id) 需读取主键值并判断非 NULL，稍慢。

MyISAM 专门维护行数计数，count(*) 无 WHERE 时 O(1) 直接返回。
Q3 什么是幻读？InnoDB 如何解决？ +
幻读：同一事务中，两次相同范围查询返回的行数不同（另一个事务在中间插入了新行）。

InnoDB 在 RR 级别通过两种方式解决：

  • 快照读（普通 SELECT）：MVCC ReadView 在事务开始时固定，看不到后续插入的行
  • 当前读（SELECT FOR UPDATE / INSERT / UPDATE）：通过间隙锁（Gap Lock）锁住范围，阻止其他事务在间隙中插入
Q4 redo log 和 binlog 的区别？为什么需要两阶段提交？ +
redo log：InnoDB 引擎层，物理日志（记录页面改动），循环覆盖写，用于崩溃恢复（WAL 机制）。
binlog：Server 层，逻辑日志（SQL 或行变更），追加写不覆盖，用于主从复制和时间点恢复。

两阶段提交的必要性：若不用 2PC，可能出现 redo log 写了但 binlog 没写（或反之），导致主库和从库数据不一致，时间点恢复也无法对齐。2PC 通过 prepare/commit 两个状态确保两者的原子性。
Q5 int(10) 和 int(11) 有区别吗？VARCHAR(100) 和 VARCHAR(10) 性能一样吗？ +
int(10) vs int(11)：存储空间完全相同，都是 4 字节。括号内是显示宽度，仅配合 ZEROFILL 填充零，对存储范围和性能无任何影响。MySQL 8.0.17 后已废弃该语法。

VARCHAR(100) vs VARCHAR(10)：磁盘存储无差异，都只占实际数据长度 + 1~2 字节。但内存中排序和创建临时表时，MySQL 按最大声明长度分配内存，VARCHAR(100) 消耗内存是 VARCHAR(10) 的 10 倍，影响排序和临时表性能。
Q6 为什么不建议用 UUID 作主键？ +
InnoDB 聚簇索引按主键值有序存储数据页。UUID 是随机值，插入时目标页随机分散：
  • 目标页不在 Buffer Pool 中 → 频繁磁盘 IO
  • 插入位置随机 → 页分裂和碎片化严重，写性能差
  • UUID 16 字节，所有二级索引叶节点都存主键，空间占用大
推荐使用自增主键（顺序写，无页分裂）或雪花算法 ID（趋势递增，适合分布式）。
Q7 优化器什么时候会放弃索引选择全表扫描？ +
优化器基于代价模型（rows × cost_per_row），当预估索引扫描代价高于全表扫描时选择全表扫，通常出现在：
  • 索引返回行数估算超过总行数 20~30%（如区分度低的字段）
  • 统计信息过期（可用 ANALYZE TABLE 更新）
  • 查询条件导致索引无法有效过滤
可用 FORCE INDEX 强制走索引，但根本方法是优化查询条件或重新设计索引。
Q8 联合索引的最左匹配原则是什么？哪些情况会导致索引失效？ +
联合索引按从左到右的顺序构建 B+ Tree，查询条件必须从索引最左列开始连续匹配才能走完整联合索引。

常见索引失效场景：
  • 跳过最左列：WHERE b=1（index(a,b,c)）→ 走不到联合索引
  • 范围条件右侧截断：WHERE a=1 AND b>2 AND c=3，c 列不走索引
  • 对列做函数/运算：WHERE YEAR(create_time)=2024，函数导致索引失效（8.0.13+ 可建函数索引）
  • 隐式类型转换：varchar 字段用整数比较，MySQL 内部转换导致全表扫描
  • LIKE 前缀通配符：LIKE '%abc' 不走索引；LIKE 'abc%' 可走
  • NOT IN / != / IS NULL：通常走全表扫描（行数多时优化器放弃索引）
本质联合索引是多列拼接后的有序结构，跳过首列后顺序性消失，B+ Tree 无法二分定位。
Q9 什么是覆盖索引？为什么能避免回表，性能提升有多大？ +
回表：命中二级索引后，叶节点只存主键值，若 SELECT 的列不在该索引中，引擎必须拿主键再去聚簇索引查一次完整行——这额外一次随机 IO 就是回表。

覆盖索引：查询所需的所有列（SELECT + WHERE + ORDER BY）都包含在同一索引中，无需回表，直接从索引叶节点返回结果。EXPLAIN 中 Extra: Using index 表示命中覆盖索引。

性能差距：高并发场景可提升数倍到数十倍，尤其在机械盘/远程存储中，覆盖索引把随机 IO（聚簇索引随机页）变为顺序 IO（扫索引页）。

设计建议：高频查询按 (WHERE列, SELECT列) 建联合索引，或用 SELECT 主键, 关键列 代替 SELECT *。
Q10 MVCC 的实现原理是什么？ReadView 什么时候创建？RC 和 RR 的区别体现在哪？ +
核心结构：
  • 隐藏字段：每行记录有 DB_TRX_ID（最近修改的事务 ID）、DB_ROLL_PTR（指向 undo log 的回滚指针）
  • undo log 版本链：每次修改通过 roll_ptr 串成历史版本链，旧版本在 undo log 中保留
  • ReadView：快照读时创建，记录 4 个关键字段：活跃事务列表 m_ids、最小活跃事务 ID min_trx_id、下一个待分配 ID max_trx_id、当前事务 ID creator_trx_id
可见性判断：沿版本链找第一个可见版本：DB_TRX_ID < min_trx_id 已提交可见；在 m_ids 中则活跃不可见；≥ max_trx_id 是未来事务不可见。

RC vs RR 的本质区别：
  • RC：每次快照读都重新生成 ReadView → 能读到最新已提交 → 不可重复读
  • RR：同一事务内首次快照读生成 ReadView 后复用 → 读视图固定 → 可重复读
Q11 MySQL 死锁是如何产生的？InnoDB 如何检测死锁？出现死锁如何排查和预防？ +
成因：两个或多个事务互相持有对方所需的锁，形成循环等待。典型场景：事务 A 锁行 1 再锁行 2，事务 B 反序锁行 2 再锁行 1。

InnoDB 检测：维护等待图（Wait-for Graph），后台线程持续检测循环，发现死锁后回滚代价较小的事务（undo log 量少的），另一个继续执行。

排查方法：
  • SHOW ENGINE INNODB STATUS\G → LATEST DETECTED DEADLOCK 段，显示涉及的 SQL 和锁类型
  • 开启 innodb_print_all_deadlocks=ON，将死锁写入 error log，便于历史分析
预防手段：
  • 多个事务按相同顺序获取锁（消除循环等待）
  • 事务尽量短小，减少锁持有时间
  • 批量更新按主键排序，避免乱序加锁
  • 合理使用索引，避免间隙锁范围过大引入额外锁竞争
Q12 如何排查慢查询？EXPLAIN 输出中哪些字段最重要？ +
排查步骤：
  1. 开启慢查询日志：slow_query_log=ON，long_query_time=1
  2. 用 pt-query-digest 或 mysqldumpslow 聚合，找出频率最高/耗时最长的 SQL
  3. 对目标 SQL 执行 EXPLAIN 分析执行计划
EXPLAIN 关键字段：
  • type：访问类型，从优到差：system > const > eq_ref > ref > range > index > ALL。生产环境出现 ALL 需告警
  • key：实际使用的索引，NULL 表示未走索引
  • rows：预估扫描行数，越小越好
  • Extra：Using index（覆盖索引，好）；Using filesort（额外排序，需优化）；Using temporary（临时表，需优化）
  • filtered：条件过滤后剩余行比例，结合 rows 评估实际访问量
Q13 大表（千万行级）如何安全地加字段或加索引？Online DDL 的原理是什么？ +
直接 ALTER 的风险：锁全表阻塞所有读写，耗时与表大小成正比，生产环境几乎不可接受。

MySQL Online DDL（5.6+）：
  • INSTANT 算法（8.0.12+）：仅修改数据字典，不触碰数据页，加字段毫秒级完成，推荐优先使用
  • INPLACE 算法：在原表上操作，扫描全表构建索引，期间用 ROW_LOG 记录增量，最后短暂 MDL X 锁完成替换。允许并发 DML
  • COPY 算法：创建新表 → 全量复制 → 原子替换，期间阻止写操作，耗时最长
pt-online-schema-change（pt-osc）：创建影子表 → 触发器同步增量 → rename 原子替换。适合旧版 MySQL 或需更精细控制的场景。

生产实践：优先用 MySQL 8.0 INSTANT DDL；不支持时用 pt-osc 或 gh-ost，在业务低峰期执行，监控主从延迟和锁等待。
Q14 主从复制延迟的常见原因有哪些？如何监控和缓解？ +
复制流程：Master 写 binlog → IO 线程传到 Slave Relay Log → SQL 线程重放。

延迟原因：
  • 单线程重放（5.6 前）：Master 并发写，Slave SQL 线程串行重放，高 QPS 时严重落后
  • 大事务：单条 SQL 更新大量行，binlog 是一个大事务，重放耗时长
  • 从库硬件弱：磁盘 IO、CPU 性能不如主库
  • 网络抖动：主从之间带宽不足或延迟大
监控：SHOW REPLICA STATUS\G 中 Seconds_Behind_Source；更精确：用 pt-heartbeat 在主库写时间戳，从库计算真实延迟。

缓解手段：
  • MySQL 5.7+ 开启并行复制（replica_parallel_type=LOGICAL_CLOCK）
  • 拆分大事务，单次操作不超过万行
  • 对延迟敏感的读请求路由到主库（结合业务一致性要求判断）
Q15 InnoDB 的 change buffer 是什么？什么场景有效，什么场景无效？ +
作用：当写操作（INSERT/UPDATE/DELETE）的目标二级索引页不在 Buffer Pool 中时，InnoDB 不立即从磁盘读入页面，而是将变更记录在 change buffer 中，之后批量合并写入，把随机 IO 转为顺序 IO，提升写性能。

Merge 时机：查询该索引页时（读页的同时顺带 merge）、后台定时线程、数据库正常/异常关闭。

有效场景：写多读少（流水日志类）、索引选择性高（每次写的页不同，Buffer Pool 命中率低）

无效场景：
  • 唯一索引：写时必须读页验证唯一性，无法缓冲。change buffer 对唯一索引完全不起作用
  • 写后立即读：刚写的数据马上被查询，change buffer 未 merge，反而多一次 IO
实践意义这是"能用普通索引就不用唯一索引"的性能原因之一——唯一索引禁用了 change buffer 优化。
