Redis 核心知识

Redis 是单线程、纯内存的高性能 KV 存储，以丰富的数据类型、多种持久化策略和灵活的集群方案被广泛应用于缓存、会话管理、排行榜、消息队列等场景。

核心数据类型与底层编码

Redis 每种类型都有两种以上底层编码，根据数据量和大小自动切换，兼顾内存效率与读写性能。Redis 7.0 用 listpack 替代 ziplist，解决了连锁更新（级联修改）问题。

持久化机制

Redis 提供三种持久化方式：RDB（内存快照）、AOF（写命令追加日志）、混合持久化（4.0+ 推荐）。生产环境建议开启混合持久化，兼顾恢复速度与数据安全。

RDB（Redis Database）— 内存快照
触发方式
自动：save 900 1
　　　900s 内有 1 次写操作则触发
手动：BGSAVE（后台异步，推荐）
　　　SAVE（阻塞主线程，慎用）
面试要点：RDB 是全量快照，适合灾备和快速重启；AOF 是命令日志，更精确但体积大。实践推荐：AOF + everysec（最多丢 1 秒数据）+ 定期 BGREWRITEAOF 压缩。Redis 4.0+ 的 AOF-RDB 混合持久化兼顾两者优点。面试常问：BGSAVE / BGREWRITEAOF 如何不阻塞主线程——fork 子进程 + Copy-on-Write。
执行流程（Copy-On-Write）
主进程 fork 子进程
子进程将内存数据写入临时 .rdb
写完后原子替换旧文件
主进程继续处理请求（CoW 保护）
优点
文件紧凑（二进制压缩），恢复速度快
fork 后由子进程完成，主线程影响小
缺点
两次快照间宕机会丢失数据
大内存实例 fork 时会短暂阻塞（复制页表）
AOF（Append Only File）— 写命令追加日志

将每条写命令以文本格式追加到 .aof 文件，重启时逐条重放恢复数据。文件过大时触发 AOF 重写（BGREWRITEAOF），fork 子进程将内存当前状态转换为最小命令集，原子替换旧文件。

fork 阻塞问题：fork 需复制进程页表，内存越大阻塞越明显。建议单实例内存 不超过 8 GB，在业务低峰期触发 BGSAVE；同时开启 no-appendfsync-on-rewrite yes，减少 AOF 重写期间的 fsync 压力。

过期策略 & 内存淘汰策略

Redis 用惰性删除 + 定期删除两种策略互补清理过期 key；当内存触达 maxmemory 上限时，再由淘汰策略决定牺牲哪些 key 腾出空间。两者独立触发，目的不同。

Key 过期删除策略 — 两种配合使用
惰性删除
读写 key 时检查 TTL，过期则删除并返回 nil
✅ CPU 友好，不占用额外资源
❌ 过期 key 若不被访问，永不清理 → 内存泄漏
定期删除
每 100ms 随机抽取 20 个有 TTL 的 key，删除其中已过期的；
若过期比例 > 25% 则继续抽，单次最多执行 25ms
✅ 定期清理，弥补惰性删除的内存泄漏
❌ 抽样随机，不保证 100% 及时清理
内存淘汰策略（maxmemory-policy）— 内存触顶时触发
策略 作用范围 算法 适用场景
noeviction（默认） — 不淘汰，写操作直接报错 宁可报错也不丢数据
allkeys-lru 全部 key 淘汰最近最少使用的 通用缓存场景 ★ 推荐
volatile-lru 有 TTL 的 key 淘汰最近最少使用的 混合存储（部分 key 需永久保留）
allkeys-lfu 全部 key 淘汰访问频率最低的（4.0+） 热点数据差异明显 ★ 推荐
volatile-lfu 有 TTL 的 key 淘汰访问频率最低的 混合存储 + 冷热区分
allkeys-random 全部 key 随机淘汰 访问分布均匀无热点时
volatile-random 有 TTL 的 key 随机淘汰 —
volatile-ttl 有 TTL 的 key 淘汰剩余 TTL 最小的 希望短期 key 优先被释放
过期删除 vs 内存淘汰 — 核心区别
维度过期删除内存淘汰
触发条件key 超过设定的 TTLRedis 内存达到 maxmemory 上限
作用范围仅针对设了 TTL 的 key可针对全部 key 或仅带 TTL 的 key
目的清理过期数据，释放内存腾出内存空间，接受新写入
实现方式惰性删除 + 定期删除按策略（LRU/LFU/Random/TTL）选牺牲 key
💡
生产推荐：纯缓存场景优先选 allkeys-lfu（Redis 4.0+），对热点 key 的保护比 LRU 更精准；若版本较低或访问分布较均匀，选 allkeys-lru。有持久化 key 不希望被淘汰时，改用 volatile-lru / volatile-lfu。

缓存三大问题

缓存穿透、击穿、雪崩是高并发系统中最常见的缓存失效问题，三者现象相似但成因不同，需要针对性解决。

缓存穿透 — 查询数据库也不存在的 key
⚠
现象
id=-1
每次穿透
直打 DB
面试要点：缓存三大问题：① 穿透（key 不存在）→ 布隆过滤器 / 缓存空值；② 击穿（热点 key 过期）→ 互斥锁 / 逻辑过期；③ 雪崩（大量 key 同时过期 / Redis 宕机）→ 过期时间加随机抖动 / 多级缓存 / 限流降级。缓存与数据库一致性：推荐「先更新 DB，再删除缓存」+ 订阅 binlog 补偿。
查询一个数据库中也不存在的 key（如 id=-1），每次都穿过缓存直达 DB。攻击者用大量不存在的 key 可打垮数据库。
① 布隆过滤器（推荐）  — 请求先查布隆过滤器，判断 key 是否可能存在；不存在直接拦截，不查 DB。误判率约 1%，内存极小，适合大规模场景。缺点：不支持删除（可用 Counting Bloom Filter）。
② 缓存空值  — DB 查询结果为空时也写入缓存（SET key "" EX 60），设置较短 TTL 防止 key 泛滥。缺点：攻击者换不同 key 仍可绕过。
③ 参数校验  — 在入口层过滤明显非法参数（负数 ID、超长字符串、非法格式等），最低成本的防线。
缓存击穿 — 热点 key 突然过期
🔥

过期瞬间

① 互斥锁（强一致）
缓存 miss → SETNX lock EX 30
成功 → 查 DB → 写缓存 → DEL lock
失败 → sleep(50ms) → 重试

② 逻辑过期（高可用，推荐热点场景）
缓存永不设 TTL
value 内存储逻辑过期时间
发现过期 → 返回旧值 + 异步重建
优点：用户无感知，无阻塞等待

❄

场景原因解决方案

高可用方案

Redis 高可用从低到高分三个层次：主从复制（读写分离基础）→ 哨兵模式（自动故障转移）→ Cluster 集群（水平扩展 + 故障转移）。

主从复制 — 全量同步 vs 增量同步
全量同步（首次 / 差异过大）
① Slave 发送 PSYNC ? -1
② Master BGSAVE 生成 RDB
期间写命令写入 replication buffer
③ 传输 RDB → Slave 加载
④ 发送 buffer 中的增量命令
面试要点：哨兵模式：奇数个哨兵节点，主观下线 + 客观下线（超半数哨兵同意）后触发故障转移，选出新主库（优先级 > 复制偏移量最大 > runid 最小）。Cluster 模式：16384 个槽位分散到各分片，客户端通过 MOVED/ASK 路由，支持在线扩缩容（槽位迁移），最少 3 主 3 从，脑裂风险由 quorum 机制控制。
增量同步（断线重连）
① Slave 携带 replid + offset
② Master 检查 offset 是否在
repl_backlog（环形缓冲，默认 1MB）内
③ 在范围内 → 只发差异数据
④ 超出范围 → 触发全量同步
⚠
缺点：主节点宕机需手动切换，无自动故障转移，生产环境需配合哨兵或 Cluster 使用。
哨兵模式（Sentinel）— 自动故障转移
故障转移流程（3 个 Sentinel，quorum = 2）
① 单个 Sentinel 心跳超时 → 主观下线（SDOWN）
② ≥ quorum 个 Sentinel 同意 → 客观下线（ODOWN）
③ Sentinel 之间用 Raft 算法选出 Leader
④ Leader 从 Slave 中选新 Master
优先级 → 复制偏移量（越大越新）→ run ID 字典序
⑤ 通知其他 Slave 切换，更新客户端连接信息
quorum
认定客观下线所需的最少 Sentinel 数，通常设为 (n/2)+1
适用场景
数据量在单机范围内，需要自动故障转移，部署 3 个 Sentinel 即可
局限
不支持水平扩展，单节点内存上限即为数据容量上限

分片原理
数据空间 = 16384 个 slot
映射：CRC16(key) % 16384
每个 Master 负责一段 slot 范围
每个 Master 配 ≥1 个 Slave
最少 3 主 3 从（6 节点）

访问错误节点 →
MOVED 重定向（迁移完成，永久）
ASK 重定向（slot 迁移中，临时）

分布式锁

Redis 分布式锁核心是 SET NX EX 原子命令，配合唯一 value 标识持有者，释放锁必须用 Lua 脚本保证"查询 + 删除"的原子性，避免误删他人锁。

基础实现
# 加锁：NX（不存在才写）+ EX（自动过期防死锁）
SET lock:order:1001 "uuid-xxxx" NX EX 30
# 释放锁：Lua 脚本保证"验证 owner + 删除"原子执行
EVAL " if redis.call('GET', KEYS[1]) == ARGV[1] then return redis.call('DEL', KEYS[1]) else return 0 end" 1 lock:order:1001 "uuid-xxxx"
NX
key 不存在时才写入，保证互斥，同一时刻只有一个线程能加锁成功
EX 30
自动过期，防止业务崩溃后锁永不释放（死锁）
unique_value
UUID 等唯一标识，释放时验证是否是自己持有的锁，防止误删
Lua 释放
GET + DEL 非原子，必须用 Lua 脚本封装，避免并发时误删他人锁
三个常见问题与解决方案
问题原因解决方案
锁过期业务未完成 任务执行超时，锁自动释放，其他线程趁机获得锁 看门狗（Watchdog）：持有锁期间定时自动续期
误删他人锁 锁过期后其他线程已获锁，自己释放了别人的锁 释放前用 Lua 脚本原子验证 value 是否为自己的 UUID
主从切换丢锁 Master 加锁后宕机，Slave 晋升但未同步该 key Redlock 算法：向多个独立节点申请锁
Redisson 看门狗（Watchdog）机制
加锁时
Redisson 后台启动看门狗线程
默认每 10 秒自动将锁续期为 30 秒
业务完成时
主动调用 unlock()
看门狗线程停止，锁立即释放
进程崩溃时
看门狗线程随 JVM 停止
锁在 TTL 到期后自动释放，不会死锁
优点：无需手动估算锁超时时间，既防死锁又保证业务执行完才释放，是生产环境分布式锁的推荐方式。
Redlock 算法（5 个独立节点）
① 记录起始时间 t₀
② 依次向 5 个独立节点请求加锁（单节点超时约 5ms）
③ ≥ 3 个节点加锁成功，且总耗时 < 锁 TTL → 加锁成功
④ 锁的有效期 = 原始 TTL − 获取锁总耗时
⑤ 加锁失败 → 向所有节点发送 DEL 释放
⚠
争议：Martin Kleppmann 指出 Redlock 在 GC 暂停、时钟漂移场景下仍不安全。对强一致性要求极高的场景，推荐使用 ZooKeeper / etcd 的分布式锁（基于 ZAB / Raft，具备线性一致性保证）。

SCAN 完整用法（替代 KEYS 的正确姿势）

# 遍历所有匹配 user:* 的 key（cursor 从 0 开始，返回 0 说明遍历完毕）
cursor=0
while True:
    cursor, keys = redis.scan(cursor, match='user:*', count=100)
    process(keys)
    if cursor == 0:
        break

# 只扫描哈希类型（TYPE 过滤，Redis 6.0+）
SCAN 0 MATCH order:* COUNT 200 TYPE hash

# 哈希/集合/有序集合 的大 key 分批操作
HSCAN bigHash 0 MATCH * COUNT 500
SSCAN bigSet  0 COUNT 500
ZSCAN bigZSet 0 COUNT 500
⚠
生产禁用 / 慎用命令：
  • KEYS * → 全量扫描 O(N)，阻塞主线程。唯一替代：SCAN
  • DEL 大 key → 同步释放内存，可能阻塞数百毫秒。替代：UNLINK（异步）
  • HGETALL / SMEMBERS 大集合 → 一次返回大量数据。替代：HSCAN / SSCAN 分批
  • SORT 大列表 → 排序+网络传输双重开销。能用 ZSet 提前维护有序就用 ZSet
  • FLUSHDB / FLUSHALL → 必须 ASYNC 参数，否则清空期间阻塞所有命令
  • MONITOR → 严重影响 Redis 性能（每条命令都要额外广播），只在本地调试短暂使用

Redis 高频面试题

Q1 Redis 为什么这么快？ +
  • 纯内存操作：数据全在内存，读写延迟微秒级，比磁盘 IO 快 10 万倍
  • 单线程命令处理：无锁竞争，无线程切换开销（Redis 6.0 I/O 多线程化，但命令执行仍单线程）
  • I/O 多路复用：epoll（Linux）驱动，单线程可同时处理大量并发连接
  • 高效数据结构：跳表、listpack、哈希表等专为内存访问优化的结构
  • RESP 协议简洁：文本协议，解析开销极低
Q2 Redis 单线程为什么还那么快？瓶颈在哪里？ +
单线程避免了锁竞争和上下文切换，内存操作本身极快，CPU 不是瓶颈。

真正的瓶颈：
  • 内存带宽：大 value 传输消耗内存带宽
  • 网络带宽：高并发时网络 I/O 成为瓶颈（Redis 6.0 引入 I/O 多线程解决）
  • 单实例 QPS 上限：约 10～20 万 QPS，超过需 Cluster 水平分片
Q3 RDB 和 AOF 如何选择？ +
  • 纯缓存场景（允许数据丢失）：关闭持久化或仅开 RDB
  • 数据不能丢失：AOF everysec 策略
  • 生产推荐：开启混合持久化（aof-use-rdb-preamble yes），兼顾 RDB 的快速恢复和 AOF 的数据安全

RDB 的问题：两次快照间宕机丢数据；大内存 fork 会短暂阻塞主线程。
AOF 的问题：文件体积大，重启逐条重放命令，恢复慢。
Q4 如何保证缓存与数据库的数据一致性？ +
常用方案：Cache Aside Pattern（旁路缓存）

读：先查缓存 → 命中返回；未命中查 DB → 写缓存 → 返回
写：先更新 DB → 再删除缓存（不是更新缓存）

为什么删缓存而不是更新？
并发写时，多个线程可能把旧数据先后写回缓存，造成数据覆盖错误；写多读少时更新缓存也是浪费。

为什么先更新 DB 再删缓存？
先删缓存时，删后有请求读到 DB 旧值并回写缓存，脏数据存活至 TTL；先更新 DB 再删缓存出现脏数据的窗口期极短。

延迟双删：更新 DB → 删缓存 → 延迟 500ms → 再次删缓存，进一步压缩脏数据窗口。
最终一致方案：Canal 监听 MySQL binlog，异步删除/更新缓存，与业务代码解耦。
Q5 Redis Cluster 如何扩容？slot 迁移过程是什么？ +
  • 向集群添加新节点：CLUSTER MEET
  • 分配 slot 给新节点（手动或 redis-cli --cluster reshard）
  • 对目标节点执行 CLUSTER SETSLOT {slot} IMPORTING，对源节点执行 MIGRATING
  • 用 MIGRATE 命令将 key 逐个原子迁移到目标节点
  • 迁移完成后通知所有节点更新 slot 归属

迁移期间的访问：
key 已迁走 → 源节点返回 ASK 重定向（临时）
迁移完成后 → 返回 MOVED 重定向（永久），客户端更新路由表
Q6 什么情况下 Redis 会阻塞？如何避免？ +
阻塞操作 原因 替代方案
KEYS *全量扫描 O(N)SCAN 游标分批扫描
DEL 大 key同步删除大量内存UNLINK 异步删除（4.0+）
HGETALL 大集合返回大量数据HSCAN / SSCAN 分批读
BGSAVE / BGREWRITEAOF大内存 fork 复制页表控制实例内存 < 8G；低峰期触发
Lua 脚本过长阻塞事件循环控制脚本复杂度；设置 lua-time-limit
Q7 Big Key 问题如何排查和处理？ +
定义：String > 10 KB，或集合类型元素数 > 5000（具体阈值按业务定）。

危害：
  • 网络传输慢，占用带宽，影响其他命令响应
  • 主从同步 / 迁移耗时，加大主从延迟
  • DEL 大 key 会长时间阻塞主线程（同步释放内存）
排查：
  • redis-cli --bigkeys：扫描各类型最大 key，对线上有一定影响
  • rdbtools 离线分析 RDB 文件，不影响线上性能
处理：
  • 拆分：大 Hash 按业务字段拆为多个小 key
  • 压缩：序列化时开启 gzip 等压缩
  • 删除：用 UNLINK（异步，4.0+）代替 DEL；大集合先 HSCAN 分批 HDEL 再删 key
Q8 热点 Key 问题如何解决？ +
现象：某个 key 访问量极大，单 Redis 实例 CPU / 网络被打满。

  • 本地缓存（首选）：JVM 层用 Caffeine 缓存热点 key，大幅减少 Redis 请求
  • key 分片：将 hot_key 复制为 hot_key_0 ~ hot_key_N，读时随机取一个，分散到多个节点
  • 读写分离：热点 key 只读时，配置多个 Slave 分担读请求
  • 逻辑过期：热点 key 不设 TTL，后台异步刷新，避免击穿时大量并发
Q9 Redis 事务与 Lua 脚本的区别？ +
维度 MULTI/EXEC 事务 Lua 脚本
原子性顺序执行，中间失败不回滚完全原子，执行中不被打断
错误处理运行时错误，其他命令继续执行可在脚本内捕获并处理错误
网络开销MULTI + N 条命令 + EXEC，多次 RTT脚本一次发送，1 次 RTT
适用场景简单批量操作，不要求严格原子分布式锁释放、限流、复杂原子操作

注意：Redis 事务不支持回滚。语法错误 → EXEC 时全部命令取消；运行时错误 → 其他命令照常执行，与关系型数据库事务语义不同。
Q10 如何用 Redis 实现限流？ +
方案一：固定窗口计数器
INCR user:rate:{uid}:{minute} + EXPIRE 60，每分钟最多 N 次。
缺点：窗口切换时可能出现两倍流量（窗口末尾 N 次 + 新窗口开头 N 次）。

方案二：滑动窗口（Sorted Set）
  • ZADD key {ts} {ts} 记录请求时间戳
  • ZREMRANGEBYSCORE key 0 (now-window) 清除窗口外记录
  • ZCARD key 统计当前窗口内请求数，超限则拒绝
更精准，但内存随请求量增长。

方案三：令牌桶（Lua 脚本原子实现，推荐）
用 Lua 脚本原子计算令牌数（基于上次请求时间 + 生成速率），支持突发流量又能平滑限流，是生产最常用方案。
Q11 缓存穿透、击穿、雪崩如何区分？各自如何解决？ +
问题 根本原因 核心解法
穿透 查询 DB 中也不存在的 key，每次均穿透缓存直打 DB 布隆过滤器拦截；或缓存空值（短 TTL）
击穿 单个热点 key 过期瞬间，大量并发同时 miss 涌入 DB 互斥锁（强一致）或逻辑过期（永不设 TTL，异步重建）
雪崩 大量 key 同时过期 / Redis 宕机，缓存层整体失效 过期时间加随机偏移；高可用（哨兵/Cluster）；本地缓存兜底
穿透 vs 击穿的本质区别：穿透是 key 在 DB 里根本不存在；击穿是 key 在 DB 里存在，只是缓存里刚好过期了。

击穿两种解法的取舍：
互斥锁：缓存 miss → SETNX lock 30s → 仅一个线程查 DB 重建，其他线程自旋等待。数据强一致，但等待期间有延迟。
逻辑过期：key 永不设 TTL，value 中存储逻辑过期时间；发现逻辑过期时立即返回旧值 + 异步线程重建缓存。用户无感知无阻塞，但短暂返回旧数据（最终一致）。

答题技巧先用"三者都是缓存失效引发 DB 压力的问题，但成因不同"破题，再逐一展开，最后对比互斥锁与逻辑过期的适用场景，展现系统设计深度。
Q12 Redis 分布式锁如何实现？Redlock 解决了什么问题？ +
基础实现：SET lock_key {uuid} NX EX 30（原子命令，Redis 2.6.12+）
释放时用 Lua 脚本保证原子性（判断 value == uuid 再 DEL），防止误删其他线程持有的锁。

单节点方案的两个问题：
  • 业务超时锁自动过期：用 Redisson watchdog 机制——锁持有时每 10s 自动续期，业务执行完主动删除
  • 主从切换锁丢失：主节点写锁后宕机，Slave 未同步，新 Master 上可再次加同一把锁 → 两个客户端同时持锁
Redlock 算法（多节点方案）：向 N（通常 5）个独立 Redis 实例依次加锁，超过半数（≥3）成功且总耗时 < 锁 TTL 才认为加锁成功，任意少数节点宕机时锁仍安全。

Redlock 的争议：Martin Kleppmann 指出 GC STW / NTP 时钟跳变 / 网络延迟等原因可能导致锁已过期但持有者不知道，从而出现并发安全问题。强一致性场景建议用 ZooKeeper / etcd（基于顺序临时节点，不依赖时间，具备线性一致性）。

答题技巧回答完基础实现后主动提 Redlock，再说争议和 ZK 替代方案，展示对分布式系统设计权衡的理解。
Q13 Redis 主从复制原理是什么？哨兵如何实现自动故障切换？ +
主从复制流程：
  • 全量同步（首次 / repl_backlog 溢出）：Master 执行 BGSAVE 生成 RDB 传给 Slave；Slave 清空本地载入 RDB；RDB 生成期间的写命令暂存 replication buffer，传输 RDB 后补发
  • 增量同步（正常运行 / 短暂断线重连）：Slave 上报 repl_id + offset，Master 在 repl_backlog 环形缓冲区中找到对应位置，只补发增量命令
哨兵（Sentinel）自动故障切换四步骤：
  1. 监控：每个 Sentinel 每秒向 Master/Slave 发送 PING，超时 → 标记主观下线（sdown）
  2. 客观下线（odown）：向其他 Sentinel 询问，确认票数超过 quorum（通常过半）→ 确认 Master 不可达
  3. 选举 Leader Sentinel：基于 Raft 协议从 Sentinel 集群中选出一个 Leader 执行故障转移
  4. 故障转移：Leader Sentinel 从 Slave 中选出新 Master（优先级高、offset 最大的），发送 SLAVEOF NO ONE，通知其他 Slave 指向新 Master，并更新客户端路由
Q14 Redis 的内存淘汰策略有哪些？如何选择？ +
策略 含义 适用场景
noeviction内存满时拒绝写入，返回错误数据不可丢失，宁可报错
allkeys-lru全部 key 中淘汰最近最少使用的纯缓存场景（最常用）
volatile-lru仅在设置了 TTL 的 key 中淘汰 LRU缓存与持久数据混存
allkeys-lfu全部 key 中淘汰访问频率最低的（4.0+）热点分布不均匀时比 LRU 更精准
volatile-lfu仅在有 TTL 的 key 中淘汰频率最低的混存时需要 LFU 精度
volatile-ttl有 TTL 的 key 中优先淘汰剩余存活时间最短的希望快过期的先被踢出
allkeys-random全部 key 中随机淘汰访问分布均匀时可用，少用

选择建议：
• 纯缓存：allkeys-lru（最常用）或 allkeys-lfu（热点分布不均时更精准）
• 缓存与持久数据混存：volatile-lru / volatile-lfu，只淘汰有 TTL 的 key，保护持久数据
• 不允许任何数据丢失：noeviction，配合监控告警及时扩容
Q15 Redis Sentinel 与 Cluster 如何选型？ +
维度 Sentinel（哨兵） Cluster（集群）
数据分片 ❌ 无分片，数据全在单主节点 ✅ 16384 个 slot 分布到多主节点
水平扩容 ❌ 内存上限 = 单机内存 ✅ 增加节点即可线性扩容
高可用 ✅ Sentinel 自动 Failover ✅ 每个 Master 有 Slave，内置 Failover
多 key 事务 ✅ 支持（单节点） ⚠ 仅限同一 slot（用 hash tag {} 规避）
运维复杂度 低，配置简单 较高，需管理 slot 分配与迁移

选型建议：
• 数据量 < 单机内存上限（通常 < 50 GB）且不需横向扩容 → Sentinel，运维简单，支持完整 Redis 命令
• 数据量大、需水平扩展或写 QPS 极高 → Cluster，读写分片到多主节点，天花板更高
• Cluster 的限制：不支持跨 slot 的 MGET / MSET / pipeline（需用 hash tag 强制同 slot）
⚡ 实战 线上 Redis 内存暴涨怎么排查？ +

第一步：查看内存概况。使用 INFO memory 命令获取关键指标：

  • used_memory：Redis 实际使用的内存总量。
  • used_memory_rss：操作系统视角分配给 Redis 的物理内存（含碎片）。
  • mem_fragmentation_ratio：碎片率 = used_memory_rss / used_memory。正常范围 1.0～1.5；若 > 1.5 说明内存碎片严重，可考虑启用 activedefrag yes 或重启实例。
  • maxmemory / maxmemory-policy：确认是否设置了上限及当前淘汰策略。

第二步：定位大 Key。大 Key 是内存暴涨最常见的原因之一。

  • 使用 redis-cli --bigkeys 快速扫描各数据类型中最大的 Key（内部基于 SCAN，不会阻塞主线程）。
  • 对于疑似大 Key，用 MEMORY USAGE <key> 精确查看单个 Key 占用字节数（Redis 4.0+）。
  • 也可通过 RDB 分析工具（如 redis-rdb-tools）离线解析 RDB 快照，按内存排序导出 Top-N Key。

第三步：检查未设 TTL 的 Key。大量没有过期时间的 Key 会持续累积。执行：

redis-cli --scan --pattern '*' | xargs -I {} redis-cli TTL {} | grep -c ':-1'

如果返回的 -1（永不过期）数量非常大，说明有大量 Key 未设置 TTL。业务侧应排查写入逻辑，确保缓存 Key 都带有合理的过期时间。

第四步：检查过期策略是否正常。Redis 的过期删除采用惰性删除 + 定期删除双重机制。如果短时间内写入了大量相同过期时间的 Key，定期删除的采样可能无法及时清理，导致内存尖峰。可通过 INFO stats 中的 expired_keys 和 evicted_keys 观察过期和淘汰速率。

第五步：调整 maxmemory 和淘汰策略。如果确认是正常业务增长导致内存不足：

  • 在线调整上限：CONFIG SET maxmemory 4gb
  • 切换淘汰策略：CONFIG SET maxmemory-policy allkeys-lru（纯缓存场景推荐）
  • 长期方案：扩容实例或迁移至 Cluster 模式进行水平扩展。
⚡ 实战 缓存和数据库双写一致性怎么保证？ +

核心原则：Cache Aside Pattern（旁路缓存模式）。读流程：先读缓存，命中则返回；未命中则读 DB，将结果写入缓存。写流程：先更新数据库，再删除缓存（而非更新缓存），这是业界推荐的标准做法。

方案一：先更新 DB + 再删缓存 + 延迟双删。

在更新 DB 后，先删除缓存，等待一小段时间（如 500ms，大于一次读请求耗时）后再删一次缓存。第二次删除是为了兜底：在第一次删除和 DB 更新之间的时间窗口内，可能有并发读请求将旧数据重新写入缓存，延迟双删可以清除这个脏数据。

deleteCache(key); updateDB(data); Thread.sleep(500); deleteCache(key);

缺点：sleep 阻塞业务线程，且延迟时间不好精确设定。

方案二：Canal 订阅 binlog 异步更新缓存。

使用 Canal 伪装为 MySQL Slave，订阅数据库的 binlog 变更事件。当监听到数据变更时，通过消息队列（如 Kafka / RocketMQ）异步发送缓存删除或更新指令。优点是与业务代码完全解耦，不侵入业务逻辑，且消息队列提供重试机制保证最终一致性。

最终一致性 vs 强一致性选择：

  • 最终一致性（推荐）：上述两种方案均为最终一致性方案。绝大多数业务场景（商品详情、用户资料等）可以接受短暂的不一致。
  • 强一致性：如果业务对一致性要求极高（如金融余额），应绕过缓存直接读 DB，或使用分布式锁串行化读写，但会显著降低吞吐量。

极端并发场景的解决方案：

当读请求和写请求高度并发时，可能出现：读请求先于写请求删除缓存之前读到旧值并写回缓存，导致脏数据。解决方案：

  • 分布式锁串行化：对同一 Key 的读写操作使用分布式锁（如 Redisson 的 RLock），保证同一时刻只有一个线程操作该 Key，彻底消除并发脏数据。
  • 消息队列重试删除：删缓存失败时将删除操作投入消息队列，由消费者重试删除，直到成功。配合 Canal 方案可实现可靠的最终一致性。
⚡ 实战 Redis Cluster 某节点挂了，故障转移全过程是什么？ +

第一阶段：故障检测 —— subjective down → objective down。

Cluster 中每个节点通过 Gossip 协议定期互相 PING/PONG。当某个 Master 节点在 cluster-node-timeout（默认 15s）内没有回复 PONG 时，检测节点将其标记为 PFAIL（Possible Failure，主观下线）。随后，该节点通过 Gossip 传播 PFAIL 状态；当集群中大多数 Master 节点（超过半数）都报告该节点为 PFAIL 时，状态升级为 FAIL（Objective Failure，客观下线），此时触发故障转移。

第二阶段：Slave 选举新 Master。

故障 Master 的所有 Slave 发起选举。选举规则：

  • 优先选择数据最新（复制偏移量 replication offset 最大）的 Slave。
  • 候选 Slave 向集群中所有 Master 发起 FAILOVER_AUTH_REQUEST。
  • 获得超过半数 Master 投票的 Slave 胜出，晋升为新 Master。

第三阶段：Slot 重新分配与角色切换。

新 Master 接管故障 Master 负责的所有 hash slot（Redis Cluster 共有 16384 个 slot）。它向集群广播自己的新角色和 slot 映射表，其他节点更新本地的 slot → node 映射。原故障 Master 的所有 slot 现在由新 Master 服务。

第四阶段：客户端连接更新。

客户端向旧节点发送命令时，会收到重定向响应：

  • MOVED <slot> <new-master-ip:port>：永久重定向，客户端应更新本地 slot 映射缓存。
  • ASK <slot> <node-ip:port>：临时重定向（slot 正在迁移中），客户端需先发 ASKING 命令再发实际命令。

成熟的客户端（如 Jedis Cluster、Lettuce、Redisson）会自动处理重定向并刷新拓扑。

脑裂问题与防护：

脑裂（Split-Brain）发生在网络分区时：少数分区中的 Master 无法收到大多数节点的 PONG，但自身仍在接受写入，导致分区恢复后数据丢失。防护措施：

  • 配置 min-replicas-to-write 1：要求 Master 至少要有 1 个可通信的 Slave 才接受写入。当 Master 被隔离到少数分区、Slave 被其他分区拉走后，该 Master 将拒绝写入，避免数据丢失。
  • 配置 min-replicas-max-lag 10：Slave 与 Master 之间的复制延迟超过 10 秒时，Master 也拒绝写入，防止数据落后过多。
  • 这两个参数配合使用，能有效防止脑裂场景下的数据丢失。
