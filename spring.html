Spring 核心原理

深入剖析 Spring IoC 容器、Bean 生命周期、AOP 代理机制、声明式事务与三级缓存循环依赖解决方案，是后端面试的必考核心。

● Spring · IoC & DI

IoC & Bean 容器

IoC（控制反转）是 Spring 的核心思想——对象的创建和依赖关系的管理 从应用代码移交给容器，代码只声明"需要什么"而不关心"怎么创建"。 DI（依赖注入）是 IoC 的实现方式。

BeanFactory vs ApplicationContext

容器继承体系与核心差异
BeanFactory
· IoC 容器根接口
· 懒加载：首次 getBean() 时才实例化
· 功能最基础：实例化、配置、装配
· 适合内存极度受限环境
⊂
ApplicationContext（推荐）
· BeanFactory 的超集，企业级扩展
· 饥饿加载：启动时实例化所有单例 Bean
· 内置 MessageSource（国际化）
· 内置 ApplicationEvent 事件机制
· 支持 AOP、@Transactional 等
常用实现类
ClassPathXmlApplicationContext
AnnotationConfigApplicationContext
AnnotationConfigServletWebServerApplicationContext
（Spring Boot Web）

Bean Scope（作用域）

singleton（默认）
容器内唯一实例，所有请求共享同一对象。启动时创建，容器销毁时销毁。
prototype
每次 getBean() 返回新实例。Spring 不管理其销毁，调用方负责释放。
request
每个 HTTP 请求创建一个实例，请求结束后销毁。仅 Web 容器有效。
session
每个 HTTP Session 一个实例，Session 失效后销毁。仅 Web 容器有效。
application
ServletContext 生命周期内唯一，类似 singleton 但跨多个 ApplicationContext。
websocket
每个 WebSocket 会话一个实例，连接断开后销毁。

面试要点：IoC 容器的本质是一个工厂 + 注册表，依赖注入的最佳实践是构造器注入（依赖可 final、可测试）。区分 BeanFactory（延迟加载）与 ApplicationContext（饥饿加载 + 事件机制）。@Autowired 按类型注入，@Resource 按名称注入，两者混用时注意歧义 Bean 的 @Primary / @Qualifier 处理。
● Spring · Bean Lifecycle

Bean 生命周期

Spring Bean 从 BeanDefinition 到可用实例，经历完整的初始化流程， 每个阶段都暴露扩展点，AOP 代理、事务拦截器都在这里织入。

Bean 完整生命周期——11 个关键节点
① 实例化（Instantiation）
容器通过反射调用构造方法创建对象。若有多个构造器，@Autowired 指定优先，否则选无参构造。此时对象字段全为默认值。
② 属性填充（Property Population）
populateBean() 处理 @Autowired / @Resource / @Value 注入。三级缓存在此阶段解决循环依赖。
③ Aware 接口回调
按顺序：BeanNameAware → BeanClassLoaderAware → BeanFactoryAware → ApplicationContextAware
④ BeanPostProcessor#beforeInitialization()
@PostConstruct 方法由 CommonAnnotationBeanPostProcessor 在此阶段调用。
⑤ InitializingBean.afterPropertiesSet()
实现 InitializingBean 接口的 Bean 在此执行初始化逻辑。
⑥ 自定义 init-method / @Bean(initMethod)
XML init-method 或 @Bean(initMethod="...") 指定的方法在此调用。
⑦ BeanPostProcessor#afterInitialization() ⭐ AOP 代理在此创建
AnnotationAwareAspectJAutoProxyCreator 在此检查是否需要创建 AOP 代理。若 Bean 匹配切点，返回代理对象替换原始 Bean。容器中存放的是代理。
⑧ Bean 就绪，供业务使用
注册到 singletonObjects（一级缓存），所有注入点拿到此对象（可能是代理）。
⑨ @PreDestroy（容器关闭时）
推荐的清理入口，用于资源释放、连接关闭等。
⑩ DisposableBean.destroy() / 自定义 destroy-method
实现 DisposableBean 接口或 @Bean(destroyMethod) 指定的销毁方法。
■ 容器感知   ■ 初始化扩展点   ■ AOP 织入点（重要！）   ■ 销毁阶段
面试要点：Bean 生命周期的扩展点顺序：BeanPostProcessor.postProcessBeforeInitialization → @PostConstruct / init-method → postProcessAfterInitialization（AOP 代理在此生成）。面试常问：为什么循环依赖不能用构造器注入？因为实例化阶段对象还未放入缓存，无法解决依赖。
● Spring · AOP

AOP 原理

AOP（面向切面编程）将横切关注点（日志、事务、权限校验等）从业务逻辑中分离， 通过代理模式在运行时将增强逻辑织入目标方法，实现解耦。

Aspect（切面）
横切逻辑的模块化封装，通常是一个带 @Aspect 的类，包含 Pointcut + Advice。
JoinPoint（连接点）
程序执行过程中可以被拦截的点，Spring AOP 仅支持方法级别。
Pointcut（切点）
用表达式描述"要拦截哪些 JoinPoint"，如 execution(* com.example..*.*(..))。
Advice（通知）
在 JoinPoint 处执行的增强代码，分 Before / After / Around 等类型。
Weaving（织入）
将 Aspect 应用到目标对象的过程。Spring AOP 是运行时动态代理织入。
Target（目标对象）
被代理的原始对象，业务代码调用的是代理，代理再委托给 target。
JDK 动态代理 vs CGLIB 代理——选择机制
JDK 动态代理
条件：目标类实现了至少一个接口
原理：Proxy.newProxyInstance() 生成实现相同接口的代理类，调用转发到 InvocationHandler.invoke()
限制：只能代理接口中声明的方法
CGLIB 代理（Spring Boot 默认）
条件：无接口，或强制指定 proxyTargetClass=true
原理：ASM 字节码生成目标类的子类，重写非 final 方法
限制：final 类/方法无法代理；需要无参构造
Spring Boot 2.x+ 默认使用 CGLIB（即使有接口），因为 CGLIB 可以解决 @Configuration 类的 @Bean 方法自调用问题。
强制使用 JDK 代理：@EnableAspectJAutoProxy(proxyTargetClass=false)
Advice 类型与执行顺序（@Around 包裹所有）
@Around — proceedingJoinPoint.proceed() 前的代码 ↓
@Before — 目标方法执行前 ↓
目标方法执行（target.method()）
@AfterReturning（正常返回后）↑
@AfterThrowing（异常抛出后，与 AfterReturning 互斥）↑
@After（finally，无论正常/异常都执行）↑
@Around — proceed() 后的代码 ↑
多个 Aspect 时按 @Order 从小到大排序（外层先进后出）。
面试要点：AOP 代理选择规则：实现接口用 JDK 动态代理（基于 InvocationHandler），无接口或强制使用用 CGLIB（继承子类）。Spring Boot 2.x 起默认 CGLIB。self 调用失效：同类内部方法互调绕过代理，解决方案：注入自身 / AopContext.currentProxy() / 拆分类。
● Spring · Transaction

事务管理

Spring 声明式事务基于 AOP，@Transactional 通过代理拦截方法调用， 在方法前后自动管理事务的开启、提交与回滚。

7 种事务传播行为（Propagation）
传播行为外层有事务时外层无事务时典型场景
REQUIRED（默认）加入外层事务创建新事务绝大多数写操作
REQUIRES_NEW挂起外层，创建独立新事务创建新事务日志记录（独立提交）
SUPPORTS加入外层事务以非事务方式运行只读查询，灵活适配
NOT_SUPPORTED挂起外层，以非事务运行以非事务运行发送 MQ 消息
MANDATORY加入外层事务抛 IllegalTransactionStateException强制要求调用方开启事务
NEVER抛 IllegalTransactionStateException以非事务运行强制要求无事务环境
NESTED创建嵌套事务（Savepoint）等同 REQUIRED部分操作可独立回滚
REQUIRES_NEW vs NESTED：REQUIRES_NEW 是两个独立事务（外层回滚不影响 NEW 已提交的数据）；NESTED 是父子关系（父回滚则子全部撤销）。
@Transactional 四大失效陷阱
① 同类方法自调用（最常见）
this.method() 直接调用，绕过了 Spring AOP 代理。
修复：注入自身 @Autowired ApplicationService self，通过 self 调用；或拆分到不同 Bean。
② 异常类型不匹配
默认只回滚 RuntimeException 和 Error。抛出受检异常（IOException 等）不自动回滚。
修复：@Transactional(rollbackFor = Exception.class)
③ 方法非 public
CGLIB 子类无法重写 private/final 方法。修复：改为 public。
④ 异常被 catch 吞掉
try-catch 捕获后未重抛，代理层收不到异常不触发回滚。
修复：重新 throw，或调用 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()
面试要点：事务失效常见场景：① 非 public 方法；② 同类 self 调用；③ 异常被 catch 吞掉；④ 抛出的是 Checked Exception（默认只回滚 RuntimeException）。传播行为面试重点：REQUIRED（默认，共用事务）vs REQUIRES_NEW（独立事务，外层回滚不影响）vs NESTED（嵌套事务，依赖 savepoint）。
● Spring · Circular Dependency

循环依赖与三级缓存

A 依赖 B，B 依赖 A——Spring 通过三级缓存在 Bean 尚未完成初始化时 提前暴露"早期引用"来打破循环，而不需要对象提前完全就绪。

三级缓存结构
一级缓存
singletonObjects
Map<name, Object>
存放完全初始化完成的单例 Bean（含 AOP 代理）。getBean() 优先命中此处。
二级缓存
earlySingletonObjects
Map<name, Object>
存放早期暴露的 Bean 引用（已实例化但未完成属性注入），防止对同一 Bean 重复调用 ObjectFactory。
三级缓存
singletonFactories
Map<name, ObjectFactory>
存放 ObjectFactory lambda，用于生成早期 Bean 引用（若有 AOP 则在此生成代理）。Bean 实例化后立即存入。
A ↔ B 循环依赖解析流程（Setter / Field 注入）
1
getBean(A) → 缓存未命中 → 实例化 A → 将 A 的 ObjectFactory 放入 三级缓存
2
populateBean(A) → 需注入 B → getBean(B) → 缓存未命中 → 实例化 B → B 的 ObjectFactory 放入 三级缓存
3
populateBean(B) → 需注入 A → getBean(A) → 三级缓存命中 → 调用 A 的 ObjectFactory（若有 AOP 则生成代理）→ 早期 A 存入 二级缓存
4
B 成功注入早期 A → B 完成初始化 → B 存入 一级缓存
5
回到 A 的注入流程 → A 成功注入完整 B → A 完成初始化 → A 存入 一级缓存
✓
循环依赖解决完成。A 和 B 均在一级缓存，各自持有对方的完整/早期引用。
⚠
以下情况无法解决循环依赖：
  • ▸构造器注入：实例化时就需要依赖，无法先放入三级缓存，直接抛 BeanCurrentlyInCreationException。
  • ▸Prototype Bean：每次 getBean 都创建新实例，Spring 不维护早期引用，无法打破循环。
  • ▸@Async + 循环依赖：@Async 的代理创建时机晚于三级缓存暴露，前后代理不一致，Spring 检测到后抛异常。解决：@Lazy 打破循环。
💡
为什么需要三级缓存？二级不够吗？
若 A 存在 AOP，早期引用需要是代理对象。三级缓存存 ObjectFactory，延迟但唯一地生成代理——保证整个生命周期中 A 的代理对象只有一个，所有早期引用都指向同一个代理，维护了 Bean 的一致性。
面试要点：三级缓存解决循环依赖的关键是提前暴露半成品对象。一级缓存存完整 Bean，二级存早期代理引用，三级存 ObjectFactory（延迟生成代理）。构造器注入无法解决（对象未创建就要注入），@Lazy 可绕过构造器循环。Spring Boot 2.6+ 默认禁止循环依赖，需显式开启。
● Spring Boot · Core

Spring Boot 启动与自动配置

Spring Boot 的核心价值是约定大于配置——通过自动配置（Auto-Configuration）和 Starter 依赖管理， 让开发者无需繁琐的 XML 配置即可快速启动生产级应用。

SpringApplication.run() 启动流程

Spring Boot 启动的 8 个关键阶段
① 创建 SpringApplication 实例
推断应用类型（SERVLET / REACTIVE / NONE），从 spring.factories 加载 ApplicationContextInitializer 和 ApplicationListener 列表，推断 main class。
② 创建 SpringApplicationRunListeners & 发布 starting 事件
RunListeners 包装 ApplicationEventMulticaster，对外广播各阶段事件。EventPublishingRunListener 是默认实现。
③ prepareEnvironment — 准备环境
按优先级加载配置源：命令行参数 → 系统属性 → 环境变量 → application.yml → 默认值。Profile 激活逻辑在此处理。
④ createApplicationContext — 创建 ApplicationContext
根据应用类型选择：AnnotationConfigServletWebServerApplicationContext（Servlet Web）、AnnotationConfigReactiveWebServerApplicationContext（Reactive）或 AnnotationConfigApplicationContext（非 Web）。
⑤ prepareContext — 准备上下文
应用所有 ApplicationContextInitializer，将主配置类（带 @SpringBootApplication 的类）注册为 BeanDefinition。
⑥ refreshContext ⭐ 核心阶段
调用 AbstractApplicationContext.refresh()，完成：BeanDefinition 扫描注册 → 自动配置类导入 → 所有单例 Bean 实例化 → AOP 代理创建 → 内嵌 Tomcat 启动（onRefresh() 回调） → Servlet/Filter 注册。
⑦ afterRefresh — Runner 回调
按顺序调用所有 ApplicationRunner（接收 ApplicationArguments）和 CommandLineRunner（接收 String[]）。常用于初始化数据、预热缓存。
⑧ 发布 ApplicationReadyEvent，应用就绪，开始接受请求
控制台打印启动耗时。全部 shutdown hook 已注册，JVM 退出时触发 @PreDestroy 等销毁逻辑。

自动配置原理

@SpringBootApplication → 自动配置触发链路
@SpringBootApplication = 三个注解的组合
@SpringBootConfiguration
等价于 @Configuration
@ComponentScan
扫描主类同包及子包
@EnableAutoConfiguration
触发自动配置的入口
↓ @Import(AutoConfigurationImportSelector.class)
Spring Boot 2.x
读取所有 jar 下的 META-INF/spring.factories，取 EnableAutoConfiguration key 对应的类名列表（~130 个候选类）。
Spring Boot 3.x（SPI 变更）
改用独立文件 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports，每行一个类名，加载更快，不再混入 spring.factories。
↓ 条件注解过滤（Conditional Filtering）— 用户定义优先
@ConditionalOnClass
classpath 存在指定类时才注册（如有 DataSource 才配置数据库池）
@ConditionalOnMissingBean
容器中不存在该 Bean 时才注册——保证用户定义优先于自动配置
@ConditionalOnProperty
指定配置项等于某值时才注册，支持 havingValue / matchIfMissing
@ConditionalOnWebApplication
仅在 Web 上下文中生效，Servlet 与 Reactive 可分别区分

内嵌 Tomcat 启动原理

内嵌容器的启动与与外置 Tomcat 的对比
  1. 自动配置注册工厂：ServletWebServerFactoryAutoConfiguration 检测 classpath 有 Tomcat，注册 TomcatServletWebServerFactory Bean。
  2. refreshContext 触发 onRefresh()：ServletWebServerApplicationContext.createWebServer() 调用工厂创建 Tomcat 并绑定端口（默认 8080）。
  3. DispatcherServlet 注册：DispatcherServletAutoConfiguration 将 DispatcherServlet 注册为前端控制器。
维度内嵌 Tomcat（推荐）外置 Tomcat
打包方式Fat JAR（java -jar app.jar）WAR 包（部署到 webapps/）
启动入口JVM 直接运行 main()，Spring 控制一切Tomcat 调用 SpringBootServletInitializer.onStartup()
适用场景微服务、容器化（Docker/K8s）多应用共享容器的传统部署
面试要点：Spring Boot 启动的核心是 refreshContext()，它触发容器完整初始化（Bean扫描 → 实例化 → AOP代理 → 内嵌Tomcat启动）。自动配置本质是 SPI + 条件注解：2.x 从 spring.factories 加载候选类，3.x 改为 AutoConfiguration.imports；再通过 @ConditionalOnXxx 过滤，遵循"用户定义优先"（@ConditionalOnMissingBean）。
● Spring Boot · Configuration

Spring Boot 配置体系

Spring Boot 统一了配置来源，支持 properties / YAML / 环境变量 / 命令行等多种方式， 通过明确的优先级规则解决多配置源的冲突，Profile 机制进一步实现多环境管理。

配置文件加载优先级（高 → 低）

常用配置源优先级顺序
1
命令行参数（--server.port=8081）— 优先级最高
2
Java System Properties（-Dserver.port=8081）
3
OS 环境变量（SERVER_PORT=8081，支持松散绑定）
4
application-{profile}.yml / .properties（jar 包外 → jar 包内，/config/ 子目录优先）
5
application.yml / .properties（jar 包外 → jar 包内，/config/ 子目录优先）
6
@PropertySource 注解的配置文件（不支持 YAML）
7
SpringApplication.setDefaultProperties()（程序内默认值，优先级最低）
同级时：jar 包外部目录 > jar 包内 classpath；/config/ 子目录 > 根目录。
profile 文件优先于不带 profile 的主配置文件——application-prod.yml 中相同 key 会覆盖 application.yml。

@ConfigurationProperties vs @Value

两种配置注入方式对比
维度@ConfigurationProperties（推荐）@Value
绑定方式批量绑定到 Java 类字段，前缀匹配单个属性注入，${key} 或 SpEL #{expr}
松散绑定支持（camelCase / kebab-case / 下划线 / 大写互转）不支持，key 必须精确匹配
复杂类型支持 List / Map / 嵌套对象，IDE 有自动补全String / 基本类型，类型错误运行时才报错
JSR-303 验证支持（@Validated + @NotNull/@Min 等），启动时即校验不支持
SpEL 表达式不支持支持（#{T(Math).PI}）
适用场景多个相关配置项（连接池、限流阈值等）偶发性单值注入或需要动态计算的场景

Profile 多环境管理

Profile 激活方式与使用规范
激活方式（优先级从高到低）
--spring.profiles.active=prod（命令行）
-Dspring.profiles.active=prod（JVM 参数）
spring.profiles.active=prod（配置文件）
SpringApplication.setAdditionalProfiles()（代码）
配置文件命名约定
application-dev.yml — 开发
application-test.yml — 测试
application-prod.yml — 生产
profile 文件会合并并覆盖 application.yml 中相同的 key
@Profile 注解
标注在 @Bean / @Configuration / @Component 上，仅在匹配 Profile 时注册。
Spring Boot 3.1+ 支持逻辑表达式：@Profile("prod | staging")
面试要点：配置优先级：命令行 > JVM属性 > 环境变量 > profile配置文件 > 默认配置文件。多个相关配置项一律用 @ConfigurationProperties——支持松散绑定、类型安全和 Bean Validation，可在启动时就检测出配置错误。生产环境通过命令行 --spring.profiles.active=prod 激活 Profile，不要在代码中硬编码。
