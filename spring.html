Spring 核心原理

深入剖析 Spring IoC 容器、Bean 生命周期、AOP 代理机制、声明式事务与三级缓存循环依赖解决方案，是后端面试的必考核心。

● Spring · IoC & DI

IoC & Bean 容器

IoC（控制反转）是 Spring 的核心思想——对象的创建和依赖关系的管理 从应用代码移交给容器，代码只声明"需要什么"而不关心"怎么创建"。 DI（依赖注入）是 IoC 的实现方式。

BeanFactory vs ApplicationContext

容器继承体系与核心差异
BeanFactory
· IoC 容器根接口
· 懒加载：首次 getBean() 时才实例化

· 内置 ApplicationEvent 事件机制
· 支持 AOP、@Transactional 等

（Spring Boot Web）

Bean Scope（作用域）

singleton（默认）
容器内唯一实例，所有请求共享同一对象。启动时创建，容器销毁时销毁。
prototype
每次 getBean() 返回新实例。Spring 不管理其销毁，调用方负责释放。
request
每个 HTTP 请求创建一个实例，请求结束后销毁。仅 Web 容器有效。
session
每个 HTTP Session 一个实例，Session 失效后销毁。仅 Web 容器有效。
application
ServletContext 生命周期内唯一，类似 singleton 但跨多个 ApplicationContext。
websocket
每个 WebSocket 会话一个实例，连接断开后销毁。

● Spring · Bean Lifecycle

Bean 生命周期

Spring Bean 从 BeanDefinition 到可用实例，经历完整的初始化流程， 每个阶段都暴露扩展点，AOP 代理、事务拦截器都在这里织入。

Bean 完整生命周期——11 个关键节点
① 实例化（Instantiation）
容器通过反射调用构造方法创建对象。若有多个构造器，@Autowired 指定优先，否则选无参构造。此时对象字段全为默认值。
② 属性填充（Property Population）
populateBean() 处理 @Autowired / @Resource / @Value 注入。三级缓存在此阶段解决循环依赖。
③ Aware 接口回调
按顺序：BeanNameAware → BeanClassLoaderAware → BeanFactoryAware → ApplicationContextAware
④ BeanPostProcessor#beforeInitialization()
@PostConstruct 方法由 CommonAnnotationBeanPostProcessor 在此阶段调用。
⑤ InitializingBean.afterPropertiesSet()
实现 InitializingBean 接口的 Bean 在此执行初始化逻辑。
⑥ 自定义 init-method / @Bean(initMethod)
XML init-method 或 @Bean(initMethod="...") 指定的方法在此调用。
⑦ BeanPostProcessor#afterInitialization() ⭐ AOP 代理在此创建
AnnotationAwareAspectJAutoProxyCreator 在此检查是否需要创建 AOP 代理。若 Bean 匹配切点，返回代理对象替换原始 Bean。容器中存放的是代理。
⑧ Bean 就绪，供业务使用
注册到 singletonObjects（一级缓存），所有注入点拿到此对象（可能是代理）。
⑨ @PreDestroy（容器关闭时）
推荐的清理入口，用于资源释放、连接关闭等。
⑩ DisposableBean.destroy() / 自定义 destroy-method
实现 DisposableBean 接口或 @Bean(destroyMethod) 指定的销毁方法。
■ 容器感知   ■ 初始化扩展点   ■ AOP 织入点（重要！）   ■ 销毁阶段
面试要点：Bean 生命周期的扩展点顺序：BeanPostProcessor.postProcessBeforeInitialization → @PostConstruct / init-method → postProcessAfterInitialization（AOP 代理在此生成）。面试常问：为什么循环依赖不能用构造器注入？因为实例化阶段对象还未放入缓存，无法解决依赖。
● Spring · AOP

AOP 原理

AOP（面向切面编程）将横切关注点（日志、事务、权限校验等）从业务逻辑中分离， 通过代理模式在运行时将增强逻辑织入目标方法，实现解耦。

Aspect（切面）
横切逻辑的模块化封装，通常是一个带 @Aspect 的类，包含 Pointcut + Advice。
JoinPoint（连接点）
程序执行过程中可以被拦截的点，Spring AOP 仅支持方法级别。
Pointcut（切点）
用表达式描述"要拦截哪些 JoinPoint"，如 execution(* com.example..*.*(..))。
Advice（通知）
在 JoinPoint 处执行的增强代码，分 Before / After / Around 等类型。
Weaving（织入）
将 Aspect 应用到目标对象的过程。Spring AOP 是运行时动态代理织入。
Target（目标对象）
被代理的原始对象，业务代码调用的是代理，代理再委托给 target。
JDK 动态代理 vs CGLIB 代理——选择机制
JDK 动态代理
条件：目标类实现了至少一个接口
原理：Proxy.newProxyInstance() 生成实现相同接口的代理类，调用转发到 InvocationHandler.invoke()
限制：只能代理接口中声明的方法
CGLIB 代理（Spring Boot 默认）
条件：无接口，或强制指定 proxyTargetClass=true
原理：ASM 字节码生成目标类的子类，重写非 final 方法
限制：final 类/方法无法代理；需要无参构造
Spring Boot 2.x+ 默认使用 CGLIB（即使有接口），因为 CGLIB 可以解决 @Configuration 类的 @Bean 方法自调用问题。
强制使用 JDK 代理：@EnableAspectJAutoProxy(proxyTargetClass=false)
Advice 类型与执行顺序（@Around 包裹所有）
@Around — proceedingJoinPoint.proceed() 前的代码 ↓
@Before — 目标方法执行前 ↓
目标方法执行（target.method()）
@AfterReturning（正常返回后）↑
@AfterThrowing（异常抛出后，与 AfterReturning 互斥）↑
@After（finally，无论正常/异常都执行）↑
@Around — proceed() 后的代码 ↑
多个 Aspect 时按 @Order 从小到大排序（外层先进后出）。
面试要点：AOP 代理选择规则：实现接口用 JDK 动态代理（基于 InvocationHandler），无接口或强制使用用 CGLIB（继承子类）。Spring Boot 2.x 起默认 CGLIB。self 调用失效：同类内部方法互调绕过代理，解决方案：注入自身 / AopContext.currentProxy() / 拆分类。
● Spring · Transaction

事务管理

Spring 声明式事务基于 AOP，@Transactional 通过代理拦截方法调用， 在方法前后自动管理事务的开启、提交与回滚。

7 种事务传播行为（Propagation）
传播行为外层有事务时外层无事务时典型场景
REQUIRED（默认）加入外层事务创建新事务绝大多数写操作
REQUIRES_NEW挂起外层，创建独立新事务创建新事务日志记录（独立提交）
SUPPORTS加入外层事务以非事务方式运行只读查询，灵活适配
NOT_SUPPORTED挂起外层，以非事务运行以非事务运行发送 MQ 消息
MANDATORY加入外层事务抛 IllegalTransactionStateException强制要求调用方开启事务
NEVER抛 IllegalTransactionStateException以非事务运行强制要求无事务环境
NESTED创建嵌套事务（Savepoint）等同 REQUIRED部分操作可独立回滚
REQUIRES_NEW vs NESTED：REQUIRES_NEW 是两个独立事务（外层回滚不影响 NEW 已提交的数据）；NESTED 是父子关系（父回滚则子全部撤销）。
@Transactional 四大失效陷阱
① 同类方法自调用（最常见）
this.method() 直接调用，绕过了 Spring AOP 代理。
修复：注入自身 @Autowired ApplicationService self，通过 self 调用；或拆分到不同 Bean。
② 异常类型不匹配
默认只回滚 RuntimeException 和 Error。抛出受检异常（IOException 等）不自动回滚。
修复：@Transactional(rollbackFor = Exception.class)
③ 方法非 public
CGLIB 子类无法重写 private/final 方法。修复：改为 public。
④ 异常被 catch 吞掉
try-catch 捕获后未重抛，代理层收不到异常不触发回滚。
修复：重新 throw，或调用 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()
面试要点：事务失效常见场景：① 非 public 方法；② 同类 self 调用；③ 异常被 catch 吞掉；④ 抛出的是 Checked Exception（默认只回滚 RuntimeException）。传播行为面试重点：REQUIRED（默认，共用事务）vs REQUIRES_NEW（独立事务，外层回滚不影响）vs NESTED（嵌套事务，依赖 savepoint）。
● Spring · Circular Dependency

循环依赖与三级缓存

A 依赖 B，B 依赖 A——Spring 通过三级缓存在 Bean 尚未完成初始化时 提前暴露"早期引用"来打破循环，而不需要对象提前完全就绪。

三级缓存结构
一级缓存
singletonObjects
Map<name, Object>
存放完全初始化完成的单例 Bean（含 AOP 代理）。getBean() 优先命中此处。
二级缓存
earlySingletonObjects
Map<name, Object>
存放早期暴露的 Bean 引用（已实例化但未完成属性注入），防止对同一 Bean 重复调用 ObjectFactory。
三级缓存
singletonFactories
Map<name, ObjectFactory>
存放 ObjectFactory lambda，用于生成早期 Bean 引用（若有 AOP 则在此生成代理）。Bean 实例化后立即存入。
A ↔ B 循环依赖解析流程（Setter / Field 注入）
1
getBean(A) → 缓存未命中 → 实例化 A → 将 A 的 ObjectFactory 放入 三级缓存
2
populateBean(A) → 需注入 B → getBean(B) → 缓存未命中 → 实例化 B → B 的 ObjectFactory 放入 三级缓存
3
populateBean(B) → 需注入 A → getBean(A) → 三级缓存命中 → 调用 A 的 ObjectFactory（若有 AOP 则生成代理）→ 早期 A 存入 二级缓存
4
B 成功注入早期 A → B 完成初始化 → B 存入 一级缓存
5
回到 A 的注入流程 → A 成功注入完整 B → A 完成初始化 → A 存入 一级缓存
✓
循环依赖解决完成。A 和 B 均在一级缓存，各自持有对方的完整/早期引用。
⚠
以下情况无法解决循环依赖：
  • ▸构造器注入：实例化时就需要依赖，无法先放入三级缓存，直接抛 BeanCurrentlyInCreationException。
  • ▸Prototype Bean：每次 getBean 都创建新实例，Spring 不维护早期引用，无法打破循环。
  • ▸@Async + 循环依赖：@Async 的代理创建时机晚于三级缓存暴露，前后代理不一致，Spring 检测到后抛异常。解决：@Lazy 打破循环。
💡
为什么需要三级缓存？二级不够吗？
若 A 存在 AOP，早期引用需要是代理对象。三级缓存存 ObjectFactory，延迟但唯一地生成代理——保证整个生命周期中 A 的代理对象只有一个，所有早期引用都指向同一个代理，维护了 Bean 的一致性。
面试要点：三级缓存解决循环依赖的关键是提前暴露半成品对象。一级缓存存完整 Bean，二级存早期代理引用，三级存 ObjectFactory（延迟生成代理）。构造器注入无法解决（对象未创建就要注入），@Lazy 可绕过构造器循环。Spring Boot 2.6+ 默认禁止循环依赖，需显式开启。
● Interview · Spring IoC

Spring IoC 面试题

涵盖容器原理、Bean 生命周期、作用域与依赖注入等高频考点。

IoC·01 Spring IoC 容器的实现原理是什么？BeanFactory 和 ApplicationContext 有什么区别？ +
IoC 容器本质是一个 Bean 工厂，通过读取配置（XML / 注解 / Java Config）生成 BeanDefinition（Bean 的元数据），再通过反射创建对象、完成依赖注入，最终注册到缓存供使用。
  • BeanFactory：根接口，懒加载（首次 getBean 才实例化），功能基础。
  • ApplicationContext：BeanFactory 子接口，启动时饥饿加载所有单例，额外支持：国际化、事件发布、AOP/事务集成。
  • Spring Boot 使用 AnnotationConfigServletWebServerApplicationContext，内嵌 Tomcat。
IoC·02 请描述 Spring Bean 完整的生命周期，并说明哪个阶段创建了 AOP 代理？ +
  • ① 实例化：反射调用构造方法
  • ② 属性填充：@Autowired / @Value 注入
  • ③ Aware 回调：BeanNameAware → BeanFactoryAware → ApplicationContextAware
  • ④ BPP#beforeInitialization：@PostConstruct 在此调用
  • ⑤ InitializingBean.afterPropertiesSet()
  • ⑥ 自定义 init-method
  • ⑦ BPP#afterInitialization：AOP 代理在此创建！ AnnotationAwareAspectJAutoProxyCreator 检查是否需要代理，若匹配切点则返回代理对象
  • ⑧ Bean 就绪，注册到一级缓存
  • ⑨~⑩ 销毁：@PreDestroy → DisposableBean.destroy() → destroy-method
关键点 容器中注册的是第⑦步产生的代理对象。@Transactional、@Async 的增强逻辑都通过这个代理执行。
IoC·03 @Autowired 和 @Resource 有什么区别？注入时发现多个匹配 Bean 怎么处理？ +
  • @Autowired（Spring 原生）：默认按类型注入，多个候选 Bean 时用 @Qualifier 指定，或 @Primary 标记优先 Bean。
  • @Resource（JSR-250）：默认按名称注入（先 name → 再 type），适合需要按名精准注入的场景。
  • 多 Bean 冲突处理：@Primary 首选 / @Qualifier 精确 / 注入 List<T> 收集所有 / @ConditionalOnXxx 控制注册。
IoC·04 singleton Bean 注入 prototype Bean 时为什么只有一个实例？如何获取每次不同的实例？ +
问题根因
singleton Bean 只初始化一次，注入在属性填充时完成，之后字段不再变化。即使注入 prototype Bean，注入动作只发生一次，后续调用永远拿到那个初始实例。
  • 方案①：注入 ApplicationContext，每次手动 context.getBean(PrototypeBean.class)
  • 方案②：注入 ObjectFactory<PrototypeBean>，调用 .getObject()
  • 方案③：使用 @Lookup 注解，Spring 用 CGLIB 重写该方法每次返回新实例（最优雅）
IoC·05 Spring Boot 自动配置（Auto-Configuration）的原理是什么？ +
@SpringBootApplication 组合了 @EnableAutoConfiguration，后者通过 @Import(AutoConfigurationImportSelector.class) 触发加载。
  • Spring Boot 2.x：读取所有 jar 包下的 META-INF/spring.factories 获取自动配置类列表。
  • Spring Boot 3.x：改为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports。
  • 条件过滤：@ConditionalOnClass（classpath 有某类才生效）、@ConditionalOnMissingBean（用户未自定义时才生效）确保"有需要才注册、用户优先"。
IoC·06 @Configuration 的 Full 模式和 Lite 模式有什么区别？BeanFactoryPostProcessor 和 BeanPostProcessor 有什么区别？ +
Full 模式（@Configuration）：Spring 用 CGLIB 子类代理该配置类，@Bean 方法之间互相调用会经过容器，返回同一单例 Bean。

Lite 模式（@Component 上的 @Bean / proxyBeanMethods=false）：无 CGLIB 代理，@Bean 方法是普通 Java 方法，互相调用会 new 出新实例，可能出现多个不同实例的 Bug。

BeanFactoryPostProcessor vs BeanPostProcessor：
  • BeanFactoryPostProcessor：在所有 Bean 实例化之前，修改 BeanDefinition（元数据）。典型实现：PropertySourcesPlaceholderConfigurer 替换 ${...} 占位符
  • BeanPostProcessor：在 Bean 实例化之后的初始化前后，对 Bean 实例做增强。@Autowired 注入（AutowiredAnnotationBeanPostProcessor）和 AOP 代理创建（AnnotationAwareAspectJAutoProxyCreator）均通过 BPP 实现
记忆口诀BFPP 改元数据（Before），BPP 改实例（After init）。
● Interview · Spring AOP

Spring AOP 面试题

聚焦代理机制、切点表达式、通知类型与常见失效场景。

AOP·01 JDK 动态代理和 CGLIB 代理的实现原理有何不同？Spring 如何选择？ +
  • JDK 动态代理：运行时通过 Proxy.newProxyInstance() 生成实现相同接口的代理类，调用委托给 InvocationHandler.invoke()。要求目标类必须实现接口，只能代理接口方法。
  • CGLIB 代理：通过 ASM 字节码生成目标类的子类，重写非 final 方法。无需接口，但无法代理 final 类/方法/private 方法。
  • Spring 选择逻辑：目标类没有接口 → CGLIB；有接口 → 默认也用 CGLIB（Spring Boot 2.x 起），除非显式配置 proxyTargetClass=false。
AOP·02 AOP 失效的常见原因有哪些？如何排查"同类调用 @Transactional 不生效"？ +
最高频失效场景
同类内 this 调用：this.methodB() 是原始对象，不是代理，AOP 代理只在被外部调用时生效。
  • 修复①：提取到另一个 Bean
  • 修复②：@Autowired ApplicationService self，通过 self.methodB()
  • 修复③：((ApplicationService) AopContext.currentProxy()).methodB()（需 exposeProxy=true）
其他失效：方法非 public；Bean 不由 Spring 管理；@Aspect 类未被管理；切点表达式写错。
AOP·03 @Around、@Before、@After、@AfterReturning、@AfterThrowing 的执行顺序？多个切面如何排序？ +
正常流程：@Around(前) → @Before → 目标方法 → @AfterReturning → @After → @Around(后)

异常流程：@Around(前) → @Before → 目标方法抛异常 → @AfterThrowing → @After → @Around(异常传播)

多切面排序：使用 @Order(n)，n 越小优先级越高（先进后出，类似栈）。不加 @Order 则按类名字母序（不稳定，应显式指定）。
AOP·04 Spring AOP 和 AspectJ 有什么区别？各自适用什么场景？ +
  • Spring AOP：基于动态代理，运行时织入，只支持方法级别 JoinPoint，仅对 Spring Bean 有效。无需额外编译，覆盖绝大多数业务场景。
  • AspectJ：完整 AOP 框架，支持编译时/类加载时/运行时织入，支持字段级别、构造器级别 JoinPoint，可增强任何 Java 对象。需要 ajc 编译器或 LTW agent。
  • 选择：Spring AOP 满足 90% 的事务/日志/权限场景；AspectJ 用于非 Spring Bean 增强、字段拦截或解决 this 调用问题。
AOP·05 @Async 的实现原理是什么？有哪些常见失效场景？异常如何捕获？ +
实现原理：@EnableAsync 注册 AsyncAnnotationBeanPostProcessor，对含 @Async 方法的 Bean 生成 AOP 代理。调用时代理拦截，将任务提交到线程池异步执行，方法立即返回。

默认线程池：使用 SimpleAsyncTaskExecutor（每次调用新建线程，无复用），生产环境必须自定义 ThreadPoolTaskExecutor Bean（名称为 taskExecutor）覆盖。

常见失效场景（与 @Transactional 同根因）：
  • 同类 this 调用：绕过代理，退化为同步执行。解决：注入自身 Bean 或拆分到新 Bean
  • 方法非 public：CGLIB 无法重写，代理失效
  • @SpringBootApplication 类与 @Async Bean 循环依赖：代理未正确创建，启动报错
异常处理：异步方法抛出的异常不会传播到调用方，需通过以下方式处理：
  • 方法返回 Future / CompletableFuture，调用方 get() 时捕获 ExecutionException
  • 实现 AsyncUncaughtExceptionHandler 接口，统一处理无返回值方法的异常

Spring 事务面试题

聚焦传播行为、隔离级别、@Transactional 失效与循环依赖等高频考点。

事务·01 REQUIRED 和 REQUIRES_NEW 的区别？NESTED 和 REQUIRES_NEW 的区别？ +
  • REQUIRED：有外层事务则加入，共享同一 Connection。内/外层回滚互相影响（同一事务）。
  • REQUIRES_NEW：挂起外层，开启独立新事务（新 Connection）。两者完全独立，日志记录的典型场景。
  • NESTED：在外层事务内创建嵌套事务（Savepoint），同一 Connection。内层回滚到 Savepoint 不影响外层；但外层回滚会连带子事务全部撤销。
关键区别 REQUIRES_NEW 两个独立事务互不影响；NESTED 是父子关系，父回滚则子也回滚。
事务·02 @Transactional 有哪些常见失效场景？分别如何解决？ +
  • ① 同类 this 调用：绕过代理。解决：注入自身或拆分 Bean。
  • ② 方法非 public：CGLIB 无法重写。解决：改为 public。
  • ③ 受检异常不回滚：默认只回滚 RuntimeException。解决：rollbackFor=Exception.class。
  • ④ 异常被 catch 吞掉：代理收不到异常。解决：重新 throw 或 setRollbackOnly()。
  • ⑤ Bean 不由 Spring 管理：new 出来的对象无代理。解决：通过容器获取。
  • ⑥ 多线程跨线程：事务绑定在 ThreadLocal，子线程不在同一事务。解决：子线程开新事务或操作放回主线程。
事务·03 Spring 事务的隔离级别有哪些？DEFAULT 对应的是什么？ +
  • DEFAULT：使用数据库默认隔离级别（MySQL InnoDB 默认 REPEATABLE_READ）
  • READ_UNCOMMITTED：可读未提交，有脏读/不可重复读/幻读
  • READ_COMMITTED：只读已提交，解决脏读（Oracle 默认）
  • REPEATABLE_READ：解决脏读和不可重复读（MySQL 默认，InnoDB 通过 MVCC+Gap Lock 也解决大部分幻读）
  • SERIALIZABLE：完全串行化，解决所有并发问题，性能最差
事务·04 Spring 三级缓存为什么能解决循环依赖？为什么构造器注入无法解决？ +
能解决的原因
Setter/Field 注入时，Bean 可以先实例化（new 出来）再填充依赖。Spring 实例化后立即把 ObjectFactory 放入三级缓存。当 B 需要 A 时，从三级缓存拿到 A 的早期引用（可能是 AOP 代理）让 B 完成初始化，再回头让 A 注入 B。
  • 构造器注入无法解决：实例化 A 时就必须传入 B，但 B 还未实例化，A 无法先放入缓存，Spring 直接抛 BeanCurrentlyInCreationException。
  • 为什么需要三级而非二级：三级缓存存 ObjectFactory，调用时才生成 AOP 代理，保证整个生命周期中只有一个一致的代理对象。
事务·05 @Transactional 的 readOnly=true 有什么作用？ +
  • 数据库层优化：MySQL Connector/J 会设置 SET SESSION TRANSACTION READ ONLY，数据库跳过 undo log 生成，减少锁竞争，提升查询性能。
  • Spring 层语义：Hibernate Session 不做 dirty check，不 flush，减少不必要操作。
  • 适用场景：纯查询方法（find/get/list）加 readOnly=true，既优化性能又声明代码意图。
  • 注意：readOnly 不阻止写操作（代码层面仍可执行 insert），只是 hint，实际约束由数据库执行。
事务·06 微服务中 @Transactional 无法解决跨服务事务，常见的分布式事务方案有哪些？各自适用什么场景？ +
@Transactional 基于本地 JDBC Connection，跨服务/跨库时无法保证原子性。常见方案：
  • AT 模式（Seata）：自动代理 DataSource，一阶段直接提交本地事务（高性能），二阶段根据全局结果执行 undo log 补偿回滚。侵入性低，但有 undo_log 表开销，不支持 DDL 类 SQL。
  • TCC（Try-Confirm-Cancel）：业务方实现三个接口——Try 预留资源，Confirm 提交，Cancel 释放。性能最好，但侵入大，需自行处理幂等、空回滚、悬挂问题。
  • SAGA：长事务拆成多个本地事务，失败时执行补偿操作（逆操作链）。无全局锁、高并发友好，但隔离性差，中间状态可见，需业务层保证幂等。
  • 可靠消息最终一致：本地事务写消息表，异步投递 MQ，消费方幂等消费。不保证强一致，适合订单→库存扣减等最终一致场景。
选型原则强一致可接受性能损耗 → AT/TCC；高并发长流程 → SAGA；最终一致 → 可靠消息。
⚡ 实战 @Transactional 不生效，你遇到过哪些坑？怎么排查？ +

一、常见失效原因

  • ① 方法非 public：Spring AOP 默认只拦截 public 方法。如果 @Transactional 标注在 protected、private 或包级可见的方法上，事务切面不会织入，方法执行时不在事务上下文中。可通过 TransactionAttributeSource 确认方法是否被识别为事务方法。
  • ② 同类内部调用未经过代理：同一个类中，方法 A 调用方法 B（B 加了 @Transactional），由于调用走的是 this 引用而非代理对象，事务切面被完全绕过。这是最常见的踩坑点。例如：OrderService.createOrder() 内部调用 this.deductStock()，即使 deductStock 有 @Transactional 也不会开启事务。
  • ③ 异常类型不匹配：@Transactional 默认只对 RuntimeException 和 Error 回滚，checked exception（如 IOException、SQLException）不会触发回滚。如果业务方法抛出 checked exception 但希望回滚，必须显式声明 @Transactional(rollbackFor = Exception.class)。
  • ④ 异常被 catch 吞掉：方法内部 try-catch 捕获了异常但没有重新抛出，Spring 事务拦截器感知不到异常，认为方法正常完成，直接提交事务。这是隐蔽性最高的问题——代码看起来正常，但数据已经不一致。
  • ⑤ 传播级别设为 NOT_SUPPORTED / NEVER：如果 @Transactional 的 propagation 设为 NOT_SUPPORTED（挂起当前事务，非事务执行）或 NEVER（如果有事务则抛异常），方法不会在事务中运行。排查时注意检查注解参数。

二、排查手段

  • 检查代理对象是否生效：在调用方打断点，查看被调用对象是否是代理对象（CGLIB 子类或 JDK Proxy）。可以使用 AopUtils.isAopProxy(obj) 判断——返回 false 说明没有代理，事务切面不会生效。也可以在方法内打印 this.getClass().getName()，如果是原始类名而非 $$SpringCGLIB$$ 后缀，说明未经过代理。
  • Debug TransactionInterceptor：在 org.springframework.transaction.interceptor.TransactionInterceptor#invoke 上打断点。如果调用时没有进入该方法，说明事务切面未织入，问题出在代理层面。如果进入了但事务未生效，检查 TransactionAttribute 的解析结果。
  • 日志对比 this 与 proxy：在 Bean 中注入自身（@Autowired private OrderService self;），分别用 this.method() 和 self.method() 调用，观察日志中两者的类名差异。如果 this 是原始类而 self 是代理类，即可确认自调用问题。
  • 开启事务日志：在 application.yml 中设置 logging.level.org.springframework.jdbc.datasource.DataSourceTransactionManager=DEBUG（或 JPA 对应的事务管理器），观察日志中是否有 "Creating new transaction" 和 "Committing transaction" 输出。如果没有，说明事务根本没有启动。

三、解决方案

  • 自调用问题：使用 AopContext.currentProxy() 获取当前代理对象来调用（需在启动类加 @EnableAspectJAutoProxy(exposeProxy = true)），例如：((OrderService) AopContext.currentProxy()).deductStock()。或者将方法拆分到不同的 Service 中，通过 Spring 注入调用，自然经过代理。
  • 异常类型问题：统一使用 @Transactional(rollbackFor = Exception.class)，确保所有异常都能触发回滚。团队规范中应将其作为默认写法，避免依赖默认的 RuntimeException 行为。
  • 避免 catch 吞异常：如果必须 catch，在 catch 块中重新抛出运行时异常：catch (Exception e) { throw new RuntimeException(e); }，或使用 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly() 手动标记回滚。
⚡ 实战 线上应用启动报循环依赖，怎么解决？ +

一、背景：Spring Boot 2.6+ 默认禁止循环依赖

从 Spring Boot 2.6 开始，spring.main.allow-circular-references 默认值改为 false。如果存在循环依赖，启动时直接抛出 BeanCurrentlyInCreationException，错误信息类似：The dependencies of some of the beans in the application context form a cycle: serviceA → serviceB → serviceA。这个变更的目的是鼓励开发者在编码阶段就消除不合理的依赖关系，而非依赖 Spring 的三级缓存机制"静默解决"。

二、排查步骤

  • 看异常堆栈定位循环链：启动失败的异常信息会明确列出形成循环的 Bean 名称链条。例如：orderService (field com.example.StockService) → stockService (field com.example.OrderService)。根据这条链路可以快速定位到哪两个（或多个）Bean 互相依赖。
  • 检查注入方式：确认循环依赖中的注入方式是构造器注入还是 setter/field 注入。两者的解决难度完全不同——构造器注入的循环依赖无法被三级缓存解决（实例化 A 时就需要 B，而 B 还未创建），Spring 会直接报错；setter/field 注入的循环依赖可以通过三级缓存解决（先实例化 A 的半成品放入缓存，再创建 B 时注入 A 的早期引用）。
  • 检查是否涉及 AOP 代理：如果循环依赖中的 Bean 需要被 AOP 代理（如加了 @Transactional、@Async），三级缓存中存储的是 ObjectFactory，在需要时才生成代理对象。但如果代理对象与原始对象的引用不一致（如 SmartInstantiationAwareBeanPostProcessor 介入），仍可能抛出异常。

三、解决方案

  • 构造器注入的循环依赖——必须重构：三级缓存无法解决构造器循环依赖，因为实例化阶段就需要依赖对象。推荐做法：① 引入中间 Bean：将两个 Bean 共同依赖的逻辑抽取到第三个 Bean C，A 和 B 都依赖 C 而非互相依赖；② 事件解耦：A 发布事件，B 监听事件处理，消除直接依赖。例如 OrderCreatedEvent 由 OrderService 发布，StockService 用 @EventListener 监听并扣减库存。
  • setter/field 注入的循环依赖——临时方案：在 application.yml 中设置 spring.main.allow-circular-references: true，让 Spring 回到旧版行为，通过三级缓存自动解决。这只是临时止血，不应作为长期方案，因为循环依赖本身就是设计不合理的信号。
  • @Lazy 延迟注入（推荐折中方案）：在注入点加 @Lazy，Spring 不会立即创建依赖 Bean，而是注入一个代理，首次调用方法时才真正解析。例如：@Autowired @Lazy private StockService stockService;。这样既不需要改配置，也不需要大规模重构，适合存量项目快速修复。
  • 最佳实践——重新设计依赖关系：循环依赖通常意味着两个模块职责边界不清。推荐的长期方案：① 接口隔离：将 A 中被 B 依赖的方法抽取到独立接口 AQueryService，B 只依赖查询接口而非完整的 A；② 事件驱动：用 Spring 的 ApplicationEventPublisher + @EventListener 替代直接调用，实现松耦合；③ DDD 聚合根审视：如果两个实体总是互相引用，可能属于同一个聚合根，应合并为一个领域对象。

四、生产环境应急

如果线上紧急需要恢复服务（如升级 Spring Boot 版本后启动失败），最快的方式是先在配置中加 spring.main.allow-circular-references: true 恢复启动，然后排期逐步消除循环依赖。注意：Spring Boot 3.x 中该配置仍然存在，但官方明确表示未来版本可能彻底移除循环依赖支持，建议尽早治理。
